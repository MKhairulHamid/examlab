AWS Certified Data Engineer – Associate (DEA-C01)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified Data Engineer – Associate (DEA-C01) validates a candidate's ability to implement data pipelines and to monitor, troubleshoot, and optimize cost and performance issues in accordance with best practices. It is an Associate-level exam that tests whether a candidate can ingest and transform data, orchestrate pipelines, choose and model data stores, operate and monitor pipelines, ensure data quality, and apply security and governance to the data they manage.

  • Target audience: Data engineers who build and operate ETL/ELT pipelines, data lakes, and analytics data stores on AWS.
  • Prerequisite experience: 2–3 years of data engineering experience and 1–2 years of hands-on experience with AWS services. Comfort with SQL, programming concepts, and Git source control.
  • Why candidates take it: To validate practical data-engineering capability for data engineer, analytics engineer, and ETL developer roles, and as the data-focused complement to the Solutions Architect and Data Analytics tracks.

Out-of-Scope Skills (Do NOT Test)

The official exam guide explicitly lists these job tasks as out of scope for the target candidate. Any question that hinges on them is out of scope — cut it.

  • Performing ML training and inferences
  • Demonstrating knowledge of programming language-specific syntax
  • Drawing business conclusions based on data
Exam tip: DEA-C01 tests building and operating data pipelines — how to ingest, transform, store, catalog, monitor, and secure data with the right managed service. The crux of most questions is "which service or configuration best meets the cost, performance, and access requirement?" — not language syntax or data-science modeling.

Domain 1 · 34%

Data Ingestion and Transformation

By far the largest domain. Covers reading data from streaming and batch sources, transforming and processing it, orchestrating pipelines, and applying programming and software-engineering concepts.

Task 1.1 — Perform data ingestion

Knowledge of: Throughput and latency characteristics of AWS services that ingest data; data ingestion patterns (frequency, data history); streaming data ingestion; batch data ingestion (scheduled, event-driven); replayability of ingestion pipelines; stateful vs. stateless transactions.

Skills in:

  • Reading from streaming sources (e.g. Kinesis, Amazon MSK, DynamoDB Streams, AWS DMS, AWS Glue, Amazon Redshift)
  • Reading from batch sources (e.g. Amazon S3, AWS Glue, Amazon EMR, AWS DMS, Amazon Redshift, AWS Lambda, Amazon AppFlow)
  • Setting up schedulers (Amazon EventBridge, Apache Airflow, time-based) and event triggers (S3 Event Notifications, EventBridge)
  • Calling a Lambda function from Kinesis; managing fan-in and fan-out for streaming distribution
  • Implementing throttling and overcoming rate limits (DynamoDB, Amazon RDS, Kinesis); creating IP allowlists for source connections

Task 1.2 — Transform and process data

Knowledge of: Creation of ETL pipelines based on business requirements; volume, velocity, and variety of data (structured/unstructured); how to clean data; AWS transformation services (EMR, Glue, Lambda, Redshift); transforming data between formats; troubleshooting common transformation failures.

Skills in:

  • Optimizing container usage for performance (Amazon EKS, Amazon ECS)
  • Connecting to different data sources (JDBC, ODBC) and integrating data from multiple sources
  • Implementing transformation services per requirement (Amazon EMR, AWS Glue, Lambda, Amazon Redshift) and converting formats (e.g. .csv to Apache Parquet)
  • Optimizing costs while processing data; troubleshooting and debugging transformation failures and performance issues
  • Creating data APIs to make data available to other systems; integrating large language models (LLMs) for data processing

Task 1.3 — Orchestrate data pipelines

Knowledge of: How to integrate various AWS services to create ETL pipelines; event-driven architecture; how to configure AWS services for data pipelines based on schedules or dependencies; serverless workflows.

Skills in:

  • Using orchestration services to build workflows (Lambda, EventBridge, Amazon MWAA, AWS Step Functions, AWS Glue workflows)
  • Building data pipelines for performance, availability, scalability, resiliency, and fault tolerance
  • Implementing and maintaining serverless workflows
  • Using notification services to send alerts (Amazon SNS, Amazon SQS)

Task 1.4 — Apply programming concepts

Knowledge of: Continuous integration and continuous delivery (CI/CD); SQL queries; infrastructure as code (IaC) for repeatable deployments; distributed computing; data structures and algorithms; SQL query optimization.

Skills in:

  • Optimizing code to reduce runtime for ingestion and transformation; configuring Lambda concurrency and performance
  • Using programming languages and frameworks for data engineering (Python, SQL, Scala, R, Java, Bash, PowerShell)
  • Applying software engineering best practices (version control, testing, logging, monitoring)
  • Using IaC for repeatable deployment (AWS CloudFormation, AWS CDK) and AWS SAM to package serverless data pipelines
  • Using and mounting storage volumes from within Lambda functions
Domain 2 · 26%

Data Store Management

Choosing and modeling where data lives. Covers selecting the right data store, cataloging schemas, managing data lifecycle, and designing data models that evolve.

Task 2.1 — Choose a data store

Knowledge of: Storage platforms and their characteristics; storage services and configurations for specific performance demands; data storage formats (e.g. .csv, .txt, Parquet); how to align data storage with migration requirements; how to determine the appropriate storage solution for access patterns; open table formats; vector index types.

Skills in:

  • Implementing the appropriate storage services for cost and performance (Amazon Redshift, Amazon EMR, AWS Lake Formation, Amazon RDS, DynamoDB, Kinesis Data Streams, Amazon MSK)
  • Applying storage services to use cases (e.g. HNSW indexing with Amazon Aurora PostgreSQL; Amazon MemoryDB for fast key/value access)
  • Integrating migration tools (e.g. AWS Transfer Family) and remote access methods (Redshift federated queries, materialized views, Redshift Spectrum)
  • Managing locks to prevent access to data (Amazon Redshift, Amazon RDS); managing open table formats (e.g. Apache Iceberg)
  • Describing vector index types (e.g. HNSW, IVF)

Task 2.2 — Understand data cataloging systems

Knowledge of: How to create a data catalog; data classification based on requirements; components of metadata and data catalogs.

Skills in:

  • Using data catalogs to consume data from the data's source
  • Building and referencing a technical data catalog (AWS Glue Data Catalog, Apache Hive metastore)
  • Discovering schemas and using AWS Glue crawlers to populate catalogs; synchronizing partitions with a catalog
  • Creating new source or target connections for cataloging (e.g. AWS Glue); creating and managing business data catalogs (e.g. Amazon SageMaker Catalog)

Task 2.3 — Manage the lifecycle of data

Knowledge of: Appropriate storage solutions for specific access patterns; hot/warm/cold data storage cost optimization; how to manage S3 object lifecycle; how to delete data to meet business and legal requirements; data retention policies and archiving strategies; protecting data with appropriate resiliency and availability.

Skills in:

  • Performing load and unload operations to move data between Amazon S3 and Amazon Redshift
  • Managing S3 Lifecycle policies to change storage tier and to expire data at a specific age
  • Managing S3 versioning and DynamoDB TTL; deleting data to meet business and legal requirements
  • Protecting data with appropriate resiliency and availability

Task 2.4 — Design data models and schema evolution

Knowledge of: Data modeling concepts; how to ensure accuracy and trustworthiness using data lineage; best practices for indexing, partitioning, compression, and other data optimization techniques; how to model structured, semi-structured, and unstructured data; schema evolution techniques; vectorization concepts.

Skills in:

  • Designing schemas for Amazon Redshift, DynamoDB, and Lake Formation; addressing changes to the characteristics of data
  • Performing schema conversion (e.g. AWS Schema Conversion Tool [AWS SCT], AWS DMS Schema Conversion)
  • Establishing data lineage with AWS tools (Amazon SageMaker ML Lineage Tracking, Amazon SageMaker Catalog)
  • Applying indexing, partitioning strategies, compression, and other optimization techniques; describing vectorization (e.g. Amazon Bedrock knowledge base)
Domain 3 · 22%

Data Operations and Support

Running pipelines in production. Covers automating data processing, analyzing data, maintaining and monitoring pipelines, and ensuring data quality.

Task 3.1 — Automate data processing by using AWS services

Knowledge of: How to maintain and troubleshoot data processing for repeatable business outcomes; API calls for data processing; which services accept scripting (e.g. EMR, Redshift, AWS Glue).

Skills in:

  • Orchestrating data pipelines (Amazon MWAA, AWS Step Functions); troubleshooting Amazon managed workflows
  • Calling SDKs to access Amazon features from code; using AWS service features to process data (Amazon EMR, Amazon Redshift, AWS Glue)
  • Preparing data for transformation (AWS Glue DataBrew, Amazon SageMaker Unified Studio); querying data (Amazon Athena)
  • Using AWS Lambda to automate data processing; managing events and schedulers (Amazon EventBridge)

Task 3.2 — Analyze data by using AWS services

Knowledge of: Tradeoffs between provisioned and serverless services; SQL queries (e.g. SELECT statements with multiple qualifiers or JOIN clauses); how to visualize data for analysis; data aggregation, rolling average, grouping, and pivoting.

Skills in:

  • Visualizing data with AWS services and tools (AWS Glue DataBrew, Amazon QuickSight)
  • Verifying and cleaning data (Lambda, Athena, QuickSight, Jupyter notebooks, Amazon SageMaker Data Wrangler)
  • Using SQL in Amazon Redshift and Athena to query data or create views; using Athena notebooks that use Apache Spark to explore data

Task 3.3 — Maintain and monitor data pipelines

Knowledge of: How to log application data; best practices for performance tuning; how to log access to AWS services; Amazon Macie, AWS CloudTrail, and Amazon CloudWatch.

Skills in:

  • Extracting logs for audits; deploying logging and monitoring solutions to facilitate auditing and traceability
  • Using notifications during monitoring to send alerts; troubleshooting performance issues
  • Using AWS CloudTrail to track API calls; using Amazon CloudWatch Logs to log application data
  • Analyzing logs with AWS services (Athena, Amazon EMR, Amazon OpenSearch Service, CloudWatch Logs Insights)

Task 3.4 — Ensure data quality

Knowledge of: Data sampling techniques; how to implement data skew mechanisms; data validation (completeness, consistency, accuracy, integrity).

Skills in:

  • Running data quality checks while processing the data (e.g. checking for empty fields)
  • Defining data quality rules (e.g. AWS Glue DataBrew); investigating data consistency (e.g. DataBrew)
  • Describing data sampling techniques; implementing data skew mechanisms
Domain 4 · 18%

Data Security and Governance

Protecting and governing data. Covers authentication, authorization, encryption and masking, preparing logs for audit, and data privacy and governance.

Task 4.1 — Apply authentication mechanisms

Knowledge of: VPC security networking concepts; differences between managed and unmanaged services; authentication methods (password-based, certificate-based, role-based); differences between AWS managed policies and customer managed policies.

Skills in:

  • Updating VPC security groups; creating and updating IAM groups, roles, endpoints, and services
  • Creating and rotating credentials for password management (e.g. AWS Secrets Manager)
  • Setting up IAM roles for access (AWS Lambda, Amazon API Gateway, AWS CLI, AWS CloudFormation)
  • Applying IAM policies to roles, endpoints, and services (S3 Access Points, AWS PrivateLink)

Task 4.2 — Apply authorization mechanisms

Knowledge of: Authorization methods (role-based, policy-based, tag-based, attribute-based); principle of least privilege for access; role-based access control and expected access patterns; methods to protect data from unauthorized access across services.

Skills in:

  • Creating custom IAM policies when a managed policy does not meet the need; constructing policies that meet least privilege
  • Storing application and database credentials (Secrets Manager, AWS Systems Manager Parameter Store)
  • Providing database users, groups, and roles access and authority in a database (e.g. Amazon Redshift)
  • Managing permissions through AWS Lake Formation (for Redshift, EMR, Athena, and S3)

Task 4.3 — Ensure data encryption and masking

Knowledge of: Data encryption options available in AWS analytics services; differences between client-side and server-side encryption; protecting sensitive data; data anonymization, masking, and key salting.

Skills in:

  • Applying data masking and anonymization according to compliance laws or company policies
  • Using encryption keys to encrypt or decrypt data (e.g. AWS KMS); configuring encryption across AWS account boundaries
  • Enabling encryption in transit or before transit for data

Task 4.4 — Prepare logs for audit

Knowledge of: How to log application data; how to log access to AWS services; centralized AWS logs.

Skills in:

  • Using AWS CloudTrail to track API calls; using Amazon CloudWatch Logs to store application logs
  • Using AWS CloudTrail Lake for centralized logging queries
  • Analyzing logs by using AWS services (Athena, CloudWatch Logs Insights, Amazon OpenSearch Service)
  • Integrating various AWS services to perform logging (e.g. Amazon EMR for large volumes of log data)

Task 4.5 — Understand data privacy and governance

Knowledge of: How to protect personally identifiable information (PII); data sovereignty; how to maintain data privacy and governance.

Skills in:

  • Granting permissions for data sharing (e.g. data sharing for Amazon Redshift); implementing PII identification (e.g. Amazon Macie with Lake Formation)
  • Implementing data privacy strategies to prevent backups or replications to disallowed AWS Regions; maintaining data sovereignty
  • Viewing configuration changes in an account (e.g. AWS Config); managing data access through Amazon SageMaker Catalog projects
Candidate Prep

How We Help DEA-C01 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 720+.

Who Takes This Exam

  • Working data engineers — already building ETL/ELT pipelines who want to validate their AWS data skills.
  • Analytics & BI engineers — moving from on-premises or single-tool stacks to AWS data lakes and warehouses.
  • Backend & cloud engineers — taking on data-pipeline responsibility and needing the data-store and orchestration depth.

What Candidates Struggle With

  • Streaming vs. batch ingestion — Kinesis Data Streams vs. Firehose vs. MSK vs. Glue/DMS batch, and which fits a latency, throughput, or replay requirement.
  • Choosing the right data store — Redshift vs. RDS vs. DynamoDB vs. S3 + Athena vs. EMR, matched to access pattern, cost, and scale.
  • Glue ecosystem depth — Glue ETL vs. crawlers vs. Data Catalog vs. DataBrew vs. Glue Data Quality, and how partitions and Iceberg tables fit together.
  • Orchestration choices — Step Functions vs. MWAA (Airflow) vs. Glue workflows vs. EventBridge, and when each is the least-overhead fit.
  • Lake Formation permissions — fine-grained access across Redshift, Athena, EMR, and S3, and how it differs from raw IAM and S3 policies.
  • Cost & performance optimization — Parquet/partitioning/compression, provisioned vs. serverless trade-offs, and S3 storage-class lifecycle.

Recommended Study Path

  1. Domain 1 — Ingestion & Transformation first. Over a third of the exam (34%) and the spine of every pipeline. Build the streaming-vs-batch and Glue/EMR/Lambda/Redshift transformation mental model before anything else.
  2. Domain 2 — Data Store Management. 26% and tightly coupled to D1. Master data-store selection, the Glue Data Catalog, S3 lifecycle, and schema/partitioning design.
  3. Domain 3 — Operations & Support. 22%. Automation, analysis with Athena/QuickSight, pipeline monitoring with CloudWatch/CloudTrail, and data-quality checks.
  4. Domain 4 — Security & Governance last. Smallest (18%) but it threads through every pipeline — IAM and Lake Formation authorization, KMS encryption, audit logging, and PII/governance with Macie and Config.

AI Explanation Quality for This Exam

DEA-C01 questions are scenario-based and often have several workable services where one is the best fit on cost, latency, access pattern, or operational overhead. Every explanation must cover WHY a plausible option is inferior — not just "this is wrong." Common patterns to cover:

  • For ingestion questions: explain why a service misses a constraint (e.g. "Kinesis Data Firehose buffers and cannot meet a sub-second latency requirement; Kinesis Data Streams delivers records in real time").
  • For data-store questions: explain why an option is wrong for the access pattern (e.g. "DynamoDB is poor for ad-hoc analytical scans; Redshift or Athena over S3 fits the OLAP query").
  • For format/cost questions: explain why a choice wastes money or scan time (e.g. "querying raw CSV in Athena scans the whole file; Parquet with partitioning prunes data and cuts cost").

Question Writing Rules (DEA-C01 specific)

  • Scenario stems — "A data engineering team needs to ingest…" Establish the real constraint (latency, throughput, cost, access pattern, compliance) before asking.
  • Best-of-several-workable answers — most distractors are technically valid services but lose on cost, latency, operational overhead, or one unmet requirement. Engineer the crux to be the trade-off.
  • Managed and least-overhead framing — the right answer is usually the most managed, serverless, purpose-fit option. Make hand-rolled or mismatched services the plausible-but-wrong choices.
  • Cross-domain wiring — DEA scenarios routinely span ingestion, storage, cataloging, and security. Let questions combine services as the real exam does.
  • No ML modeling, no language syntax, no business conclusions — keep questions inside the in-scope service set; cut anything that hinges on out-of-scope depth.

Distractor Quality Guidelines

  • Use real data-engineering confusions as distractors: Kinesis Data Streams vs. Firehose vs. MSK; Glue ETL vs. EMR vs. Lambda; Redshift vs. Athena vs. RDS; Step Functions vs. MWAA vs. Glue workflows.
  • For ingestion questions: make one distractor a service whose latency or buffering model cannot meet the stated requirement, and one a higher-overhead self-managed approach.
  • For storage/format questions: make one distractor a store mismatched to the access pattern, and one a format/layout that wastes scan cost.
  • For security questions: make one distractor broad IAM where Lake Formation fine-grained control is required, and one an option that breaks least privilege or data sovereignty.

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — the conceptual reason it best meets every stated constraint
  • Why each distractor is wrong — one sentence per option, naming the specific unmet constraint or excess overhead
  • Concept link — which domain/task statement this tests (e.g. "D1 · Task 1.1 — Perform data ingestion")
  • Service context — if a service appears, one line on what it does and what distinguishes it from the similar services in the distractors
