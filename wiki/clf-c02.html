AWS Certified Cloud Practitioner (CLF-C02)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified Cloud Practitioner (CLF-C02) validates broad, foundational knowledge of the AWS Cloud — not role-specific depth. It is the entry point to the AWS certification path and the recommended first cert for anyone beginning their cloud journey regardless of technical background.

  • Target audience: Early-career cloud learners, non-technical roles (sales, finance, ops) working alongside AWS teams, and anyone wanting to validate foundational cloud literacy.
  • Prerequisite experience: Up to 6 months of AWS exposure — design, implementation, or operations. No prior certification required.
  • Why candidates take it: Career pivots into cloud, employer-required certification, or as a stepping stone before associate-level exams (SAA, DVA, etc.).

Out-of-Scope Skills (Do NOT Test)

The official exam guide explicitly excludes these. Any question that touches them is out of scope — cut it.

  • Writing or reviewing code
  • Designing cloud architecture
  • Troubleshooting technical configurations
  • Step-by-step implementation instructions
  • Load testing or performance testing
Exam tip: CLF-C02 tests awareness and recognition, not implementation depth. If a question requires knowing HOW to configure a service, it is out of scope. Questions test WHICH service fits, not HOW to use it.

Domain 1 · 24%

Cloud Concepts

The "why cloud?" domain. Covers the value proposition of AWS, design principles, migration strategies, and cost economics.

Task 1.1 — Define the benefits of the AWS Cloud

Knowledge of: Value proposition of the AWS Cloud.

Skills in:

  • Understanding the benefits of global infrastructure (e.g. deployment speed, global reach)
  • Understanding the advantages of high availability, elasticity, and agility

Task 1.2 — Identify design principles of the AWS Cloud

Knowledge of: AWS Well-Architected Framework.

Skills in:

  • Understanding the pillars of the Well-Architected Framework (operational excellence, security, reliability, performance efficiency, cost optimization, sustainability)
  • Identifying differences among the six pillars

Task 1.3 — Understand the benefits of and strategies for migration to the AWS Cloud

Knowledge of: Cloud adoption strategies; resources that support the cloud migration journey.

Skills in:

  • Understanding the components of the AWS Cloud Adoption Framework (AWS CAF) — e.g. reducing business risk; improving ESG performance; increasing revenue; increasing operational efficiency
  • Identifying appropriate migration strategies (e.g. database replication, AWS Snowball)

Task 1.4 — Understand concepts of cloud economics

Knowledge of: Aspects of cloud economics; cost savings from migrating to the cloud.

Skills in:

  • Understanding the role of fixed costs versus variable costs
  • Understanding costs associated with on-premises environments
  • Understanding the difference between licensing strategies (e.g. BYOL vs. included licensing)
  • Understanding the concept of rightsizing
  • Identifying the benefits of automation
  • Understanding economies of scale (e.g. cost savings)
Domain 2 · 30%

Security and Compliance

The highest-value domain after D3. Covers the shared responsibility model, IAM, compliance, and the AWS security service landscape.

Task 2.1 — Understand the AWS shared responsibility model

Knowledge of: AWS shared responsibility model.

Skills in:

  • Recognizing the components of the shared responsibility model
  • Describing customer responsibilities on AWS
  • Describing AWS responsibilities
  • Describing shared responsibilities between customers and AWS
  • Describing how responsibilities can shift depending on the service used (e.g. Amazon RDS vs. AWS Lambda vs. Amazon EC2)

Task 2.2 — Understand AWS Cloud security, governance, and compliance concepts

Knowledge of: AWS compliance and governance concepts; benefits of cloud security (e.g. encryption); where to find security-related logs.

Skills in:

  • Identifying where to find AWS compliance information (e.g. AWS Artifact)
  • Understanding compliance requirements across geographic locations or industries
  • Describing how customers secure resources on AWS (e.g. Amazon Inspector, AWS Security Hub, Amazon GuardDuty, AWS Shield)
  • Identifying encryption options (encryption at rest, encryption in transit)
  • Recognizing services that assist with governance and compliance (e.g. Amazon CloudWatch monitoring; AWS CloudTrail, AWS Audit Manager, AWS Config for auditing)
  • Recognizing compliance requirements that vary across AWS services

Task 2.3 — Identify AWS access management capabilities

Knowledge of: Identity and access management (AWS IAM); importance of protecting the AWS root user account; principle of least privilege; AWS IAM Identity Center (AWS Single Sign-On).

Skills in:

  • Understanding access keys, password policies, and credential storage (e.g. AWS Secrets Manager, AWS Systems Manager)
  • Identifying authentication methods in AWS (e.g. MFA, IAM Identity Center, cross-account IAM roles)
  • Defining groups, users, custom policies, and managed policies to comply with least privilege
  • Identifying tasks only the root account user can perform
  • Understanding methods that can achieve root user protection
  • Understanding types of identity management (e.g. federated)

Task 2.4 — Identify components and resources for security

Knowledge of: Security capabilities provided by AWS; security-related documentation provided by AWS.

Skills in:

  • Describing AWS security features and services (e.g. AWS WAF, AWS Firewall Manager, AWS Shield, Amazon GuardDuty)
  • Understanding that third-party security products are available in AWS Marketplace
  • Identifying where AWS security information is available (e.g. AWS Knowledge Center, AWS Security Center, AWS Security Blog)
  • Understanding how to use AWS services to identify security issues (e.g. AWS Trusted Advisor)
Domain 3 · 34%

Cloud Technology and Services

The broadest domain. Covers the core service categories — compute, database, network, storage, AI/ML, analytics, and a long tail of other services. More than a third of the exam.

Task 3.1 — Define methods of deploying and operating in the AWS Cloud

Knowledge of: Different ways to provision and operate in AWS; different ways to access AWS services; cloud deployment model types.

Skills in:

  • Deciding between programmatic access (API, SDK, CLI), AWS Management Console, and Infrastructure as Code (IaC)
  • Evaluating requirements to determine whether to use one-time operations or repeatable processes
  • Identifying deployment models (cloud, hybrid, on-premises)

Task 3.2 — Define the AWS global infrastructure

Knowledge of: AWS Regions, Availability Zones, and edge locations; high availability; use of multiple Regions; benefits of edge locations.

Skills in:

  • Describing the relationship between Regions, Availability Zones, and edge locations
  • Explaining how to achieve high availability using multiple Availability Zones
  • Recognizing that Availability Zones do not share single points of failure
  • Describing when to use multiple Regions (e.g. disaster recovery, business continuity, low latency for end users, data sovereignty)

Task 3.3 — Identify AWS compute services

Knowledge of: AWS compute services.

Skills in:

  • Recognizing appropriate use cases for different Amazon EC2 instance types (e.g. compute optimized, storage optimized)
  • Recognizing appropriate use cases for different container options (e.g. Amazon ECS, Amazon EKS)
  • Recognizing appropriate use cases for serverless compute options (e.g. AWS Fargate, AWS Lambda)
  • Recognizing that auto scaling provides elasticity
  • Identifying the purpose of load balancers

Task 3.4 — Identify AWS database services

Knowledge of: AWS database services; database migration.

Skills in:

  • Deciding when to use EC2-hosted databases versus AWS managed databases
  • Identifying relational databases (e.g. Amazon RDS, Amazon Aurora)
  • Identifying NoSQL databases (e.g. Amazon DynamoDB)
  • Identifying in-memory databases (e.g. Amazon ElastiCache)
  • Identifying database migration tools (e.g. AWS DMS, AWS Schema Conversion Tool)

Task 3.5 — Identify AWS network services

Knowledge of: AWS networking services.

Skills in:

  • Identifying VPC components (e.g. subnets, gateways)
  • Understanding security within a VPC (e.g. network ACLs, security groups, Amazon Inspector)
  • Understanding the purpose of Amazon Route 53
  • Identifying network connectivity options to AWS (e.g. AWS VPN, AWS Direct Connect)

Task 3.6 — Identify AWS storage services

Knowledge of: AWS storage services.

Skills in:

  • Identifying use cases for object storage
  • Recognizing differences in Amazon S3 storage classes
  • Identifying block storage solutions (e.g. Amazon EBS, instance store)
  • Identifying file storage services (e.g. Amazon EFS, Amazon FSx)
  • Identifying caching file systems (e.g. AWS Storage Gateway)
  • Understanding use cases for lifecycle policies
  • Understanding use cases for AWS Backup

Task 3.7 — Identify AWS AI/ML services and analytics services

Knowledge of: AWS AI/ML services; AWS analytics services.

Skills in:

  • Understanding AI/ML services and the tasks they solve (e.g. Amazon SageMaker AI, Amazon Lex, Amazon Kendra)
  • Identifying data analytics services (e.g. Amazon Athena, Amazon Kinesis, AWS Glue, Amazon QuickSight)

Task 3.8 — Identify services from other in-scope AWS service categories

Knowledge of: Application integration (Amazon EventBridge, Amazon SNS, Amazon SQS); business applications (Amazon Connect, Amazon SES); developer tools (AWS CodeBuild, AWS CodePipeline, AWS X-Ray); end-user computing (Amazon AppStream 2.0, Amazon WorkSpaces); frontend/mobile (AWS Amplify, AWS AppSync); IoT (AWS IoT Core).

Skills in:

  • Selecting the appropriate service for messaging and sending alerts and notifications
  • Selecting the right service for business application needs
  • Selecting the appropriate option for business support
  • Identifying tools for developing, deploying, and troubleshooting applications
  • Identifying services that can present VM output on end-user machines
  • Identifying services that can create and deploy frontend and mobile services
  • Identifying services that manage IoT devices
Domain 4 · 12%

Billing, Pricing, and Support

The smallest domain by weight but a reliable source of easy marks for prepared candidates. Covers pricing models, cost management tools, and AWS Support tiers.

Task 4.1 — Compare AWS pricing models

Knowledge of: Compute purchasing options (On-Demand, Reserved Instances, Spot Instances, AWS Savings Plans, Dedicated Hosts, Dedicated Instances, Capacity Reservations); storage options and tiers.

Skills in:

  • Identifying when to use different compute purchasing options
  • Describing the flexibility of Reserved Instances
  • Describing Reserved Instance behavior in AWS Organizations
  • Understanding data transfer pricing (inbound vs. outbound; same-Region vs. cross-Region)
  • Understanding pricing options for storage options and tiers

Task 4.2 — Understand resources for billing, budget, and cost management

Knowledge of: Billing support and information; AWS service pricing information; AWS Organizations; AWS cost allocation tags.

Skills in:

  • Understanding the appropriate use and capabilities of AWS Budgets and AWS Cost Explorer
  • Understanding the appropriate use and capabilities of the AWS Pricing Calculator
  • Understanding consolidated billing and cost allocation in AWS Organizations
  • Understanding the different types of cost allocation tags and their relationship to billing reports (e.g. AWS Cost and Usage Report)

Task 4.3 — Identify AWS technical resources and AWS Support options

Knowledge of: AWS documentation and resources; AWS Support plans; AWS Partner Network (APN) roles; AWS Support Center.

Skills in:

  • Finding AWS whitepapers, blogs, and documentation
  • Identifying and locating technical AWS resources (e.g. AWS Prescriptive Guidance, AWS Knowledge Center, AWS re:Post)
  • Identifying AWS Support options for customers (Basic, Developer, Business, Enterprise On-Ramp, Enterprise)
  • Identifying the role of AWS Trusted Advisor, AWS Health Dashboard, and AWS Health API for cost optimization and monitoring
  • Identifying the role of the AWS Trust & Safety team for reporting abuse
  • Understanding the role of AWS Partners (e.g. AWS Marketplace, ISVs, systems integrators)
  • Identifying the benefits of being an AWS Partner (e.g. partner training and certification, partner events, volume discounts)
  • Identifying key services offered in AWS Marketplace (e.g. cost management, governance, entitlement)
  • Identifying technical support options available at AWS (e.g. AWS Professional Services, AWS Solutions Architects)
Candidate Prep

How We Help CLF-C02 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 700+.

Who Takes This Exam

  • Career switchers — non-technical professionals moving into cloud roles (project managers, business analysts, sales engineers).
  • Technical beginners — developers or sysadmins new to AWS who want to validate foundational knowledge before taking SAA or DVA.
  • Organization-wide certification programs — companies requiring all staff to hold a foundational cert.

What Candidates Struggle With

  • The shared responsibility model — which layer AWS manages vs. the customer, and how it shifts between EC2, RDS, and Lambda.
  • Service disambiguation — EC2 vs. Fargate vs. Lambda; RDS vs. Aurora vs. DynamoDB; S3 vs. EBS vs. EFS. The exam forces exact choices.
  • Compute purchasing models — On-Demand vs. Reserved vs. Spot vs. Savings Plans. Candidates conflate flexibility with cost.
  • Support plan tiers — confusing Basic/Developer/Business/Enterprise On-Ramp/Enterprise. Who gets a TAM? Who gets a 15-minute response time?
  • Well-Architected Framework pillars — candidates can list them but struggle to match a scenario to the correct pillar.
  • Global infrastructure nuances — when to use multiple AZs (HA) vs. multiple Regions (DR, data sovereignty, latency).

Recommended Study Path

  1. Domain 2 — Security first. Highest weight (30%) and heavily interconnected with every other domain. Build the shared responsibility model mental model before anything else.
  2. Domain 3 — Services breadth. Most questions come from here. Study by category (compute → database → network → storage → AI/ML) in a single sweep, not all at once.
  3. Domain 1 — Cloud concepts. The WHY behind AWS. Well-Architected pillars and CAF are small, well-bounded topics — efficient marks.
  4. Domain 4 — Billing last. Smallest domain (12%). Pricing models and support tiers are pure memorization — the best place to pick up fast marks close to exam day.

AI Explanation Quality for This Exam

CLF-C02 questions use concrete business scenarios. Every wrong answer needs an explanation that says WHY the distractor is wrong — not just "this is not correct." Common patterns to cover:

  • For shared responsibility questions: explain which layer the wrong option gets wrong (e.g. "Security of the cloud is AWS's job; this answer confuses it with security in the cloud, which is the customer's job").
  • For service selection questions: explain the key differentiator that makes the wrong service wrong (e.g. "EBS is block storage and cannot be shared across multiple EC2 instances — that is EFS").
  • For pricing questions: explain what scenario each model is optimized for (e.g. "Spot Instances are for fault-tolerant, flexible workloads — not steady-state production").

Question Writing Rules (CLF-C02 specific)

  • Scenario-based stems — "A company needs to…" or "A developer wants to…" Never "What is the definition of…?"
  • No coding, no configuration — questions test service awareness and cloud concepts, not implementation steps.
  • Service selection over service operation — ask which service fits, not how to set it up.
  • Shared responsibility framing — many D2 questions reduce to: is this AWS's job or the customer's job? Engineer the scenarios to make that choice the crux.
  • Pricing model scenarios — for D4, anchor questions in real usage patterns (spiky demand → On-Demand, predictable steady-state → Reserved, fault-tolerant batch → Spot).

Distractor Quality Guidelines

  • Use real AWS service confusions as distractors: EC2 vs. Lambda vs. Fargate; EBS vs. S3 vs. EFS; GuardDuty vs. Inspector vs. Shield.
  • For shared responsibility questions: make one distractor assume AWS handles more than it does, and one assume the customer handles more than they do.
  • For pricing questions: make each distractor represent a real but wrong use case for the option named (e.g. "On-Demand for 3-year steady-state workload" is wrong but plausible).

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — with a brief conceptual reason
  • Why each distractor is wrong — one sentence per option, naming the specific reason
  • Concept link — which domain/task statement this tests (e.g. "D2 · Task 2.1 — Shared responsibility model")
  • Service context — if a service appears, one line on what it does and what distinguishes it from similar services
