Exam & Question Authoring Standard

Exam-agnostic rules for building high-quality, exam-accurate practice simulations. Generalized from the AIF-C01 simulation design so every new program authors questions the same way.

How to use this page: the shape of a simulation (set size, distribution, type mix) is per-exam and comes from that exam's blueprint (e.g. the Simulation Design section of wiki/aif-c01.html). The rules for writing individual questions on this page are the same for every exam.

Distribution

Domain Distribution

Allocate questions in proportion to the official domain weights. This is the rule that makes a set "representative."

Formula: questions_in_domain = round(total_questions × domain_weight). Compute against both the full set and the scored subset.

Rules

Question Writing Rules

These apply to every exam, every question.

  • Scenario-based stems. "A company wants to…" — not "What is X?" Applied judgment is tested; abstract recall rarely is.
  • Single best answer. For multiple choice there must be exactly one clearly best option. The others are plausible but wrong for a specific, teachable reason.
  • No implementation questions. Never ask candidates to write code, configure step-by-step, or do model math. Test selection and judgment, not configuration.
  • Service knowledge, not service configuration. Questions may name specific services, but test whether the candidate picks the right one — not how to set it up.
  • Ground scenarios in real business impact where the domain calls for it (e.g. responsible-AI questions: hiring bias, loan approval, medical advice, content moderation).
  • Test recognition, not memorization of syntax — e.g. for technique-heavy domains, ask when to apply a technique, not its exact syntax.
Distractors

Distractor Quality

A practice set is only as good as its wrong answers.

  • Each distractor is wrong for a specific, teachable reason — not merely random plausibility.
  • Avoid distractors that are obviously out of scope (e.g. a code snippet in a service-selection question).
  • Lean on real confusions candidates have: similar services mixed up, competing solution approaches confused with each other, similarly-named metrics swapped.
Explanation

Explanation Template

Every question carries an explanation with four parts. This is where the deepest learning happens.

  1. Why the correct answer is right — a brief conceptual reason, not just "this is correct."
  2. Why each distractor is wrong — one sentence per option.
  3. Concept link — which domain / task statement this question tests.
  4. Service context — if a service appears, one line on what it does.
Calibration

Difficulty Calibration

A good question is answerable by a thoughtful candidate who studied the official exam guide, but stumps one who only skimmed. If a question requires deep model-building or implementation knowledge, it is out of scope — cut it.

Authoring checklist

  • ☐ Set size, duration, passing line, and domain distribution match the blueprint
  • ☐ Every stem is a scenario, not a definition
  • ☐ Exactly one best answer (or a precise select-N count)
  • ☐ Every distractor is wrong for a teachable reason
  • ☐ Explanation covers all four parts (correct / distractors / concept link / service context)
  • ☐ No question tests code, configuration, or math
  • ☐ At least one free sample set published
