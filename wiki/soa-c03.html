AWS Certified CloudOps Engineer – Associate (SOA-C03)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified CloudOps Engineer – Associate (SOA-C03) validates the ability to deploy, manage, and operate workloads on AWS. It is the operations-focused associate certification — the successor to the SysOps Administrator – Associate (SOA-C02) — and the only AWS associate exam built around the day-to-day work of running, monitoring, securing, and troubleshooting systems already in production.

  • Target audience: CloudOps engineers, systems administrators, and operations staff who deploy, manage, and operate AWS workloads.
  • Prerequisite experience: About 1 year of hands-on AWS experience in deployment, management, troubleshooting, networking, and security — plus 1 year in a related operations role.
  • Why candidates take it: To validate operational competence on AWS, qualify for CloudOps/SysOps/SRE/DevOps roles, and build toward the DevOps Engineer – Professional certification.
Name change: SOA-C03 renames the credential from "SysOps Administrator – Associate" to "CloudOps Engineer – Associate." The exam code, domains, and operational focus are continuous with the prior version; the new title reflects the modern CloudOps role.

Out-of-Scope Job Tasks (Do NOT Test)

The official exam guide explicitly excludes these. SOA-C03 is an operations exam — it tests running and maintaining systems, not designing or building them. Any question that requires the following is out of scope — cut it.

  • Designing distributed architectures
  • Designing CI/CD pipelines
  • Designing hybrid and multi-VPC networking
  • Developing software
  • Defining security, compliance, and governance requirements
  • Developing ransomware defense strategies
  • Assessing and planning resource capacity
  • Analyzing costs and total cost of ownership
  • Managing billing and invoicing
Exam tip: SOA-C03 tests operation and remediation, not design. The distinction is consistent: a candidate is asked to configure, monitor, troubleshoot, and fix — not to architect from a blank page. If a question reads "design a new multi-Region architecture," it belongs on SAA/SAP, not here. If it reads "an existing system is failing — identify and remediate," it is on target.

Domain 1 · 22%

Monitoring, Logging, Analysis, Remediation, and Performance Optimization

The observability domain. Covers CloudWatch metrics/alarms/dashboards, log collection, event-driven and automated remediation, and tuning compute, storage, and database performance.

Task 1.1 — Implement metrics, alarms, and filters by using AWS monitoring and logging services

Skills in:

  • Configuring AWS monitoring and logging (e.g. Amazon CloudWatch, AWS CloudTrail, Amazon Managed Service for Prometheus)
  • Configuring and managing the CloudWatch agent to collect metrics and logs from EC2 instances, Amazon ECS clusters, or Amazon EKS clusters
  • Configuring, identifying, and troubleshooting CloudWatch alarms that invoke AWS services directly or through Amazon EventBridge (e.g. composite alarms and their invokable actions)
  • Creating and managing customizable, shareable CloudWatch dashboards across multiple accounts and Regions
  • Configuring AWS services to send notifications to Amazon SNS, and invoking alarms that notify SNS

Task 1.2 — Identify and remediate issues by using monitoring and availability metrics

Skills in:

  • Analyzing performance metrics and automating remediation (e.g. CloudWatch, AWS User Notifications, AWS Lambda, AWS Systems Manager, CloudTrail, auto scaling)
  • Using EventBridge to route, enrich, and deliver events, and troubleshooting event bus rules
  • Creating or running custom and predefined Systems Manager Automation runbooks (e.g. via AWS SDKs or custom scripts) to automate tasks

Task 1.3 — Implement performance optimization strategies for compute, storage, and database resources

Skills in:

  • Optimizing compute resources and remediating performance problems using performance metrics, resource tags, and AWS tools
  • Analyzing Amazon EBS performance metrics, troubleshooting issues, and optimizing volume types to improve performance and reduce cost
  • Implementing and optimizing Amazon S3 performance strategies (e.g. AWS DataSync, S3 Transfer Acceleration, multipart uploads, S3 Lifecycle policies)
  • Evaluating and selecting shared storage solutions (e.g. Amazon EFS, Amazon FSx) and optimizing them (e.g. EFS lifecycle policies)
  • Monitoring Amazon RDS metrics (e.g. RDS Performance Insights, CloudWatch alarms) and modifying configurations for efficiency (e.g. Performance Insights proactive recommendations, RDS Proxy)
  • Implementing, monitoring, and optimizing EC2 instances and associated storage and networking (e.g. EC2 placement groups)
Domain 2 · 22%

Reliability and Business Continuity

The "keep it running, and recover when it doesn't" domain. Covers scaling and elasticity, high availability and fault tolerance, and backup, restore, and disaster recovery.

Task 2.1 — Implement scalability and elasticity

Skills in:

  • Configuring and managing scaling mechanisms in compute environments (e.g. EC2 Auto Scaling)
  • Implementing caching to enhance dynamic scalability (e.g. Amazon CloudFront, Amazon ElastiCache)
  • Configuring and managing scaling in AWS managed databases (e.g. Amazon RDS, Amazon DynamoDB)

Task 2.2 — Implement highly available and resilient environments

Skills in:

  • Configuring and troubleshooting Elastic Load Balancing (ELB) and Amazon Route 53 health checks
  • Configuring fault-tolerant systems (e.g. Multi-AZ deployments)

Task 2.3 — Implement backup and restore strategies

Skills in:

  • Automating snapshots and backups for AWS resources (e.g. EC2 instances, RDS DB instances, EBS volumes, S3 buckets, DynamoDB tables) using AWS Backup
  • Restoring databases by various methods (e.g. point-in-time restore) to meet RTO, RPO, and cost requirements
  • Implementing versioning for storage services (e.g. Amazon S3, Amazon FSx)
  • Following disaster recovery procedures
Domain 3 · 22%

Deployment, Provisioning, and Automation

The infrastructure-as-code and automation domain. Covers AMIs and images, CloudFormation/CDK, multi-account and multi-Region provisioning, deployment strategies, and automating the management of existing resources.

Task 3.1 — Provision and maintain cloud resources

Skills in:

  • Creating and managing AMIs and container images (e.g. EC2 Image Builder)
  • Creating and managing stacks of resources with AWS CloudFormation and the AWS CDK
  • Identifying and remediating deployment issues (e.g. subnet sizing, CloudFormation errors, permissions issues)
  • Provisioning and sharing resources across multiple Regions and accounts (e.g. AWS RAM, CloudFormation StackSets)
  • Implementing deployment strategies and services
  • Using and managing third-party tools to automate resource deployment (e.g. Terraform, Git)

Task 3.2 — Automate the management of existing resources

Skills in:

  • Using AWS services to automate operational processes (e.g. AWS Systems Manager — State Manager, Patch Manager, Maintenance Windows)
  • Implementing event-driven automation (e.g. AWS Lambda, Amazon S3 Event Notifications)
Domain 4 · 16%

Security and Compliance

The operational security domain. Covers IAM and access troubleshooting, multi-account security, compliance enforcement, and protecting data with encryption, secrets management, and the AWS security service suite.

Task 4.1 — Implement and manage security and compliance tools and policies

Skills in:

  • Implementing IAM features (e.g. password policies, MFA, roles, federated identity, resource policies, policy conditions)
  • Troubleshooting and auditing access issues (e.g. AWS CloudTrail, IAM Access Analyzer, IAM policy simulator)
  • Implementing multi-account strategies securely (e.g. AWS Organizations, SCPs)
  • Implementing remediation based on AWS Trusted Advisor security checks
  • Enforcing compliance requirements (e.g. Region and service selections)

Task 4.2 — Implement strategies to protect data and infrastructure

Skills in:

  • Implementing and enforcing a data classification scheme
  • Implementing, configuring, and troubleshooting encryption at rest (e.g. AWS KMS)
  • Implementing, configuring, and troubleshooting encryption in transit (e.g. AWS Certificate Manager)
  • Securely storing secrets using AWS services (e.g. AWS Secrets Manager, Systems Manager Parameter Store)
  • Configuring reports and remediating findings from AWS services (e.g. AWS Security Hub, Amazon GuardDuty, AWS Config, Amazon Inspector)
Domain 5 · 18%

Networking and Content Delivery

The networking operations domain. Covers VPC configuration and connectivity, DNS and content delivery, and — heavily — troubleshooting connectivity using logs and network monitoring.

Task 5.1 — Implement and optimize networking features and connectivity

Skills in:

  • Configuring a VPC (e.g. subnets, route tables, network ACLs, security groups, NAT gateways, internet gateway, egress-only internet gateway)
  • Configuring private networking connectivity (e.g. VPC peering, VPC endpoints, AWS PrivateLink, Transit Gateway)
  • Auditing AWS network protection services in a single account (e.g. Route 53 Resolver DNS Firewall, AWS WAF, AWS Shield, AWS Network Firewall)
  • Optimizing the cost of network architectures (e.g. NAT gateway placement, VPC endpoints, data transfer)

Task 5.2 — Configure domains, DNS services, and content delivery

Skills in:

  • Configuring DNS (e.g. Route 53 Resolver)
  • Implementing Route 53 routing policies, configurations, and query logging
  • Configuring content and service distribution (e.g. Amazon CloudFront, AWS Global Accelerator)

Task 5.3 — Troubleshoot network connectivity issues

Skills in:

  • Troubleshooting VPC configurations (e.g. subnets, route tables, network ACLs, security groups, transit gateways, NAT gateways)
  • Collecting and interpreting networking logs (e.g. VPC flow logs, ELB access logs, AWS WAF web ACL logs, CloudFront logs, container logs)
  • Identifying and remediating CloudFront caching issues
  • Identifying and troubleshooting hybrid connectivity and private connectivity issues
  • Configuring and analyzing Amazon CloudWatch network monitoring services (e.g. Internet Monitor, Network Monitor)
Candidate Prep

How We Help SOA-C03 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 720+.

Who Takes This Exam

  • Operations engineers and sysadmins — professionals running production AWS workloads who want to validate operational competence.
  • Cloud / DevOps / SRE engineers — practitioners who deploy and maintain systems and need a recognized operations credential.
  • Associate-level progressors — candidates who hold or are pursuing SAA and want the operations counterpart, often on the path to DevOps Professional.

What Candidates Struggle With

  • CloudWatch depth — metric vs. alarm vs. composite alarm; the CloudWatch agent for custom metrics (memory, disk) that the hypervisor does not expose; metric filters on logs vs. Logs Insights queries.
  • Remediation automation — wiring CloudWatch alarm → EventBridge → SSM Automation runbook or Lambda. Candidates know the pieces but not how they connect.
  • EBS volume types — gp3 vs. gp2 vs. io2 vs. st1/sc1; baseline vs. burst IOPS; when throughput-optimized beats provisioned IOPS.
  • Backup & DR math — matching RTO/RPO to the right strategy (backup & restore vs. pilot light vs. warm standby vs. multi-site) and to point-in-time restore.
  • CloudFormation operations — stack failure and rollback behavior, drift, StackSets across accounts/Regions, and reading deployment errors (subnet sizing, IAM permissions).
  • VPC troubleshooting — security groups (stateful) vs. network ACLs (stateless); route table and NAT gateway problems; reading VPC flow logs to find the broken hop.
  • Route 53 routing policies — simple vs. weighted vs. latency vs. failover vs. geolocation vs. multivalue, and how health checks drive failover.

Recommended Study Path

  1. Domain 1 — Monitoring first. Tied for highest weight (22%) and foundational to everything else — you cannot remediate what you cannot observe. Master CloudWatch, the agent, alarms, and the alarm→EventBridge→automation chain.
  2. Domain 2 — Reliability. Scaling, Multi-AZ, ELB/Route 53 health checks, and AWS Backup with RTO/RPO. Operationally adjacent to D1 and 22% of the exam.
  3. Domain 3 — Deployment & automation. CloudFormation/CDK, StackSets, Image Builder, and Systems Manager. 22% — heavy on reading errors and knowing what each automation tool does.
  4. Domain 5 — Networking. 18%, and the troubleshooting task (5.3) is dense. Drill SG vs. NACL, route tables, and reading flow logs.
  5. Domain 4 — Security last. Smallest domain (16%) and largely reinforced by the others. IAM troubleshooting, KMS, Secrets Manager, and the GuardDuty/Security Hub/Config/Inspector quartet.

AI Explanation Quality for This Exam

SOA-C03 questions are operational and scenario-driven. Every wrong answer needs an explanation that says WHY the distractor is wrong in an operations context — not just "this is not correct." Common patterns to cover:

  • For monitoring questions: explain why a given metric or tool does (or does not) surface the symptom (e.g. "the hypervisor cannot see in-guest memory — you need the CloudWatch agent, not a default EC2 metric").
  • For troubleshooting questions: explain the exact failure mechanism (e.g. "a security group denies by default and is stateful, so the return traffic is allowed automatically — the block is the NACL's missing ephemeral-port rule").
  • For remediation questions: explain which automation primitive fits (e.g. "an SSM Automation runbook can take the remediation action directly; EventBridge only routes the event to it").

Question Writing Rules (SOA-C03 specific)

  • Operational scenario stems — "A CloudOps engineer notices…", "An application is returning 5xx errors…", "Backups must meet a 1-hour RPO…" Never "What is Amazon CloudWatch?"
  • Operate and remediate, do not design — frame around an existing system that needs monitoring, fixing, scaling, or recovering, not a greenfield design.
  • Troubleshooting framing — many D1/D5 questions present a symptom and ask for the cause or the fix. Engineer the scenario so the diagnostic reasoning is the crux.
  • Tool disambiguation — force exact choices among operationally similar tools (SSM Automation vs. Lambda; metric filter vs. Logs Insights; security group vs. NACL; gp3 vs. io2).
  • RTO/RPO and cost trade-offs — for D2, anchor questions in concrete recovery objectives and let cost break the tie between strategies that both meet the objective.

Distractor Quality Guidelines

  • Use real operational confusions as distractors: CloudWatch agent vs. default metrics; composite vs. metric alarm; AWS Backup vs. manual snapshots; gp3 vs. provisioned IOPS; weighted vs. latency routing.
  • For troubleshooting questions: make one distractor a plausible-but-wrong layer (e.g. blame the security group when the real fault is the route table or NACL).
  • For remediation questions: make each distractor a real service that almost fits but is the wrong automation primitive for the trigger described.

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — with a brief operational reason
  • Why each distractor is wrong — one sentence per option, naming the specific reason
  • Concept link — which domain/task statement this tests (e.g. "D1 · Task 1.2 — Identify and remediate issues")
  • Service context — if a service appears, one line on what it does and what distinguishes it from the similar services it is commonly confused with
