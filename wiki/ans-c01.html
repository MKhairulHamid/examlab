AWS Certified Advanced Networking – Specialty (ANS-C01)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified Advanced Networking – Specialty (ANS-C01) validates a candidate's ability to design, implement, manage, and secure AWS and hybrid network architectures at scale. It is a Specialty-level exam — the deepest networking credential AWS offers — testing whether a candidate can build hybrid and cloud-native connectivity, implement core networking services to best practice, operate and troubleshoot complex topologies, automate networking with infrastructure as code, and secure traffic end to end.

  • Target audience: Network engineers, cloud network architects, and hybrid-connectivity specialists who design and operate AWS networking at scale.
  • Prerequisite experience: 5 or more years of networking experience with 2 or more years of cloud and hybrid networking experience — one of the most experience-intensive certifications in the catalog. Working knowledge of AWS networking nuances, AWS security best practices, and AWS compute and storage options is expected.
  • Why candidates take it: To validate specialized, hands-on networking capability for senior network engineer and cloud network architect roles, and as the networking-focused capstone alongside the Solutions Architect and DevOps tracks.

Out-of-Scope Topics (Do NOT Test)

The exam focuses on AWS and hybrid networking. Services and tasks outside the networking specialist's role are out of scope — any question that hinges on them should be cut. Representative out-of-scope areas:

  • Application, analytics, and machine-learning services not central to networking (e.g. Amazon SageMaker AI, Amazon Rekognition, AWS Glue, Amazon QuickSight)
  • Designing cryptographic algorithms or performing packet payload decryption
  • Application-level software development beyond network automation
  • End-user compute and desktop management unrelated to connectivity
Exam tip: ANS-C01 tests designing, building, operating, and securing networks on AWS — DNS, load balancing, edge, hybrid connectivity (Direct Connect, VPN, Transit Gateway, BGP), multi-account/VPC routing, monitoring, and network security. The crux of most questions is "which connectivity service or configuration best meets the performance, availability, security, and cost requirement?"

Domain 1 · 30%

Network Design

The largest domain. Designing edge acceleration, DNS, load balancing, observability, and the hybrid and multi-account routing topologies that connect on-premises networks, Regions, accounts, and VPCs.

Task 1.1 — Design a solution that incorporates edge network services to optimize user performance and traffic management for global architectures

Knowledge of:

  • Design patterns for content distribution networks (e.g. Amazon CloudFront)
  • Design patterns for global traffic management (e.g. AWS Global Accelerator)
  • Integration patterns for CDNs and global traffic management with other services (e.g. ELB, Amazon API Gateway)

Skills in: Evaluating requirements of global inbound and outbound internet traffic to design an appropriate content distribution solution.

Task 1.2 — Design DNS solutions that meet public, private, and hybrid requirements

Knowledge of:

  • DNS protocol (e.g. DNS records, TTL, DNSSEC, DNS delegation, zones)
  • DNS logging and monitoring
  • Amazon Route 53 features (e.g. alias records, traffic policies, resolvers, health checks)
  • Integration of Route 53 with other AWS networking services (e.g. Amazon VPC), and with hybrid, multi-account, and multi-Region options
  • Domain registration

Skills in: Using Route 53 public and private hosted zones, Resolver endpoints in hybrid and AWS architectures, Route 53 for global traffic management, and creating and managing domain registrations.

Task 1.3 — Design solutions that integrate load balancing to meet high availability, scalability, and security requirements

Knowledge of:

  • How load balancing works at layer 3, layer 4, and layer 7 of the OSI model
  • Different types of load balancers and how they meet design, HA, and security requirements; internal vs. external patterns; scaling factors
  • Integrations with other AWS services (e.g. Global Accelerator, CloudFront, AWS WAF, Route 53, Amazon EKS, ACM)
  • Configuration options (e.g. proxy protocol, cross-zone, session affinity, routing algorithms) and target groups (TCP, GENEVE, IP vs. instance)
  • AWS Load Balancer Controller for Kubernetes clusters; encryption and authentication (TLS termination vs. TLS passthrough)

Skills in: Selecting an appropriate load balancer for the use case, integrating auto scaling, and integrating load balancers with existing application deployments.

Task 1.4 — Define logging and monitoring requirements across AWS and hybrid networks

Knowledge of:

  • Amazon CloudWatch metrics, agents, logs, alarms, dashboards, and insights for visibility
  • AWS Transit Gateway Network Manager and VPC Reachability Analyzer for visibility
  • Flow logs and traffic mirroring; access logging (e.g. load balancers, CloudFront)

Skills in: Identifying logging and monitoring requirements, recommending metrics that show network status, and capturing baseline network performance.

Task 1.5 — Design a routing strategy and connectivity architecture between on-premises networks and the AWS Cloud

Knowledge of:

  • Routing fundamentals (e.g. dynamic vs. static, BGP)
  • Layer 1 and layer 2 concepts for physical interconnects (e.g. VLAN, LAG, optics, jumbo frames)
  • Encapsulation and encryption technologies (e.g. GRE, IPsec); resource sharing across accounts; overlay networks

Skills in: Identifying hybrid connectivity requirements; designing redundant hybrid connectivity (Direct Connect, Site-to-Site VPN); designing BGP routing with attributes for load sharing and active/passive; designing SD-WAN integration (Transit Gateway Connect, overlay networks).

Task 1.6 — Design a routing strategy and connectivity architecture that include multiple AWS accounts, Regions, and VPCs

Knowledge of:

  • Connectivity patterns and use cases (e.g. VPC peering, Transit Gateway, AWS PrivateLink)
  • Capabilities and advantages of VPC sharing
  • IP subnets and solutions for IP address overlaps

Skills in: Connecting multiple VPCs with the most appropriate services; using VPC sharing in a multi-account setup; managing IP overlaps (NAT, PrivateLink, Transit Gateway routing).

Domain 2 · 26%

Network Implementation

Building what Domain 1 designs. Implementing hybrid connectivity, multi-account and multi-VPC topologies, complex DNS, and automating network infrastructure with infrastructure as code.

Task 2.1 — Implement routing and connectivity between on-premises networks and the AWS Cloud

Knowledge of:

  • Routing protocols (static, dynamic); VPNs (security, accelerated VPN)
  • Layer 1 hardware (LOA documents, colocation facilities, Direct Connect); layer 2/3 (VLANs, IP addressing, gateways, routing, switching)
  • Traffic management and SD-WAN (Transit Gateway Connect); DNS (conditional forwarding, hosted zones, resolvers); security appliances; load balancing (L4 vs. L7, reverse proxies)
  • Infrastructure automation; AWS Organizations and AWS RAM (multi-account Transit Gateway, Direct Connect, VPC, Route 53); connectivity testing (Route Analyzer, Reachability Analyzer); VPC networking services

Skills in: Configuring physical hybrid requirements; configuring static/dynamic routing; connecting on-premises networks and name resolution to AWS; configuring load balancing, monitoring, and logging; testing and validating connectivity between environments.

Task 2.2 — Implement routing and connectivity across multiple AWS accounts, Regions, and VPCs

Knowledge of:

  • Inter-VPC and multi-account connectivity (VPC peering, Transit Gateway, VPN, third-party vendors, SD-WAN, MPLS); private application connectivity (PrivateLink)
  • Expanding connectivity (Organizations, AWS RAM); host and service name resolution (DNS); infrastructure automation
  • Authentication and authorization (SAML, Active Directory); security (security groups, network ACLs, AWS Network Firewall); connectivity testing (Route Analyzer, Reachability Analyzer)

Skills in: Configuring single- and multi-VPC architectures (DHCP, routing, security groups); configuring hybrid connectivity with third-party solutions; configuring hub-and-spoke (Transit Gateway, transit VPC); configuring DNS for hybrid connectivity; implementing security between network boundaries; configuring monitoring and logging.

Task 2.3 — Implement complex hybrid and multi-account DNS architectures

Knowledge of:

  • When to use private vs. public hosted zones; traffic management methods (latency, geography, weighting)
  • DNS delegation and forwarding (conditional forwarding); record types (A, AAAA, TXT, PTR, alias); DNSSEC
  • Sharing DNS services between accounts (AWS RAM); requirements for inbound and outbound endpoints

Skills in: Configuring DNS zones and conditional forwarding; configuring traffic management with DNS; configuring DNS for hybrid networks; configuring appropriate records; configuring DNSSEC on Route 53; configuring DNS in centralized or distributed architectures; configuring DNS monitoring and logging.

Task 2.4 — Automate and configure network infrastructure

Knowledge of:

  • Infrastructure as code (AWS CDK, CloudFormation, AWS CLI, AWS SDK, APIs)
  • Event-driven network automation
  • Common problems of hardcoded instructions in IaC templates when provisioning networking resources

Skills in: Creating repeatable network configurations; integrating event-driven networking functions; integrating hybrid automation with AWS native IaC; eliminating risk and achieving efficiency at the lowest cost; automating optimization of cloud network resources with IaC.

Domain 3 · 20%

Network Management and Operation

Keeping the network healthy. Maintaining routing and connectivity, monitoring and analyzing traffic to troubleshoot, and optimizing for performance, reliability, and cost.

Task 3.1 — Maintain routing and connectivity on AWS and hybrid networks

Knowledge of:

  • Industry-standard routing protocols in hybrid networks (e.g. BGP over Direct Connect)
  • Connectivity methods (Direct Connect gateway, Transit Gateway, VIFs); how limits and quotas affect networking services (bandwidth, route limits)
  • Private and public access methods for custom services (PrivateLink, VPC peering); inter-Regional and intra-Regional communication patterns

Skills in: Managing routing protocols over Direct Connect and VPN; maintaining private access to custom services; using route tables and propagation to direct traffic; setting up private or public access to AWS services; optimizing routing (route summarization, CIDR overlap).

Task 3.2 — Monitor and analyze network traffic to troubleshoot and optimize connectivity patterns

Knowledge of:

  • Network performance metrics and reachability constraints (e.g. routing, packet size)
  • Appropriate logs and metrics for performance and reachability issues (e.g. packet loss)
  • Tools to collect and analyze logs and metrics (CloudWatch, VPC Flow Logs, VPC Traffic Mirroring); tools to analyze routing (Reachability Analyzer, Transit Gateway Network Manager)

Skills in: Analyzing tool output to troubleshoot connectivity; mapping topology (Transit Gateway Network Manager); analyzing packets for shaping issues (Traffic Mirroring); troubleshooting misconfiguration and verifying design intent (Reachability Analyzer); automating verification of connectivity intent; resolving packet-size/MTU mismatches.

Task 3.3 — Optimize AWS networks for performance, reliability, and cost-effectiveness

Knowledge of:

  • When a VPC peer vs. a transit gateway is appropriate; methods to reduce bandwidth (unicast vs. multicast, CloudFront)
  • Cost-effective data-transfer options between a VPC and on-premises; different network interface types; Route 53 high-availability features (health checks with latency/weighted records)
  • Load balancing and traffic distribution patterns; VPC subnet optimization; frame size optimization across connection types

Skills in: Optimizing throughput; selecting the right interface (ENI, ENA, EFA); choosing peering vs. proxy vs. transit gateway; implementing multicast in VPC and on-premises; using Route 53 zones/records for availability; optimizing subnets for auto scaling and to prevent IP depletion (secondary CIDR); configuring jumbo frames; using Global Accelerator to improve performance and availability.

Domain 4 · 24%

Network Security, Compliance, and Governance

Securing the network. Implementing security and compliance features, validating and auditing with monitoring and logging, and keeping data and communications confidential in transit.

Task 4.1 — Implement and maintain network features to meet security and compliance needs

Knowledge of:

  • Different threat models based on application architecture; common security threats
  • Mechanisms to secure different application flows; AWS network architecture that meets security and compliance requirements

Skills in: Securing inbound flows (AWS WAF, AWS Shield, Network Firewall); securing outbound flows (Network Firewall, proxies, Gateway Load Balancers); securing inter-VPC traffic (security groups, network ACLs, VPC endpoint policies); implementing architectures (untrusted network, perimeter VPC, three-tier); developing threat models and mitigations; testing compliance (failover, resiliency); automating security incident reporting and alerting.

Task 4.2 — Validate and audit security by using network monitoring and logging services

Knowledge of:

  • Monitoring and logging services (CloudWatch, AWS CloudTrail, VPC Traffic Mirroring, VPC Flow Logs, Transit Gateway Network Manager); alert mechanisms (CloudWatch alarms)
  • Log creation in different services (VPC flow logs, load balancer access logs, CloudFront access logs); log delivery (Amazon Kinesis, Route 53, CloudWatch)
  • Mechanisms to audit network security configurations (security groups, AWS Firewall Manager, AWS Trusted Advisor)

Skills in: Creating and analyzing VPC flow logs (base and extended fields); creating and analyzing traffic mirroring; implementing automated alarms and custom metrics in CloudWatch; correlating across single or multiple log sources; implementing log-delivery solutions; implementing a network audit strategy across services and accounts (Firewall Manager, security groups, network ACLs).

Task 4.3 — Implement and maintain confidentiality of data and communications of the network

Knowledge of:

  • Network encryption options available on AWS; VPN connectivity over Direct Connect; encryption methods for data in transit (IPsec)
  • Network encryption under the AWS shared responsibility model; security methods for DNS communications (DNSSEC)

Skills in: Implementing network encryption to meet compliance (IPsec, TLS); securing data in transit (CloudFront, ALB/NLB, VPN over Direct Connect, managed databases, Amazon S3, custom EC2 solutions, Transit Gateway); implementing certificate management with a CA (ACM, AWS Private CA); implementing secure DNS communications.

Candidate Prep

How We Help ANS-C01 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 700+.

Who Takes This Exam

  • Network engineers — already running BGP, DNS, and load balancing on-premises who want to validate AWS and hybrid depth.
  • Cloud network architects — designing multi-account, multi-Region landing zones with Transit Gateway, Direct Connect, and PrivateLink.
  • Hybrid-connectivity specialists — building and operating Direct Connect, Site-to-Site VPN, and SD-WAN integrations at scale.

What Candidates Struggle With

  • Hybrid connectivity choices — Direct Connect vs. Site-to-Site VPN vs. both for redundancy, public vs. private vs. transit VIFs, Direct Connect gateway, and MACsec.
  • Transit Gateway vs. peering vs. PrivateLink — which to use for a given inter-VPC, multi-account, or service-exposure requirement, and how route tables and propagation behave.
  • BGP routing control — influencing path selection with AS path prepending, local preference, MED, and longest-prefix match for active/passive and load-shared designs.
  • Hybrid and multi-account DNS — Route 53 Resolver inbound/outbound endpoints, conditional forwarding, private hosted zones shared via RAM, and centralized vs. distributed DNS.
  • Route 53 routing policies — simple, weighted, latency, failover, geolocation, geoproximity, and multivalue, plus alias records and health checks.
  • Network security layering — WAF vs. Shield vs. Network Firewall vs. Gateway Load Balancer vs. security groups/NACLs, and where each belongs in inbound, outbound, and east-west flows.
  • Troubleshooting — Reachability Analyzer, VPC Flow Logs, Traffic Mirroring, and MTU/jumbo-frame and packet-size mismatches.

Recommended Study Path

  1. Domain 1 — Network Design first. The largest domain (30%) and the foundation: edge, DNS, load balancing, observability, and hybrid and multi-account routing topologies.
  2. Domain 2 — Network Implementation. 26%. Turn the designs into working hybrid connectivity, multi-account/VPC topologies, complex DNS, and automation.
  3. Domain 4 — Security, Compliance, and Governance. 24%. Secure inbound/outbound/east-west flows, audit with monitoring and logging, and encrypt data in transit.
  4. Domain 3 — Management and Operation. 20%. Maintain routing, troubleshoot with the analysis tools, and optimize for performance, reliability, and cost.

AI Explanation Quality for This Exam

ANS-C01 questions are scenario-based and frequently offer several connectivity options that "work," where one is the best fit on availability, performance, security, or cost. Every explanation must cover WHY a plausible option is inferior — not just "this is wrong." Common patterns to cover:

  • For connectivity questions: explain why an option does not scale or meet HA (e.g. "VPC peering is non-transitive, so a full mesh of peers does not scale the way a Transit Gateway hub does").
  • For DNS questions: explain why a resolver direction is wrong (e.g. "an inbound endpoint lets on-premises resolve AWS records; resolving on-premises names from AWS needs an outbound endpoint with forwarding rules").
  • For security questions: explain why a control is in the wrong layer (e.g. "a security group cannot inspect HTTP request content for SQLi — that is AWS WAF at layer 7").

Question Writing Rules (ANS-C01 specific)

  • Scenario stems — "A network engineer must…" Establish the real constraint (availability, throughput, latency, hybrid redundancy, security, cost) before asking.
  • Best-of-several-workable answers — most distractors are valid networking services but lose on scalability, transitivity, layer, or one unmet requirement. Engineer the crux to be the trade-off.
  • Topology and routing framing — the right answer is usually the most scalable, resilient, purpose-fit connectivity design. Make non-transitive, single-path, or wrong-layer options the plausible-but-wrong choices.
  • Cross-domain wiring — ANS scenarios routinely span design, implementation, DNS, and security. Let questions combine services as the real exam does.
  • No cryptographic algorithm design, no packet-payload analysis, no non-networking ML/analytics — keep questions inside the in-scope networking service set.

Distractor Quality Guidelines

  • Use real networking confusions as distractors: Transit Gateway vs. VPC peering vs. PrivateLink; Direct Connect vs. Site-to-Site VPN; inbound vs. outbound Resolver endpoints; WAF vs. Shield vs. Network Firewall; ALB vs. NLB vs. GWLB.
  • For connectivity questions: make one distractor a design that does not scale (full-mesh peering) and one that breaks on transitivity or overlapping CIDRs.
  • For DNS questions: make one distractor the wrong resolver direction and one the wrong routing policy for the requirement.
  • For security questions: make one distractor a control in the wrong OSI layer and one that leaves the origin or an east-west path exposed.

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — the conceptual reason it best meets every stated constraint
  • Why each distractor is wrong — one sentence per option, naming the specific unmet constraint or limitation
  • Concept link — which domain/task statement this tests (e.g. "D1 · Task 1.6 — Multi-account/Region/VPC routing")
  • Service context — if a service appears, one line on what it does and what distinguishes it from the similar services in the distractors
