AWS Cert Roadmap

New Program Playbook

The end-to-end checklist for launching a new certification program on CloudExamLab — from exam blueprint to shipped course. This is the spine that ties together every other standard in this section.

What "a program" means here: one certification (e.g. AWS Certified AI Practitioner, Solutions Architect Associate). A complete program has five shippable layers: an exam blueprint, a study course, interactive widgets, an exam simulation, and the platform wiring that exposes it to learners. Reference implementations: AIF-C01 (original), CLF-C02, and SAA-C03 (first build to use the 4-step plan, Multiple-Response questions, and curated videos).
This page is self-contained. The only thing you need from the user is the official exam guide URL. Every concrete shape — the course registry pattern, the Supabase table columns and gotchas, the scaffold SQL, the ai_cache structure, Multiple-Response rules, validation queries, the widget pattern, and video curation — is documented here in Build Reference. You should never need to reverse-engineer an existing program. Start with the 4-step plan.
Overview

The Five Layers

Build in this order. Each layer depends on the one above it — don't author sessions before the blueprint is locked, and don't write questions before the sessions exist.

Planning

Planning the Build — Four Token-Balanced Steps

The five layers cost wildly different amounts. Before starting, split the whole build into four work-steps so each fits one working session and ends on a clean, committable checkpoint. This is the plan to propose to the user.

Why split at all

Two layers dominate token usage: the study course (~16 richly-structured sessions) and the exam simulation (205 questions, each carrying a full ai_cache payload). The blueprint, widgets, and wiring are comparatively light. Doing everything in one pass exhausts the context window and risks quality drift. The fix is to spread the two giants across four steps that each carry roughly a quarter of the work.

The canonical 4-step split

Run one step per working session. Commit at the end of each. Keep the app building at every checkpoint.

StepScope (layers)Deliverables
1 — Blueprint + Course ALayer 1 (all) + Layer 2 first halfFetch the official guide → wiki/<exam>.html; add roadmap entry + sidebars; author the first half of the study sessions (roughly the first two domains) in src/data/<exam>Course.js
2 — Course B + WidgetsLayer 2 second half + Layer 3Author the remaining study sessions; build the interactive widgets; register the course in src/utils/sessionCourses.js; confirm the Node-20 build passes
3 — Simulation ALayer 4 scaffold + first questionsInsert the exam_types row (is_active = false) + 4 question_sets rows; author the Free sample (10 Qs) + Simulation 1 "Foundational" (65 Qs), each with full ai_cache
4 — Simulation B + LaunchLayer 4 finish + Layer 5 + QAAuthor Simulation 2 "Architectural" + Simulation 3 "Comparative" (65 Qs each); validate every set; set is_active = true; verify, build, and run the Definition of Done
Token-balance rationale: the 205 questions are the single biggest sink, so they fill Steps 3–4 (≈75 then ≈130 questions). The ~16 study sessions fill Steps 1–2. The light layers (blueprint, widgets, wiring) ride along in the bookend steps. If Step 4 feels heavy, the lever is to move Simulation 2 into Step 3.

Rules that keep the steps clean

  • One step = one session. Each step is sized to a single session's context budget. Record progress and the Supabase set IDs in memory so the next session resumes cold.
  • Keep exam_types.is_active = false until Step 4. The exam stays hidden from the dashboard while you build; flipping it to true is the launch act.
  • Steps 3–4 are pure Supabase — no repo changes — so they need no redeploy; the live site reads questions from the database. Steps 1–2 are code and must be committed + pushed to deploy.
  • The only user input required is the official exam guide URL (provided at Step 1). Everything else is on this page and the linked standards.
Layer 1

Exam Blueprint

Before writing a single session, produce the exam reference page. It is the single source of truth every later layer reads from.

Source of truth

  • Start from the official exam guide PDF — never secondary blogs. Link it in the page footer (see the "Official" group in aif-c01.html).
  • Transcribe the domain list, task statements, and domain weights verbatim. These percentages drive the question distribution in Layer 4.
  • Record the scoring model (scaled score, passing line, compensatory vs. per-domain) and the out-of-scope skills — what the exam explicitly does NOT test bounds your question difficulty.

Required sections (mirror aif-c01.html)

  1. What this exam is — audience, prerequisites, why someone takes it.
  2. Question format & scoring — question types, count, duration, passing threshold, out-of-scope list.
  3. Domain breakdown — one subsection per domain, each task statement with its testable concepts.
  4. Helping candidates — who struggles, common gaps, recommended study path.
  5. Simulation design — the per-exam parameters consumed by the authoring standard (set size, distribution table, type mix).
Do not skip the domain weights table. Layer 4 derives the exact number of questions per domain from it. Getting the weights wrong here propagates into an unrepresentative simulation.
Layer 2

Study Course

Author the sessions. This layer is fully standardized — follow the Study Materials Standard and its companion skill.

Quick reference

  • One data file per exam: src/data/<exam>Course.js. AIF-C01 → aifC01Course.js, 16 sessions.
  • Each session is the full schema: preLearningCheck → teaching sections → microQuizzes (with elaborativePrompt) → selfExplanationPrompt → sample → videos/keyTerms/awsServices/examTips.
  • Authoring skill: ~/.claude/skills/create-study-session.md — invoke it per session; it encodes the schema and writing rules.
  • Renderer: src/components/study/SessionCourse.jsx — no per-exam code needed; it renders any course that follows the schema.

How many sessions?

One session per task statement is the default. Group trivially small tasks; split a task into two sessions when it carries a heavy domain weight. Target 25–35 min per session.

Layer 3

Interactive Widgets

Add inline interactive widgets for the concepts that are hard to grasp from prose alone. Optional per session, but high-leverage. Follow the Interactive Widget Standard.

  • Build a widget only when there is one core insight that direct manipulation makes unavoidable (precision/recall trade-off, tokenization, RAG vs. fine-tune).
  • Register it: INTERACTIVE_WIDGETS = { 'key': Component } in SessionCourse.jsx, then add interactive: 'key' to the section in the course data.
  • Do not gate the course on widgets — sessions must teach fully without them. Widgets reinforce; they don't carry the lesson.
AIF-C01 shipped 7 widgets across 16 sessions — roughly one per two sessions, only where a concept genuinely benefits. That ratio is a sane default, not a quota.
Layer 4

Exam Simulation

Author the practice question sets that mimic the real exam. Questions must be derived directly from the official exam guide and follow the Exam & Question Authoring Standard in full — including per-option explanations, concept guides, and official links in every question's ai_cache.

Step 1 — Read the official exam guide before writing a single question

Every question must be grounded in a specific task statement or knowledge/skill from the official AWS exam guide for that certification. Deriving questions from memory or general AWS knowledge produces off-target questions that mislead learners.

  • Read every domain's task statements and knowledge/skills list in the official guide (the same guide used to write the blueprint in Layer 1).
  • Map each question to a specific domain → task → knowledge/skill. If you cannot cite the mapping, the question is not exam-accurate.
  • Use the blueprint's Simulation Design section (domain distribution table) to set the exact question count per domain for each set.

Step 2 — Set up the 4 question sets in Supabase first

Insert the exam_types row and all 4 question_sets rows before writing any questions. Questions reference question_set_id — the sets must exist first.

Step 3 — Plan topic coverage before writing

Before writing any question text, build a topic list that covers every task statement in the exam guide with no gaps and no duplicates within a set. Each question covers a distinct subtopic.

  • For the free practice set: distribute proportionally across all domains (e.g. 10 questions at 24/30/34/12% weights ≈ 2/3/4/1).
  • For each full simulation: cover every task statement at least once. With 65 questions across typically 10–14 task statements, plan 4–6 questions per task statement.
  • Each of the 3 full simulations must test the same topic coverage but from a different angle — write a topic coverage plan for each sim before generating questions:
SimulationAngleQuestion style
Simulation 1Foundational — know the services and concepts"Which AWS service does X?" · "Which benefit describes Y?"
Simulation 2Architectural — apply knowledge to solve a business problem"A company needs X — which solution achieves this?" · "Which architecture best achieves both X and Y?"
Simulation 3Comparative — distinguish similar services and trade-offs"What is the difference between X and Y?" · "Which option is NOT appropriate for Z?"

Step 4 — Write questions following the authoring standard

Every question must comply with all rules in the Exam & Question Authoring Standard. The non-negotiable rules:

  • Scenario-based stem — "A company wants to…" not "What is Amazon S3?" Applied judgment over abstract recall.
  • Single best answer — exactly one option is clearly correct; the other three are plausible but wrong for a specific, teachable reason.
  • No implementation questions — test service selection and judgment, not step-by-step configuration or CLI syntax.
  • Full ai_cache on every question — this is what the learner sees after answering. Empty ai_cache means no feedback, which breaks the learning loop.

Step 5 — Populate ai_cache for every question

The ai_cache column in question_items stores the full explanation shown to the learner after they answer. It must follow this exact JSON structure (mirror of AIF-C01):

{
  "explanations": {
    "overview": "One paragraph: why the correct answer is right — the conceptual reason, not just 'this is correct'.",
    "per_option": [
      { "option": "Exact text of option A", "correct": false, "explanation": "One sentence: specific teachable reason this is wrong." },
      { "option": "Exact text of option B", "correct": true,  "explanation": "One sentence: specific reason this is the best answer." },
      { "option": "Exact text of option C", "correct": false, "explanation": "One sentence: specific teachable reason this is wrong." },
      { "option": "Exact text of option D", "correct": false, "explanation": "One sentence: specific teachable reason this is wrong." }
    ]
  },
  "concept_guide": {
    "title": "Short concept name (e.g. 'S3 Storage Classes')",
    "summary": "2–3 sentence explanation of the underlying concept.",
    "key_facts": ["Bullet fact 1", "Bullet fact 2", "Bullet fact 3"],
    "related_concepts": [
      { "name": "Related concept", "description": "One sentence." }
    ]
  },
  "official_links": [
    { "url": "https://docs.aws.amazon.com/...", "title": "Official AWS documentation title" }
  ]
}
Do not insert questions with ai_cache: {}. Questions without explanations give learners no feedback and violate the authoring standard. Generate the full ai_cache for every question in the same pass as writing the question — it is harder to retrofit later.

Step 6 — Insert into Supabase

Insert questions into the question_items table. Required columns: question_set_id, question_number (sequential from 1), question_text, question_type ('Multiple Choice' or 'Multiple Response'), domain, options (jsonb array of 4–6 strings), correct_answers (jsonb array of the exact full text of each correct option — not indices), tags (jsonb), ai_cache (jsonb — the full structure above).

Insert at most 5 questions per SQL call via the Supabase MCP — larger batches can exceed the MCP payload limit and silently truncate. Escape single quotes in text by doubling them (it''s, AWS''s).

See Build Reference § Scaffold SQL for the ready-to-run exam_types + question_sets insert template, including the required questions_json = '[]' NOT-NULL vestigial column, and § Validation queries to run after each set.

Build Reference

Build Reference

Concrete schemas, SQL templates, and rules for every step. This section is what makes the playbook self-contained — you should never need to open an existing program's files.

1. Course data file & registry

Create src/data/<exam>Course.js (e.g. saaC03Course.js) and register it in src/utils/sessionCourses.js:

import newExamCourse from '../data/newExamCourse'

const SESSION_COURSES = [
  { match: ['aif', 'ai-practitioner'],            course: aifC01Course },
  { match: ['clf', 'cloud-practitioner'],          course: clfC02Course },
  { match: ['saa', 'solutions-architect-associate'], course: saaC03Course },
  // Add new exam:
  { match: ['<short-prefix>', '<full-exam-slug>'], course: newExamCourse },
]
Collision rule: use a unique short prefix (e.g. 'saa') AND the full hyphenated slug (e.g. 'solutions-architect-associate') — never a bare fragment that matches two exams. Example: 'solutions-architect' would also match SAP-C02.

2. Blueprint page template

Copy wiki/clf-c02.html as the starting template for a new blueprint page — it is the most recently updated and cleanest. Rename to wiki/<exam-slug>.html. Update the title, all domain text, scoring model, and Simulation Design table. Required sections: What this exam is · Question format & scoring · Domain breakdown · Helping candidates · Simulation design.

5. Multiple Response rules

Multiple Response questions ("Select TWO" / "Select THREE") appear on most AWS exams at ~20–25%. Target: Sim 1 is 100% Multiple Choice; Sims 2 and 3 each ~25% Multiple Response. Do not use MR in the free sample set.

  • Set question_type to 'Multiple Response'.
  • Stem must state the exact count: "Select TWO" or "Select THREE."
  • Provide 5–6 option strings (not 4) so the selection is non-trivial.
  • correct_answers holds the full text of every correct option (2 or 3 entries).
  • ai_cache.explanations.per_option must cover every option, with "correct": true on each correct one.
-- Multiple Response insert example
question_type    = 'Multiple Response'
options          = '["Option A","Option B","Option C","Option D","Option E"]'
correct_answers  = '["Option B","Option D"]'   -- exact full text, not indices
-- per_option array must have 5 entries (one per option)

6. Validation queries

Run after inserting each question set. Fix any failures before continuing.

-- Count per set
SELECT qs.name, COUNT(qi.id) AS q_count
FROM question_sets qs
LEFT JOIN question_items qi ON qi.question_set_id = qs.id
WHERE qs.exam_type_id = '<exam_type_id>'
GROUP BY qs.name ORDER BY qs.name;

-- Integrity: correct_answers must be a subset of options
SELECT question_number, question_text
FROM question_items
WHERE question_set_id = '<set_id>'
  AND NOT (correct_answers <@ options);

-- Integrity: per_option count must equal options count
SELECT question_number,
  jsonb_array_length(options) AS opt_count,
  jsonb_array_length(ai_cache->'explanations'->'per_option') AS per_opt_count
FROM question_items
WHERE question_set_id = '<set_id>'
  AND jsonb_array_length(options)
    != jsonb_array_length(ai_cache->'explanations'->'per_option');

-- Domain distribution
SELECT domain, COUNT(*) AS count
FROM question_items
WHERE question_set_id = '<set_id>'
GROUP BY domain ORDER BY domain;
Query 2 returning rows means correct_answers contains text that doesn't exactly match an option string — likely a typo or stray space. Fix and re-insert before proceeding.

7. Video curation rules

Every session must have at least one entry in its videos[] array. Videos live in the course data file (src/data/<exam>Course.js), not in Supabase.

  • Never fabricate video IDs. Verify every ID via the YouTube oEmbed endpoint before adding it: https://www.youtube.com/oembed?url=https://www.youtube.com/watch?v=VIDEO_ID&format=json. A 200 response with a title confirms the video is real and public.
  • Recommended pattern: add one full-course companion video (e.g. a freeCodeCamp exam-specific course) to every session in the program, plus one per-topic video on sessions where a good one exists.
videos: [
  {
    videoId: 'Ia-UEYYR44s',      // YouTube ID only — verified via oEmbed
    title: 'AWS Certified Solutions Architect Associate 2024 Full Course',
    channel: 'freeCodeCamp.org',
    startSeconds: null,          // null = full video; set to clip to a segment
    endSeconds: null,
    relevance: 'Full SAA-C03 curriculum — use as companion to any session.'
  }
]

QA

Definition of Done

A program is launch-ready when every box is checked.

Blueprint

  • ☐ Exam reference page exists with all five required sections
  • ☐ Domain weights transcribed verbatim from the official guide
  • ☐ Scoring model and out-of-scope list documented

Course

  • ☐ One session per task statement, each passing the Study Materials Standard checklist
  • ☐ Widgets added for the concepts that warrant them (and the course teaches fully without them)

Simulation

  • ☐ exam_types row inserted and is_active = true
  • ☐ All 4 question sets inserted: 1 free sample (10–12 Qs) + 3 full simulations (65 Qs each, $19.99)
  • ☐ Question count and domain distribution match the blueprint for each set
  • ☐ Every question stem is scenario-based (no "What is X?" recall stems)
  • ☐ Every question derived from a specific task statement in the official exam guide
  • ☐ Each simulation covers a distinct angle (foundational / architectural / comparative)
  • ☐ Every question has full ai_cache: overview + per-option explanations + concept guide + official links

Platform

  • ☐ Course registered and renders end-to-end for an enrolled user
  • ☐ Access gating verified for free / paid / non-enrolled states
  • ☐ Pricing/enrollment configured; cert added to the roadmap
  • ☐ Production build passes on Node 20 and is committed + pushed
