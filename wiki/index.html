What is CloudExamLab?

The product, the target users, and how the business model works.

CloudExamLab is a SaaS exam-prep platform focused on cloud certifications. The subscription unlocks two co-equal offerings for each exam: a comprehensive study course (guided lessons across every domain) and realistic practice exams. Both track progress and provide AI-powered explanations.

Core Value Proposition

  • Comprehensive study course — guided, exam-aligned lessons covering every domain (concepts, AWS service breakdowns, exam tips, and a practice question per session)
  • Realistic timed practice exams
  • AI explanations for every question (powered by Claude)
  • Progress tracking across multiple attempts
  • Offline support — take exams without internet
  • Study streak to encourage daily practice

Study material and practice exams are peers, not a primary feature plus an add-on: the exam-detail page surfaces both at the same level, and both are part of what the subscription sells.

Business Model

Subscription-based. One subscription unlocks all paid question sets and the full study course across all exams. Each exam offers a free taste of both: free sample question sets, and the first domain of the study course as a free preview. Everything beyond the preview requires an active subscription.

User Flow

End-to-end journey: from landing page to active learner.

Design principle: The user is making a multi-month career investment, not just buying a practice test. The flow leads with an easy signup, then immediately makes the product feel personal via the Cloud Certification Journey, then proves value with a free practice set before asking for payment.
Landing page ↓ Signup (email + password or Google OAuth — nothing else required) ↓ Cloud Certification Journey quiz (career-aware, 3 stages) ↓ Your Journey result current role → target role + salary + timeline + cert sequence ↓ Free 10-question practice set (Foundation cert by default; Associate cert if quiz shows user is past Foundation) ↓ "Subscribe to unlock your full journey" → PayPal payment ↓ Full dashboard + ongoing study

1. Landing Page /

Public marketing page. Primary CTA is "Start Free" (signup), not the quiz. Communicates: 6 cloud career paths + salaries, pricing, and FAQ. Works even without Supabase credentials — a mock client fallback renders the page.

2. Signup

Deliberately minimal friction.

  1. AuthModal opens with SignupForm
  2. User signs up with email + password or Google OAuth — no extra fields
  3. authStore.signup() → Supabase creates auth user + profile row
  4. If email verification is on, user sees "Check your email"
  5. Navigated to /dashboard

Country, city, and timezone are detected silently via ipapi.co and saved to the profile.

3. Cloud Certification Journey (post-signup)

A career-aware quiz that maps the user's background to a cert roadmap and a concrete salary outcome. Framed as a promise, not a help tool.

Stage 1 — Where you are now

  • Current role — Software Developer / SysAdmin-Ops / Data / Security / Network / AI-ML / Student / Other
  • Aspiring path — Cloud Architect / DevOps / Data / AI-ML / Security / Network (separate question — a developer can target Architect or DevOps)
  • Years of experience & AWS familiarity
  • Existing certifications held

Stage 2 — What you want to achieve

  • Primary goal (higher pay / promotion / career change / employer requirement / validate skills)
  • How far to go (Foundational → Associate → Professional → Specialty)
  • Target timeline

Stage 3 — Study capacity

  • Hours per week & region (for salary data)

Answers are persisted in localStorage under key cloudexamlab_path_answers so the dashboard "My Path" section and free-set logic can read them.

4. Your Journey Result

A career transformation statement, not just a cert list:

Current role: Software Developer (2–4 yrs experience) Target role: Senior DevOps Engineer Recommended path: Path 2 — DevOps / Operations Your Journey: CLF → DVA → DOP 5wks 10wks 16wks (at your chosen weekly pace) By month 8, you could be earning $131K–$151K/yr. 33,000+ job postings open right now on LinkedIn.

The depth of the cert sequence (how many steps show) is controlled by the "how far do you want to go" answer. Each path ends with a Specialty cert at the deepest "Domain Expert" depth.

5. Free Practice Set

  • Default: Foundation cert for their path (CLF, or AIF for the AI/ML path)
  • If quiz shows user is already past Foundation (holds an AWS cert, or 2+ years AWS experience) → first Associate cert instead — always relevant

Timed, answers save to user_progress, completion shows score + one AI explanation as a teaser.

6. Subscribe to Continue

  1. EnrollmentModal opens with the 3 subscription plans
  2. User selects plan → purchaseStore.createSubscription()
  3. Redirect to PayPal approval → /payment-success
  4. Subscription status updated to "active" in user_subscriptions

The pitch is targeted: "Unlock your full [Path Name] journey" — not a generic upsell.

7. Dashboard /dashboard

Quiz stateBehaviour
NOT completedA persistent prompt to complete the Cloud Certification Journey is always shown. Not a blocker — they can still browse.
CompletedMy Path section shows target role, salary, timeline, cert sequence + enrolled exams, study streak, daily goal, recent history, subscription status.

8. Taking an Exam /exam/:slug/take?set=:setId

Timer, per-question answer saving, free navigation between questions, AI explanations on demand. Resumes in-progress attempts automatically.

9. Review Results /exam/:slug/results

Score summary (%, scaled score, pass/fail, time) + question-by-question review with correct answers highlighted + AI explanations + retake option.

10. Ongoing Use

  • Streak tracking encourages daily sessions
  • Multiple attempts per set — each is saved separately
  • As certs are cleared, "My Path" reflects progress through the journey

Authentication

Handled entirely by Supabase Auth, wrapped in authStore (Zustand).

MethodHow
Email + passwordSupabase signUp / signInWithPassword
Magic link (OTP)Supabase signInWithOtp — sends email with one-time link
Password resetresetPasswordForEmail + updateUser

Signup Flow

  1. User submits email + password on SignupForm
  2. authStore.signup(email, password, metadata) → supabase.auth.signUp()
  3. Supabase creates auth user and fires onAuthStateChange with SIGNED_IN
  4. authStore.initialize() listener picks this up → calls loadProfile(userId)
  5. Profile row created in public.profiles (server-side trigger or upsert)
  6. Metadata saved: { country, city, timezone } — detected via ipapi.co

Login Flow

  1. authStore.login(email, password) → supabase.auth.signInWithPassword()
  2. Session tokens stored in localStorage under key cloudexamlab-auth
  3. loadProfile() called → fetches profile cache-first (1-hour TTL)
  4. Navigate to /dashboard

Magic Link Flow

  1. authStore.signInWithMagicLink(email) → supabase.auth.signInWithOtp()
  2. User receives email → clicks one-time link → redirected back to app with token in URL
  3. authStore.initialize() detects URL hash → Supabase processes token
  4. Auth state changes to SIGNED_IN → same profile load as above

Password Reset Flow

  1. User clicks "Forgot Password" → enters email
  2. authStore.resetPassword(email) → recovery email sent pointing to /reset-password
  3. ResetPassword.jsx checks URL for token_type === "recovery"
  4. User enters new password → authStore.updatePassword() → supabase.auth.updateUser()
  5. On success → redirect to landing page

Paid Content Check — ExamInterface.jsx

const isFree    = questionSet.is_free_sample || questionSet.price_cents === 0
const canAccess = isFree || isSubscribed

isSubscribed comes from purchaseStore, which checks user_subscriptions.status === 'active'.

Study Course Check — StudyMaterial.jsx / SessionCourse.jsx

The study course uses a free-preview model: the first module (domain) is open to any logged-in user; the rest require a subscription. StudyMaterial fetches the subscription and passes hasAccess + onSubscribe down.

// SessionCourse.jsx — first module is the free preview
const freeModuleId = course.modules[0]?.id
const isLocked = (session) => !hasAccess && session.domain !== freeModuleId

Locked sessions still appear in the sidebar (with a 🔒) and show a teaser header, but the lesson body is replaced by a LockedSession paywall whose CTA opens the EnrollmentModal. The legacy topic-based study path gates the same way: the first task is the preview, later tasks are replaced by a subscribe panel.

Like the paid-question-set check, this gate is application-layer only. Course content ships in the client bundle (src/data/*.js), so it is not a hard security boundary — it's a paywall, not DRM.

Admin Panel — Double-Gated

  1. Standard ProtectedRoute — must be logged in
  2. admin-api Edge Function verifies caller's email === ADMIN_EMAIL env var — returns 403 otherwise
No admin role in the database. Access is controlled by a single hardcoded email environment variable.

Row-Level Security (RLS) — Supabase Policies

TablePolicy
subscription_plansAnyone can read active plans (no auth needed)
exam_types, question_sets, question_itemsReadable by all authenticated users
user_progress, exam_attemptsUsers can only read/write their own rows
profiles, user_subscriptions, user_enrollmentsUsers can only read/write their own row(s)
RLS does not block question_items for paid sets. The subscription check is application-layer only — it happens in ExamInterface before loading the exam. A determined user could query the DB directly.

Edge Cases

  • Cancelled/suspended subscription → loses paid access immediately on next check
  • Enrollment persists even if subscription lapses — user stays enrolled but loses paid access
  • Free samples always accessible regardless of subscription state
  • Study course first domain (free preview) stays accessible to any logged-in user even without a subscription; later domains lock the moment a subscription lapses

Exam Flow

Starting, taking, submitting, and reviewing an exam.

Data Model

EntityKey fields
Exam typeduration_minutes, passing_score, max_score, total_questions
Question setquestion_count, is_free_sample, price_cents, version_number
Question itemquestion_text, question_type (Multiple Choice / Multiple Response), options[], domain, tags[], ai_cache

Starting an Exam

Route: /exam/:slug/take?set=:setId

  1. ExamInterface mounts → checks auth + access (isFree || isSubscribed)
  2. examStore.loadQuestionSet(setId) → fetches metadata + all question_items → normalizes format (derives correctAnswers[] from options[].isCorrect)
  3. progressStore.startOrResumeExam() → queries user_progress for any in_progress attempt for this set
  4. If resuming: loads saved state (question index, answers, timeElapsed) + shows resume banner
  5. Timer starts; options may be shuffled for randomisation

During the Exam

Navigation panel

All question numbers visible at once. Color coding:

  • Gray — not visited
  • Yellow/orange — visited but unanswered
  • Green — answered

User can jump to any question at any time.

Answering

  • Multiple Choice: click one option (auto-deselects others)
  • Multiple Response: toggle each option (e.g. "Select 2 answers")
  • Each selection calls progressStore.saveAnswer() → immediately upserted to user_progress

Timer

Counts up (tracks time spent, not remaining). Updates every second locally — not persisted every second. Pause state is persisted. Free samples get a proportional timer.

AI Explanations

User clicks "Explain" → AIExplanationPanel slides in → calls ai-explanation Edge Function → returns explanation + AWS doc links. Response cached in question_items.ai_cache to avoid repeat API calls.

Offline persistence

If a save fails (offline), SyncService queues the write and retries when network returns.

Submitting

Score calculation

correctCount  = answers matching question's correctAnswers[]
percentage    = (correctCount / totalQuestions) * 100
scaledScore   = percentage × (maxScore / 100)
passed        = percentage >= exam_type.passing_score

After submit

  1. Full record saved to exam_attempts
  2. user_progress.status updated to completed
  3. streakService.recordActivity() — increments streak if consecutive day
  4. Redirect to /exam/:slug/results?resultId=:id&set=:setId

Results Page

  • Score summary: raw score, %, scaled score, pass/fail badge, time spent
  • Every question reviewed: user's answer, correct answer highlighted, domain tag, explanation text
  • AI explanation panel on demand per question
  • Actions: Get AI Explanation, Retake Exam, Back to Exam page

PayPal Subscription Flow

Webhook — Keeping Subscription Status in Sync

The paypal-webhook Edge Function receives events from PayPal, validates the signature using PAYPAL_WEBHOOK_ID, and updates user_subscriptions.status.

BILLING.SUBSCRIPTION.ACTIVATEDactive
BILLING.SUBSCRIPTION.CANCELLEDcancelled

Webhook must be registered in PayPal Developer dashboard pointing to:

https://<supabase-project-ref>.supabase.co/functions/v1/paypal-webhook

Enrollment (separate concept from subscription)

Enrollment links a subscriber to a specific exam type. One subscription → many enrollments.

  1. Active subscriber clicks "Enroll" on an exam type
  2. purchaseStore.enrollInExam() → inserts into user_enrollments
  3. Unique constraint on (user_id, exam_type_id) — returns alreadyEnrolled: true if duplicate
  4. Enrolled exams appear on dashboard
Enrollment ≠ access control. Enrollment is for dashboard filtering and progress tracking. Paid content access is controlled by subscription status alone.

Changing Prices or PayPal Plans

PayPal does not allow editing prices on existing billing plans. To change prices: create new plans in PayPal → update subscription_plans table rows → update the fallback prices in EnrollmentModal.jsx:187–190. Existing subscribers on old plans continue on their old pricing until they cancel.

Design Patterns

Cache-first data loading

Profile, exam types, question sets — all check local cache (localStorage or IndexedDB) first. Cache hit: return cached data + trigger background refresh. Cache miss: fetch from Supabase + populate cache.

Offline-first exam taking

Answers are saved to Supabase immediately. If the request fails (offline), SyncService queues it and retries with exponential backoff when the network returns.

Client-generated UUIDs for attempts

exam_attempt_id is generated on the client with crypto.randomUUID() before the attempt starts. This lets local state reference the attempt ID before the first server write — no round-trip needed to start an exam.

Edge Functions for sensitive operations

Admin CRUD and PayPal API calls go through Supabase Edge Functions, keeping SUPABASE_SERVICE_ROLE_KEY and PAYPAL_CLIENT_SECRET off the client.

Zustand over Redux

Lightweight stores with no boilerplate. Each store is self-contained with its own state and actions. Stores communicate by calling each other's methods directly.

Question normalisation

examStore.normalizeQuestionSet() converts raw DB format into a consistent { correctAnswers[], options[] } structure used throughout the UI. Runs once on load.

Edge Functions

4 Supabase Edge Functions (Deno/TypeScript) in supabase/functions/. Supabase auto-injects SUPABASE_URL and SUPABASE_ANON_KEY — all other secrets must be set via supabase secrets set.

admin-api

Purpose: Secure gateway for all admin CRUD. Keeps SUPABASE_SERVICE_ROLE_KEY off the client.

Auth: Decodes the caller's JWT, fetches their email, compares to ADMIN_EMAIL. Returns 403 if no match.

Operations: exam type CRUD, question set CRUD, question item bulk-upsert & delete.

ai-explanation

Purpose: Returns an AI-generated explanation for a question + relevant AWS documentation links.

Flow: Receives question_item_id + context → checks ai_cache column → if cached, returns it immediately → otherwise calls Claude API → saves to ai_cache + logs to ai_usage.

SecretPurpose

Offline & Sync

CloudExamLab supports taking exams without internet. Built on three layers.

Layer 1 — IndexedDB (large data storage)

File: src/services/indexedDBService.js
DB name: CloudExamLabDB (version 1)

Stores large structured data locally so it survives page refreshes and is available offline. Used by examStore — subsequent loads after the first come from IDB instead of Supabase.

Object storeContents
questionSetsFull question set data (all questions + options)
examAttemptsCompleted exam records
examResultsProcessed results with scoring

Layer 2 — SyncService (background write queue)

File: src/services/syncService.js

An event-driven queue that ensures writes made during offline periods eventually reach Supabase.

Flow

  1. When online → write to Supabase directly
  2. When offline → add to in-memory queue
  3. When window fires online event → process queue
  4. Each item retried up to 3 times with exponential backoff (5 s base)
  5. On permanent failure → item marked failed, event emitted
Sync typeTable it writes to
progressuser_progress
resultexam_attempts
profileprofiles

Events emitted and consumed by syncStore (which drives the UI indicator): added, processing, synced, failed, retry, idle, offline, cleared.

Layer 3 — Service Worker (PWA)

File: /sw.js (registered in main.jsx). Caches static assets and API responses for offline access. Standard PWA setup.

localStorage Cache (cacheService)

File: src/services/cacheService.js. Simple wrapper with TTL support. Expired entries removed automatically on get().

Cached dataTTL
User profile1 hour
Progress metadataSession

API: set(key, value, ttlMs), get(key), remove(key), has(key), touch(key), clearAll().

Streak Sync (streakService)

File: src/services/streakService.js. Streaks are stored locally first in localStorage under key exam_prep_streak. On init: fetches latest from study_streaks table. Syncs back to DB every 5 minutes. Works fully offline; on next load the DB version wins (last-write-wins for multi-device conflict resolution).

localStorage Key Namespace

All app keys use prefix cloudexamlab_ (from constants.js: STORAGE_PREFIX). Prevents collisions and makes it easy to clear all app data.

Fallback Mock Client

If Supabase env vars are missing, services/supabase.js uses a mock client instead of throwing. Returns empty data for reads; "Supabase not configured" errors for writes. Keeps the landing page functional with no backend.

Environment Setup

How to configure the project locally and what's needed when switching Supabase projects.

Frontend Environment Variables

Create .env at the project root:

VITE_SUPABASE_URL=https://your-project-ref.supabase.co
VITE_SUPABASE_ANON_KEY=your-anon-key

Get from: Supabase Dashboard → Project Settings → API. These are the only two frontend env vars. Everything else lives server-side in Edge Function secrets.

Edge Function Secrets

supabase secrets set \
  SUPABASE_SERVICE_ROLE_KEY=your-service-role-key \
  ADMIN_EMAIL=your-admin-email@example.com \
  PAYPAL_CLIENT_ID=your-paypal-client-id \

  PAYPAL_WEBHOOK_ID=your-paypal-webhook-id \
  PAYPAL_MODE=live \
  --project-ref your-project-ref
SUPABASE_URL and SUPABASE_ANON_KEY are auto-injected by Supabase — do not set them.

Step 1 — Create new project

Go to supabase.com → New project → copy URL and anon key.

Step 2 — Run database schema

In the Supabase SQL editor, run each file in supabase/table/ in this order (order matters for foreign keys):

  1. exam_types.sql
  2. profiles.sql
  3. question_sets.sql
  4. question_items.sql
  5. subscription_plans.sql
  6. user_subscriptions.sql
  7. user_enrollments.sql
  8. user_progress.sql
  9. exam_attempts.sql
  10. study_streaks.sql
  11. ai_usage.sql
  12. packages.sql
  13. paypal_webhook_logs.sql

Then run migrations in supabase/migrations/ in order:

  1. create_study_streak_table.sql
  2. add_question_sets_rls.sql
  3. add_exam_dates_to_profiles.sql
  4. migrate_to_paypal.sql
  5. drop_product_purchase_tables.sql
