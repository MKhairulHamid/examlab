AWS Certified Solutions Architect – Associate (SAA-C03)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified Solutions Architect – Associate (SAA-C03) validates the ability to design solutions that incorporate AWS services to meet current and future business requirements. Unlike the foundational Cloud Practitioner, this is a design exam: it tests judgment under constraints — choosing the right service, the right architecture, and the right trade-off between security, resilience, performance, and cost.

  • Target audience: Individuals performing a solutions architect role who design architectures that are secure, resilient, high-performing, and cost-optimized — and who review existing solutions to recommend improvements.
  • Recommended experience: At least 1 year of hands-on experience designing cloud solutions using AWS services. CLF-C02 is a recommended (not required) precursor.
  • Why candidates take it: The most popular associate-level AWS certification and the standard credential for cloud / solutions architect roles. Often the gateway to the Professional (SAP-C02) and specialty tracks.

Out-of-Scope Services (Do NOT Test)

The official exam guide explicitly lists 44 services that are out of scope for SAA-C03. Any question that hinges on one of these is out of scope — cut it. The notable exclusions:

  • All IoT services, Amazon GameLift, Amazon Lightsail, Amazon Sumerian, Amazon Location Service, AWS Cloud Map, Amazon Braket, AWS Ground Station.
  • Developer tooling: AWS CDK, AWS CloudShell, AWS CodeArtifact, CodeBuild, CodeCommit, CodeDeploy, Amazon Corretto, AWS Fault Injection Simulator, AWS Tools and SDKs, AWS Distro for OpenTelemetry.
  • ML/AI specialty: Amazon Augmented AI (A2I), DeepComposer, Deep Learning AMIs/Containers, Amazon DevOps Guru, Elastic Inference, AWS Inferentia, Amazon Personalize, SageMaker Canvas, SageMaker Ground Truth, Apache MXNet / PyTorch / TensorFlow on AWS, Amazon HealthLake.
  • Media / elemental: all AWS Elemental Appliances and Software, MediaConnect, MediaConvert, MediaLive, MediaPackage, MediaTailor, Amazon Interactive Video Service (IVS).
  • Other: Amazon MWAA, Amazon Managed Blockchain, Amazon RDS on VMware, Migration Evaluator, AWS Console Mobile Application.
Exam tip: SAA-C03 tests architectural design judgment, not implementation steps or code. The crux of nearly every question is a trade-off — "MOST cost-effective", "LEAST operational overhead", "MOST resilient". Engineer questions so the right answer wins on the qualifier in the stem.

Domain 1 · 30%

Design Secure Architectures

The highest-weight domain. Covers identity and access, network and application security, and data protection — the security spine that every other domain depends on.

Task 1.1 — Design secure access to AWS resources

Knowledge of:

  • Access controls and management across multiple accounts
  • AWS federated access and identity services (e.g. IAM, AWS IAM Identity Center)
  • AWS global infrastructure (e.g. Availability Zones, AWS Regions)
  • AWS security best practices (e.g. the principle of least privilege)
  • The AWS shared responsibility model

Skills in:

  • Applying AWS security best practices to IAM users and root users (e.g. MFA)
  • Designing a flexible authorization model that includes IAM users, groups, roles, and policies
  • Designing a role-based access control strategy (e.g. AWS STS, role switching, cross-account access)
  • Designing a security strategy for multiple AWS accounts (e.g. AWS Control Tower, service control policies [SCPs])
  • Determining the appropriate use of resource policies for AWS services
  • Determining when to federate a directory service with IAM roles

Task 1.2 — Design secure workloads and applications

Knowledge of:

  • Application configuration and credentials security
  • AWS service endpoints
  • Control ports, protocols, and network traffic on AWS
  • Secure application access
  • Security services with appropriate use cases (e.g. Amazon Cognito, Amazon GuardDuty, Amazon Macie)
  • Threat vectors external to AWS (e.g. DDoS, SQL injection)

Skills in:

  • Designing VPC architectures with security components (e.g. security groups, route tables, network ACLs, NAT gateways)
  • Determining network segmentation strategies (e.g. using public and private subnets)
  • Integrating AWS services to secure applications (e.g. AWS Shield, AWS WAF, IAM Identity Center, AWS Secrets Manager)
  • Securing external network connections to and from the AWS Cloud (e.g. VPN, AWS Direct Connect)

Task 1.3 — Determine appropriate data security controls

Knowledge of:

  • Data access and governance
  • Data recovery
  • Data retention and classification
  • Encryption and appropriate key management

Skills in:

  • Aligning AWS technologies to meet compliance requirements
  • Encrypting data at rest (e.g. AWS KMS)
  • Encrypting data in transit (e.g. AWS Certificate Manager [ACM] using TLS)
  • Implementing access policies for encryption keys
  • Implementing data backups and replications
  • Implementing policies for data access, lifecycle, and protection
  • Rotating encryption keys and renewing certificates

  • Horizontal scaling and vertical scaling

  • How to migrate applications into containers
  • Load balancing concepts (e.g. Application Load Balancer [ALB])
  • Multi-tier architectures
  • Queuing and messaging concepts (e.g. publish/subscribe)
  • Serverless technologies and patterns (e.g. AWS Fargate, AWS Lambda)
  • Storage types with associated characteristics (e.g. object, file, block)
  • The orchestration of containers (e.g. Amazon ECS, Amazon EKS)
  • When to use read replicas
  • Workflow orchestration (e.g. AWS Step Functions)

Skills in:

  • Designing event-driven, microservice, and/or multi-tier architectures based on requirements
  • Determining scaling strategies for components used in an architecture design
  • Determining the AWS services required to achieve loose coupling based on requirements
  • Determining when to use containers
  • Determining when to use serverless technologies and patterns
  • Recommending appropriate compute, storage, networking, and database technologies based on requirements

Task 2.2 — Design highly available and/or fault-tolerant architectures

Knowledge of:

  • AWS global infrastructure (e.g. Availability Zones, AWS Regions, Amazon Route 53)
  • AWS managed services with appropriate use cases (e.g. Amazon Comprehend, Amazon Polly)
  • Basic networking concepts (e.g. route tables)
  • Disaster recovery (DR) strategies (e.g. backup and restore, pilot light, warm standby, active-active failover, RPO, RTO)
  • Distributed design patterns
  • Failover strategies
  • Immutable infrastructure
  • Load balancing concepts (e.g. ALB)
  • Proxy concepts (e.g. Amazon RDS Proxy)
  • Service quotas and throttling (e.g. configuring service quotas for a standby workload)
  • Storage options and characteristics (e.g. durability, replication)
  • Workload visibility (e.g. AWS X-Ray)

Skills in:

  • Determining automation strategies to ensure infrastructure integrity
  • Determining the AWS services required to provide a highly available and/or fault-tolerant architecture across AWS Regions or Availability Zones
  • Identifying metrics based on business requirements to deliver a highly available solution
  • Implementing designs to mitigate single points of failure
  • Implementing strategies to ensure the durability and availability of data (e.g. backups)
  • Selecting an appropriate DR strategy to meet business requirements
  • Using AWS services that improve the reliability of legacy applications not built for the cloud (when application changes are not possible)
  • Using purpose-built AWS services for workloads

Candidate Prep

How We Help SAA-C03 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 720+.

Who Takes This Exam

  • Cloud engineers and developers moving toward architecture roles who need to validate end-to-end design skill, not just service awareness.
  • CLF-C02 holders stepping up to the first associate-level credential — the most common next stop on the AWS path.
  • Career switchers with infrastructure backgrounds (sysadmins, network/storage engineers) mapping on-premises experience onto AWS services.

What Candidates Struggle With

  • Reading the qualifier — "MOST cost-effective", "LEAST operational overhead", "MOST resilient". Two options are often both technically valid; only one wins on the qualifier. This is the single biggest source of lost marks.
  • Service disambiguation under constraints — SQS vs. SNS vs. EventBridge vs. Kinesis; ALB vs. NLB vs. Gateway Load Balancer; EBS vs. EFS vs. FSx vs. S3; RDS vs. Aurora vs. DynamoDB vs. ElastiCache.
  • DR strategy selection — matching backup & restore / pilot light / warm standby / active-active to a stated RPO/RTO and budget.
  • VPC security model — security groups (stateful) vs. network ACLs (stateless); public vs. private subnets; NAT gateway vs. internet gateway; VPC endpoints vs. NAT.
  • Encryption & key management — at rest vs. in transit; KMS key policies vs. IAM policies; when to use customer-managed keys; ACM for TLS.
  • Decoupling for resilience and scale — recognizing when a queue, a load balancer, or an event bus is the fix for a single point of failure or a scaling bottleneck.
  • Data-transfer and NAT cost traps — cross-AZ vs. cross-Region charges, NAT gateway data processing cost, VPC endpoints to avoid NAT, CloudFront to cut origin egress.

Recommended Study Path

  1. Domain 1 — Secure first. Highest weight (30%) and foundational. IAM, VPC security, and encryption underpin every scenario in the other three domains — build this mental model before anything else.
  2. Domain 2 — Resilience. Decoupling, multi-AZ/multi-Region, and DR strategies. Master the four DR patterns and the RPO/RTO trade-offs that select between them.
  3. Domain 3 — Performance. Work category by category (storage → compute → database → network → ingestion). Focus on choosing the right purpose-built service for a stated performance demand.
  4. Domain 4 — Cost. Smallest domain (20%) and the most pattern-based. Purchasing options, S3 tiering/lifecycle, right-sizing, and data-transfer cost control — efficient marks once the patterns click.

AI Explanation Quality for This Exam

SAA-C03 questions are design trade-offs, so explanations must do more than label answers right/wrong — they must explain the trade-off the question turns on. Patterns to cover:

  • For "MOST cost-effective / LEAST overhead" questions: explain why the chosen option wins on that specific axis even though a distractor also "works" functionally.
  • For service-selection questions: name the key differentiator that disqualifies each distractor (e.g. "EFS is shared file storage; EBS attaches to a single AZ instance, so it cannot back a fleet across AZs").
  • For DR questions: tie the answer back to the stated RPO/RTO and cost — explain why the next-cheaper strategy misses the recovery objective and the next-richer one over-spends.
  • For security questions: explain which control belongs at which layer (identity vs. network vs. data) and why the distractor secures the wrong layer.

Question Writing Rules (SAA-C03 specific)

  • Scenario-based stems — "A company runs…", "A solutions architect must…". Never "What is X?"
  • Make the qualifier the crux — most stems end in "MOST cost-effective", "LEAST operational overhead", "MOST resilient", or "with the LEAST disruption". Write distractors that are functionally valid but lose on that axis.
  • Design, not implementation — ask which architecture/service fits the requirement, not the CLI steps to build it.
  • Constrain with real requirements — RPO/RTO, latency targets, compliance, budget, "without managing servers", "without code changes". The constraint should pick the answer.
  • Stay in scope — only use services from the in-scope list; never hinge a question on an out-of-scope service.

Distractor Quality Guidelines

  • Use real architectural confusions as distractors: SQS vs. SNS vs. EventBridge; ALB vs. NLB vs. GWLB; EBS vs. EFS vs. FSx vs. S3; RDS vs. Aurora vs. DynamoDB; security group vs. NACL.
  • For trade-off questions: make one distractor the cheaper-but-misses-the-requirement option and one the over-engineered-and-over-priced option.
  • For DR questions: distractors should be adjacent DR strategies (e.g. pilot light when warm standby is required) so the candidate must reason from RPO/RTO.
  • For security questions: make one distractor secure the wrong layer (network control for a data problem, or vice versa).

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — naming the trade-off axis it wins on (cost, overhead, resilience, latency)
  • Why each distractor is wrong — one sentence per option, naming the specific reason it loses
  • Concept link — which domain/task statement this tests (e.g. "D2 · Task 2.2 — Highly available / fault-tolerant architectures")
  • Service context — if a service appears, one line on what it does and what distinguishes it from the services in the distractors
