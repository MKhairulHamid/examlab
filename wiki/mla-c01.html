AWS Certified Machine Learning Engineer – Associate (MLA-C01)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified Machine Learning Engineer – Associate (MLA-C01) validates a candidate's ability to build, operationalize, deploy, and maintain machine learning (ML) solutions and pipelines on the AWS Cloud. It is the role-based successor to the retiring ML Specialty (MLS-C01), and it leans toward ML engineering and MLOps — operationalizing models in production — rather than pure data science or model research.

  • Target audience: ML engineers, MLOps engineers, and backend/DevOps/data engineers who build, deploy, and maintain ML systems on AWS (Amazon SageMaker front and center).
  • Prerequisite experience: At least 1 year using Amazon SageMaker and other AWS services for ML engineering, plus 1 year in a related role (backend developer, DevOps engineer, data engineer, or data scientist).
  • Why candidates take it: To validate production ML engineering skills, qualify for ML/MLOps engineer roles (avg ~$187K/yr, highest demand of any AWS path), and build toward the Generative AI Developer Professional path. Often paired with AI Practitioner (AIF-C01) and Data Engineer Associate (DEA-C01).

Out-of-Scope Job Tasks (Do NOT Test)

The official exam guide explicitly excludes these tasks from the target candidate role. Any question that requires them is out of scope — cut it.

  • Designing and architecting full end-to-end ML solutions from scratch
  • Setting up best practices and guiding overall ML strategy for an organization
  • Handling integration with a wide array of services or brand-new tools and technologies
  • Working deeply in two or more ML domains simultaneously (e.g. advanced NLP and computer vision)
  • Quantizing models and analyzing the impact on accuracy
Exam tip: MLA-C01 tests ML engineering — preparing data, training and tuning with SageMaker, deploying to the right endpoint type, and monitoring/securing the pipeline. If a question is about choosing the ML algorithm's underlying mathematics or designing a novel architecture, it leans toward data-science/specialty territory and is likely out of scope here.

Domain 1 · 28%

Data Preparation for Machine Learning (ML)

The largest domain. Covers ingesting and storing data, transforming it and engineering features, and ensuring data quality, integrity, and freedom from bias before modeling.

Task 1.1 — Ingest and store data

Knowledge & skills in:

  • Data formats and ingestion mechanisms — validated/non-validated formats, Apache Parquet, JSON, CSV, Apache ORC, Apache Avro, RecordIO
  • Core AWS data sources (Amazon S3, Amazon EFS, Amazon FSx for NetApp ONTAP) and streaming sources (Amazon Kinesis, Apache Flink, Apache Kafka)
  • Extracting data from storage (S3, EBS, EFS, RDS, DynamoDB) using options like S3 Transfer Acceleration and EBS Provisioned IOPS
  • Choosing data formats based on access patterns; ingesting into SageMaker Data Wrangler and SageMaker Feature Store
  • Merging data from multiple sources (AWS Glue, Apache Spark); making storage decisions based on cost, performance, and structure

Task 1.2 — Transform data and perform feature engineering

Knowledge & skills in:

  • Data cleaning and transformation — outlier detection, imputing missing data, combining, deduplication
  • Feature engineering — scaling and standardization, feature splitting, binning, log transformation, normalization
  • Encoding techniques — one-hot, binary, label encoding, tokenization
  • Tools to explore, visualize, and transform data — SageMaker Data Wrangler, AWS Glue, AWS Glue DataBrew, Spark on Amazon EMR
  • Creating and managing features with SageMaker Feature Store; labeling with SageMaker Ground Truth and Amazon Mechanical Turk; transforming streaming data with Lambda and Spark

Task 1.3 — Ensure data integrity and prepare data for modeling

Knowledge & skills in:

  • Pre-training bias metrics — class imbalance (CI), difference in proportions of labels (DPL) — for numeric, text, and image data
  • Strategies to address class imbalance (synthetic data generation, resampling) and to mitigate bias (selection, measurement) using SageMaker Clarify
  • Validating data quality with AWS Glue Data Quality and DataBrew
  • Encryption, data classification, anonymization, and masking; compliance with PII, PHI, and data residency requirements
  • Preparing data to reduce prediction bias — dataset splitting, shuffling, augmentation; configuring EFS/FSx to load data into training resources
Domain 2 · 26%

ML Model Development

Choosing the right modeling approach, training and refining models with SageMaker, and rigorously analyzing model performance to detect bias, overfitting, and convergence issues.

Task 2.1 — Choose a modeling approach

Knowledge & skills in:

  • Capabilities and appropriate uses of ML algorithms to solve business problems
  • Using AWS AI services (Amazon Translate, Transcribe, Rekognition, Comprehend, Bedrock) to solve common business needs without training a model
  • SageMaker built-in algorithms and when to apply them; foundation models and solution templates in SageMaker JumpStart and Amazon Bedrock
  • Considering interpretability and cost during model/algorithm selection
  • Assessing data availability and problem complexity to determine ML feasibility

Task 2.2 — Train and refine models

Knowledge & skills in:

  • Training elements — epoch, steps, batch size; methods to reduce training time (early stopping, distributed training)
  • Using SageMaker built-in algorithms and script mode with TensorFlow/PyTorch; fine-tuning pre-trained models (Bedrock, JumpStart); integrating externally built models into SageMaker
  • Hyperparameter tuning — random search, Bayesian optimization, SageMaker automatic model tuning (AMT); model hyperparameters and their effects
  • Preventing overfitting, underfitting, and catastrophic forgetting — regularization (dropout, weight decay, L1/L2), feature selection
  • Combining models (ensembling, stacking, boosting); reducing model size (data types, pruning, compression); versioning with SageMaker Model Registry

Task 2.3 — Analyze model performance

Knowledge & skills in:

  • Evaluation metrics — confusion matrix, F1 score, accuracy, precision, recall, RMSE, ROC, AUC; heat maps
  • Creating performance baselines; identifying overfitting and underfitting; diagnosing convergence issues
  • Selecting and interpreting metrics and detecting model bias; assessing tradeoffs between performance, training time, and cost
  • Comparing a shadow variant to a production variant; performing reproducible experiments
  • Using SageMaker Clarify to interpret model outputs and SageMaker Debugger to debug convergence
Domain 3 · 22%

Deployment and Orchestration of ML Workflows

The MLOps core: choosing the right deployment infrastructure and endpoint type, scripting it as infrastructure as code, and automating the whole workflow through CI/CD pipelines.

Task 3.1 — Select deployment infrastructure based on architecture and requirements

Knowledge & skills in:

  • Deployment best practices — versioning, rollback strategies; serving models in real time and in batches
  • Endpoint types — real-time, serverless, asynchronous endpoints, and batch inference (batch transform); their model and resource requirements
  • Provisioning compute (CPU/GPU) for production and test; choosing provided vs. custom containers; optimizing models for edge devices with SageMaker Neo
  • Evaluating performance, cost, and latency tradeoffs; selecting deployment targets (SageMaker endpoints, Kubernetes, Amazon ECS, Amazon EKS, AWS Lambda)
  • Selecting multi-model or multi-container deployments and the correct orchestrator (Apache Airflow, SageMaker Pipelines)

Task 3.2 — Create and script infrastructure based on requirements

Knowledge & skills in:

  • On-demand vs. provisioned resources; comparing scaling policies
  • Infrastructure as code — AWS CloudFormation and AWS CDK tradeoffs; automating provisioning and inter-stack communication
  • Building and maintaining containers (Amazon ECR, ECS, EKS, bring-your-own-container with SageMaker)
  • SageMaker endpoint auto scaling policies (demand- and time-based); choosing scaling metrics (model latency, CPU utilization, invocations per instance)
  • Configuring SageMaker endpoints within a VPC; deploying with the SageMaker SDK; using Spot Instances and Lambda behind endpoints for cost-effectiveness

Task 3.3 — Use automated orchestration tools to set up CI/CD pipelines

Knowledge & skills in:

  • Capabilities and quotas of AWS CodePipeline, CodeBuild, and CodeDeploy; configuring and troubleshooting their stages
  • Version control (Git) and continuous-deployment flows (Gitflow, GitHub Flow); how repositories and pipelines work together
  • Deployment strategies and rollback actions — blue/green, canary, linear
  • Configuring training and inference jobs with Amazon EventBridge rules, SageMaker Pipelines, and CodePipeline
  • Creating automated tests (unit, integration, end-to-end) in CI/CD; building mechanisms to retrain models
Domain 4 · 24%

ML Solution Monitoring, Maintenance, and Security

Keeping deployed ML systems healthy, cost-efficient, and secure: detecting drift, monitoring and right-sizing infrastructure, controlling costs, and locking down access.

Task 4.1 — Monitor model inference

Knowledge & skills in:

  • Drift in ML models; techniques to monitor data quality and model performance; ML lens design principles for monitoring
  • Monitoring models in production with Amazon SageMaker Model Monitor
  • Monitoring workflows to detect anomalies or errors in data processing and inference
  • Detecting data-distribution changes that affect performance using SageMaker Clarify
  • Monitoring production performance with A/B testing

Task 4.2 — Monitor and optimize infrastructure and costs

Knowledge & skills in:

  • Key infrastructure metrics — utilization, throughput, availability, scalability, fault tolerance
  • Observability tools — Amazon CloudWatch (Logs, Logs Insights, alarms, dashboards, Lambda Insights), AWS X-Ray, AWS CloudTrail, Amazon EventBridge, Amazon QuickSight
  • Instance type differences (memory/compute/general-purpose/inference-optimized); right-sizing with SageMaker Inference Recommender and AWS Compute Optimizer
  • Cost analysis and control — AWS Cost Explorer, Billing and Cost Management, AWS Budgets, AWS Trusted Advisor; tagging strategy for cost allocation
  • Purchasing options — Spot, On-Demand, Reserved Instances, SageMaker Savings Plans; troubleshooting capacity (provisioned concurrency, service quotas, auto scaling)

Task 4.3 — Secure AWS resources

Knowledge & skills in:

  • IAM roles, policies, and groups; bucket policies; SageMaker Role Manager — configuring least-privilege access to ML artifacts
  • SageMaker security and compliance features
  • Controls for network access to ML resources — building VPCs, subnets, and security groups to isolate ML systems
  • Security best practices for CI/CD pipelines
  • Monitoring, auditing, and logging ML systems for continued security and compliance; troubleshooting security issues

Candidate Prep

How We Help MLA-C01 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 720+.

Who Takes This Exam

  • ML / MLOps engineers — practitioners who already train and deploy models and want to validate production ML engineering on AWS.
  • Backend, DevOps, and data engineers moving into ML — engineers who know AWS infrastructure and CI/CD and are adding the ML lifecycle.
  • ML Specialty holders re-certifying — candidates migrating from the retiring MLS-C01 to the new role-based associate credential.

What Candidates Struggle With

  • SageMaker endpoint types — real-time vs. serverless vs. asynchronous vs. batch transform. Choosing by latency, payload size, traffic pattern, and cost is the single densest decision on the exam.
  • Data preparation tool sprawl — Data Wrangler vs. Glue vs. DataBrew vs. EMR Spark, plus Feature Store and Ground Truth. Knowing which tool fits which job.
  • Evaluation metrics — precision vs. recall vs. F1, ROC/AUC, RMSE, and reading a confusion matrix. Which metric matters for imbalanced classes or for a fraud-detection use case.
  • Bias and Clarify — pre-training bias metrics (CI, DPL) vs. post-training drift; what SageMaker Clarify does at each stage.
  • Overfitting remedies — regularization (dropout, L1/L2, weight decay), early stopping, more data/augmentation, and how hyperparameter tuning (AMT) finds the right settings.
  • MLOps orchestration — SageMaker Pipelines vs. Step Functions vs. Airflow (MWAA); where CodePipeline/CodeBuild/CodeDeploy and EventBridge-triggered retraining fit.
  • Monitoring & cost — Model Monitor for drift, CloudWatch/X-Ray for infrastructure, and right-sizing/purchasing options (Spot, Savings Plans, Inference Recommender) to control cost.

Recommended Study Path

  1. Domain 1 — Data Preparation first. Highest weight (28%) and the foundation of every ML pipeline. Master the ingestion/storage choices and the Data Wrangler / Glue / Feature Store toolset before modeling.
  2. Domain 2 — Model Development. SageMaker training (built-in algorithms, script mode, JumpStart), hyperparameter tuning, and—critically—evaluation metrics and bias analysis with Clarify.
  3. Domain 3 — Deployment & Orchestration. Learn endpoint types and the CI/CD + SageMaker Pipelines story as one connected MLOps workflow (build → train → deploy → retrain).
  4. Domain 4 — Monitoring, Maintenance & Security. Drift detection, observability, cost optimization, and IAM/VPC isolation. It reuses CloudWatch and IAM concepts and ties the whole lifecycle together — efficient marks close to exam day.

AI Explanation Quality for This Exam

MLA-C01 questions use concrete ML-engineering scenarios. Every wrong answer needs an explanation that says WHY the distractor is wrong — not just "this is not correct." Common patterns to cover:

  • For endpoint questions: explain the selection reason (e.g. "asynchronous endpoints queue large payloads and long-running inference; serverless suits intermittent traffic with cold-start tolerance").
  • For data-prep questions: explain the tool fit (e.g. "Data Wrangler is for visual, interactive feature engineering inside SageMaker Studio; Glue is for serverless ETL at scale").
  • For metric questions: explain what the metric optimizes (e.g. "recall matters when missing a positive is costly, as in fraud or disease detection — precision trades off against it").
  • For monitoring questions: explain the distinction (e.g. "Model Monitor detects data and quality drift on a live endpoint; CloudWatch tracks infrastructure metrics like latency and invocations").

Question Writing Rules (MLA-C01 specific)

  • Scenario-based, ML-engineer-framed stems — "An ML engineer needs to…" or "A team is training a model on SageMaker and…" Never "What is the definition of…?"
  • Lifecycle-aware — anchor each question in a pipeline stage (ingest, transform, train, tune, evaluate, deploy, monitor, secure) so the set covers the full ML workflow.
  • Service operation over service awareness — ask HOW to configure or choose (which endpoint type, which Clarify report, which scaling metric), not just which service.
  • Best-practice framing — many questions reduce to "which is the MOST cost-effective / LOWEST-latency / LEAST operational-overhead way to do X."
  • Ground every question in a task statement — tie each to a specific 1.x–4.x skill so the set stays on-blueprint.

Distractor Quality Guidelines

  • Use real ML-engineer confusions as distractors: real-time vs. serverless vs. async vs. batch endpoints; Data Wrangler vs. Glue vs. DataBrew; precision vs. recall vs. F1; CI/DPL pre-training bias vs. post-deployment drift; SageMaker Pipelines vs. Step Functions vs. MWAA.
  • For modeling questions: make one distractor an out-of-scope or wrong-fit algorithm/AI service, and one that confuses overfitting remedies (e.g. "add more layers" when the model is already overfitting).
  • For deployment questions: make each distractor a real but wrong endpoint type or strategy for the stated latency/throughput/cost goal.
  • Use out-of-scope services (Cognito, Beanstalk, DeepRacer, GuardDuty) as confident distractors that should never be the correct answer.

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — with a brief conceptual reason
  • Why each distractor is wrong — one sentence per option, naming the specific reason
  • Concept link — which domain/task statement this tests (e.g. "D3 · Task 3.1 — SageMaker endpoint types")
  • Service context — if a service appears, one line on what it does and what distinguishes it from similar services
