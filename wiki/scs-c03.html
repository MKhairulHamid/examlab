AWS Certified Security – Specialty (SCS-C03)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified Security – Specialty (SCS-C03) validates a candidate's ability to effectively secure AWS products and services. It is a Specialty-level exam — the deepest security credential AWS offers — testing whether a candidate can design and operate detection and logging pipelines, respond to incidents, secure infrastructure at the edge and inside the network, manage identity and access at scale, protect data in transit and at rest, and govern security across a multi-account organization.

  • Target audience: Security engineers, security architects, and cloud security specialists responsible for securing AWS workloads.
  • Prerequisite experience: The equivalent of 3–5 years of experience securing cloud solutions, plus working knowledge of the shared responsibility model, identity at scale, multi-account governance, incident response, and encryption methodologies.
  • Why candidates take it: To validate specialized, hands-on cloud security capability for security engineer, security architect, and DevSecOps roles, and as the security-focused capstone alongside the Solutions Architect and DevOps tracks.

Out-of-Scope Skills (Do NOT Test)

The official exam guide explicitly lists these job tasks as out of scope for the target candidate. Any question that hinges on them is out of scope — cut it.

  • Designing cryptographic algorithms
  • Analyzing traffic at the packet level
  • Architecting overall cloud deployments
  • Managing end-user compute resources
  • Training machine learning models
Exam tip: SCS-C03 tests securing and operating AWS workloads — how to detect, respond, protect, and govern with the right security service and configuration. The crux of most questions is "which control or service best meets the security, cost, and operational requirement?" — not cryptography internals or packet forensics.

Domain 1 · 16%

Detection

Seeing what is happening in the account. Covers designing monitoring and alerting, building logging solutions and log data lakes, and troubleshooting when monitoring, logging, or alerting breaks.

Task 1.1 — Design and implement monitoring and alerting solutions for an AWS account or organization

Skills in:

  • Analyzing workloads to determine monitoring requirements
  • Designing and implementing workload monitoring strategies (e.g. configuring resource health checks)
  • Aggregating security and monitoring events
  • Creating metrics, alerts, and dashboards to detect anomalous data and events (e.g. Amazon GuardDuty, Amazon Security Lake, AWS Security Hub, Amazon Macie)
  • Creating and managing automations for regular assessments and investigations (e.g. AWS Config conformance packs, Security Hub, AWS Systems Manager State Manager)

Task 1.2 — Design and implement logging solutions

Skills in:

  • Identifying sources for log ingestion and storage based on requirements
  • Configuring logging for AWS services and applications (e.g. an organization CloudTrail trail, a dedicated Amazon CloudWatch logging account, the CloudWatch Logs agent)
  • Implementing log storage and log data lakes (e.g. Security Lake) and integrating with third-party security tools
  • Using AWS services to analyze logs (e.g. CloudWatch Logs Insights, Amazon Athena, Security Hub findings)
  • Using AWS services to normalize, parse, and correlate logs (e.g. Amazon OpenSearch Service, AWS Lambda, Amazon Managed Grafana)
  • Configuring appropriate log sources based on network design, threats, and attacks (e.g. VPC Flow Logs, transit gateway flow logs, Amazon Route 53 Resolver logs)

Task 1.3 — Troubleshoot security monitoring, logging, and alerting solutions

Skills in:

  • Analyzing the functionality, permissions, and configuration of resources (e.g. Lambda function logging, Amazon API Gateway logging, health checks, Amazon CloudFront logging)
  • Remediating misconfiguration of resources (e.g. troubleshooting CloudWatch Agent configurations, troubleshooting missing logs)
Domain 2 · 14%

Incident Response

Preparing for and reacting to security events. Covers designing and testing an incident response plan, and responding to events with forensics, containment, eradication, recovery, and root cause analysis.

Task 2.1 — Design and test an incident response plan

Skills in:

  • Designing and implementing response plans and runbooks (e.g. Systems Manager OpsCenter, Amazon SageMaker AI notebooks)
  • Using AWS service features to prepare for incidents (e.g. provisioning access, deploying security tools, minimizing the blast radius, configuring AWS Shield Advanced protections)
  • Recommending procedures to test and validate the effectiveness of an incident response plan (e.g. AWS Fault Injection Service, AWS Resilience Hub)
  • Using AWS services to automatically remediate incidents (e.g. Systems Manager, Automated Forensics Orchestrator for Amazon EC2, AWS Step Functions, Amazon Application Recovery Controller, Lambda functions)

Task 2.2 — Respond to security events

Skills in:

  • Capturing and storing relevant system and application logs as forensic artifacts
  • Searching and correlating logs for security events across applications and AWS services
  • Validating findings from AWS security services to assess the scope and impact of an event
  • Responding to affected resources by containing and eradicating threats, and recovering resources (e.g. network containment controls, restoring backups)
  • Describing methods to conduct root cause analysis (e.g. Amazon Detective)
Domain 3 · 18%

Infrastructure Security

Securing the perimeter, the compute, and the network. Covers edge protection, hardening and patching compute workloads, and designing network controls and segmentation.

Task 3.1 — Design, implement, and troubleshoot security controls for network edge services

Skills in:

  • Defining and selecting edge security strategies based on anticipated threats and attacks
  • Implementing edge protection (e.g. CloudFront headers, AWS WAF, AWS IoT policies, OWASP Top 10, Amazon S3 CORS, Shield Advanced)
  • Designing edge controls and rules based on requirements (e.g. geography, geolocation, rate limiting, client fingerprinting)
  • Configuring integrations with AWS edge services and third-party services (e.g. ingesting OCSF-format data, third-party WAF rules)

Task 3.2 — Design, implement, and troubleshoot security controls for compute workloads

Skills in:

  • Designing and implementing hardened EC2 AMIs and container images (e.g. Systems Manager, EC2 Image Builder)
  • Applying instance profiles, service roles, and execution roles to authorize compute workloads
  • Scanning compute resources for known vulnerabilities (e.g. Amazon Inspector for container images and Lambda; GuardDuty runtime monitoring)
  • Deploying patches across compute resources and integrating continuous validation (e.g. Systems Manager Patch Manager, Amazon Inspector)
  • Configuring secure administrative access (e.g. Systems Manager Session Manager, EC2 Instance Connect)
  • Discovering and remediating vulnerabilities within a pipeline (e.g. Amazon Q Developer, Amazon CodeGuru Security)
  • Implementing protections and guardrails for generative AI applications (e.g. GenAI OWASP Top 10 for LLM Applications)

Task 3.3 — Design and troubleshoot network security controls

Skills in:

  • Designing and troubleshooting controls to permit or prevent traffic (e.g. security groups, network ACLs, AWS Network Firewall)
  • Designing secure connectivity between hybrid and multi-cloud networks (e.g. AWS Site-to-Site VPN, AWS Direct Connect, MACsec)
  • Configuring secure workload communication between hybrid environments and AWS (e.g. AWS Verified Access)
  • Designing network segmentation based on security requirements (e.g. north/south and east/west protections, isolated subnets)
  • Identifying unnecessary network access (e.g. AWS Verified Access, Network Access Analyzer, Amazon Inspector network reachability findings)
Domain 4 · 20%

Identity and Access Management

The largest domain. Covers designing, implementing, and troubleshooting authentication and authorization for human, application, and system access — the heart of AWS security.

Task 4.1 — Design, implement, and troubleshoot authentication strategies

Skills in:

  • Designing identity solutions for human, application, and system authentication (e.g. AWS IAM Identity Center, Amazon Cognito, MFA, IdP integration)
  • Configuring mechanisms to issue temporary credentials (e.g. AWS STS, Amazon S3 presigned URLs)
  • Troubleshooting authentication issues (e.g. CloudTrail, Amazon Cognito, IAM Identity Center permission sets, AWS Directory Service)

Task 4.2 — Design, implement, and troubleshoot authorization strategies

Skills in:

  • Designing and evaluating authorization controls for human, application, and system access (e.g. Amazon Verified Permissions, IAM paths, IAM Roles Anywhere, resource policies for cross-account access, IAM role trust policies)
  • Designing attribute-based (ABAC) and role-based (RBAC) access control strategies (e.g. resource access based on tags or attributes)
  • Designing, interpreting, and implementing IAM policies by following least privilege (e.g. permission boundaries, session policies)
  • Analyzing authorization failures to determine causes or effects (e.g. IAM Policy Simulator, IAM Access Analyzer)
  • Investigating and correcting unintended permissions or privileges (e.g. IAM Access Analyzer)
Domain 5 · 18%

Data Protection

Keeping data confidential and intact. Covers controls for data in transit, controls for data at rest, and protecting confidential data, credentials, secrets, and cryptographic key material.

Task 5.1 — Design and implement controls for data in transit

Skills in:

  • Requiring encryption when connecting to resources (e.g. ELB security policies, enforcing TLS configurations)
  • Designing secure and private access to resources (e.g. AWS PrivateLink, VPC endpoints, AWS Client VPN, AWS Verified Access)
  • Designing inter-resource encryption in transit (e.g. inter-node encryption for Amazon EMR, Amazon EKS, SageMaker AI; Nitro encryption)

Task 5.2 — Design and implement controls for data at rest

Skills in:

  • Configuring encryption at rest to requirements (e.g. selecting AWS CloudHSM vs. AWS KMS; client-side vs. server-side encryption)
  • Protecting data integrity (e.g. S3 Object Lock, S3 Glacier Vault Lock, versioning, digital code signing, file validation)
  • Designing lifecycle management and retention (e.g. S3 Lifecycle policies, S3 Object Lock, Amazon EFS Lifecycle policies, Amazon FSx for Lustre backup policies)
  • Designing secure replication and backup (e.g. Amazon Data Lifecycle Manager, AWS Backup, ransomware protection, AWS DataSync)

Task 5.3 — Design and implement controls to protect confidential data, credentials, secrets, and key material

Skills in:

  • Designing management and rotation of credentials and secrets (e.g. AWS Secrets Manager)
  • Managing and using imported key material (e.g. rotating imported key material, configuring external key stores)
  • Describing the differences between imported key material and AWS-generated key material
  • Masking sensitive data (e.g. CloudWatch Logs data protection policies, Amazon SNS message data protection)
  • Creating and managing encryption keys and certificates across one or multiple Regions (e.g. AWS KMS customer managed keys, AWS Private Certificate Authority)
Domain 6 · 14%

Security Foundations and Governance

Governing security across the whole organization. Covers centrally deploying and managing accounts, enforcing a secure and consistent deployment strategy, and evaluating the compliance of resources.

Task 6.1 — Develop a strategy to centrally deploy and manage AWS accounts

Skills in:

  • Deploying and configuring organizations by using AWS Organizations
  • Implementing and managing AWS Control Tower in new and existing environments, and deploying optional and custom controls
  • Implementing organization policies to manage permissions (e.g. SCPs, RCPs, AI service opt-out policies, declarative policies)
  • Centrally managing security services (e.g. delegated administrator accounts)
  • Managing AWS account root user credentials (e.g. centralizing root access for member accounts, managing MFA, designing break-glass procedures)

Task 6.2 — Implement a secure and consistent deployment strategy for cloud resources

Skills in:

  • Using infrastructure as code to deploy resources consistently and securely across accounts (e.g. CloudFormation stack sets, third-party IaC, CloudFormation Guard, cfn-lint)
  • Using tags to organize AWS resources into groups (e.g. by department, cost center, environment)
  • Deploying and enforcing policies and configurations from a central source (e.g. AWS Firewall Manager)
  • Securely sharing resources across AWS accounts (e.g. AWS Service Catalog, AWS Resource Access Manager)

Task 6.3 — Evaluate the compliance of AWS resources

Skills in:

  • Creating or enabling rules to detect and remediate noncompliant resources and send notifications (e.g. AWS Config to aggregate alerts and remediate, Security Hub)
  • Using AWS audit services to collect and organize evidence (e.g. AWS Audit Manager, AWS Artifact)
  • Using AWS services to evaluate architecture for compliance with best practices (e.g. AWS Well-Architected Framework tool)
Candidate Prep

How We Help SCS-C03 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 750+.

Who Takes This Exam

  • Security engineers & analysts — already operating detection, IR, and IAM on AWS who want to validate specialized depth.
  • Cloud & security architects — designing secure multi-account landing zones and data-protection strategies.
  • DevSecOps & platform engineers — embedding security controls into pipelines, infrastructure as code, and guardrails at scale.

What Candidates Struggle With

  • The detective-service map — GuardDuty vs. Security Hub vs. Macie vs. Inspector vs. Detective vs. Config, and which one answers a given "what happened / what's wrong" question.
  • Logging architecture — organization CloudTrail trails, a dedicated logging account, Security Lake, and which log source (VPC Flow Logs, Resolver logs, CloudFront logs) sees a given threat.
  • KMS depth — customer managed vs. AWS managed keys, key policies vs. IAM vs. grants, multi-Region keys, imported key material, external key stores, and CloudHSM vs. KMS.
  • IAM authorization evaluation — how SCPs, resource policies, permission boundaries, session policies, and identity policies combine, and why an Allow is or isn't effective.
  • Cross-account & org governance — Organizations, Control Tower, SCPs/RCPs, delegated administration, and Firewall Manager / RAM for central enforcement and sharing.
  • Edge & network protection — WAF vs. Shield Advanced vs. Network Firewall vs. security groups/NACLs, and where each control belongs in a layered defense.

Recommended Study Path

  1. Domain 4 — Identity and Access Management first. The largest domain (20%) and the foundation everything else builds on. Master authentication strategies and how authorization actually evaluates before anything else.
  2. Domain 3 — Infrastructure Security & Domain 5 — Data Protection. 18% each. Edge/compute/network controls and encryption in transit and at rest are the bulk of the hands-on security work.
  3. Domain 1 — Detection. 16%. The detective-service map, logging architecture, and troubleshooting when telemetry breaks.
  4. Domain 2 — Incident Response & Domain 6 — Foundations and Governance. 14% each. Tie it together: respond to events with forensics and containment, and govern security across the whole organization.

AI Explanation Quality for This Exam

SCS-C03 questions are scenario-based and frequently offer several controls that "work," where one is the best fit on security posture, blast radius, cost, or operational overhead. Every explanation must cover WHY a plausible option is inferior — not just "this is wrong." Common patterns to cover:

  • For detection questions: explain why a service misses the signal (e.g. "Macie classifies sensitive data in S3; it does not detect the IAM credential-exfiltration behavior that GuardDuty surfaces").
  • For IAM questions: explain why an Allow is ineffective (e.g. "an identity policy Allow is overridden by an SCP Deny at the organization level, so the action still fails").
  • For data-protection questions: explain why a control is weaker (e.g. "SSE-S3 meets encryption-at-rest but cannot give the customer-controlled key rotation and audit trail that SSE-KMS with a customer managed key provides").

Question Writing Rules (SCS-C03 specific)

  • Scenario stems — "A security engineer must…" Establish the real constraint (threat, blast radius, compliance, least privilege, cost) before asking.
  • Best-of-several-workable answers — most distractors are valid security services but lose on posture, scope, operational overhead, or one unmet requirement. Engineer the crux to be the trade-off.
  • Least-privilege and defense-in-depth framing — the right answer is usually the most scoped, layered, purpose-fit control. Make overly broad or single-layer options the plausible-but-wrong choices.
  • Cross-domain wiring — SCS scenarios routinely span detection, IAM, data protection, and governance. Let questions combine services as the real exam does.
  • No cryptographic algorithm design, no packet-level analysis, no ML training — keep questions inside the in-scope service set; cut anything that hinges on out-of-scope depth.

Distractor Quality Guidelines

  • Use real security confusions as distractors: GuardDuty vs. Macie vs. Inspector vs. Detective; SCP Deny vs. identity-policy Allow; KMS vs. CloudHSM; WAF vs. Shield Advanced vs. Network Firewall.
  • For detection questions: make one distractor a service that does not see the signal in question, and one a control that detects too late or in the wrong layer.
  • For IAM questions: make one distractor an Allow that a higher-level Deny overrides, and one an overly broad grant that breaks least privilege.
  • For data-protection questions: make one distractor a weaker encryption mode, and one a control that does not meet the integrity, retention, or key-custody requirement.

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — the conceptual reason it best meets every stated constraint
  • Why each distractor is wrong — one sentence per option, naming the specific unmet constraint or excess exposure
  • Concept link — which domain/task statement this tests (e.g. "D4 · Task 4.2 — Authorization strategies")
  • Service context — if a service appears, one line on what it does and what distinguishes it from the similar services in the distractors
