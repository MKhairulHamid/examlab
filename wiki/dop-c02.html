AWS Certified DevOps Engineer – Professional (DOP-C02)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified DevOps Engineer – Professional (DOP-C02) validates technical expertise in provisioning, operating, and managing distributed application systems on AWS. It is a Professional-level exam that tests whether a candidate can implement continuous delivery, automate security and governance, deploy monitoring at scale, and build systems that are highly available, scalable, and self-healing.

  • Target audience: DevOps engineers, platform engineers, and SREs who provision, operate, and manage AWS environments and own delivery and reliability.
  • Prerequisite experience: 2+ years of experience provisioning, operating, and managing AWS environments, plus experience with the software development lifecycle and programming/scripting.
  • Why candidates take it: To validate senior DevOps capability for lead/platform/SRE roles, and as the advanced continuation of the Developer Associate (DVA) or CloudOps Associate (SOA) path.

Out-of-Scope Skills (Do NOT Test)

The official exam guide explicitly lists these job tasks as out of scope for the target candidate. Any question that hinges on them is out of scope — cut it.

  • Advanced networking knowledge (e.g. advanced routing algorithms, failover techniques)
  • Providing deep-level security recommendations to developers
  • Designing, querying, and optimizing the performance of databases
  • Developing full-stack application code
Exam tip: DOP-C02 tests automation and operational judgment at scale — how to build pipelines, codify infrastructure, observe systems, and respond to events. The crux of most questions is "which automated, least-operational-overhead solution fits?" — not raw service trivia.

Domain 1 · 22%

SDLC Automation

The largest domain. Covers continuous integration and delivery — pipelines, automated testing, artifact management, and deployment strategies for every compute platform.

Task 1.1 — Implement CI/CD pipelines

Knowledge of: SDLC concepts, phases, and models; pipeline deployment patterns for single- and multi-account environments.

Skills in:

  • Configuring code, image, and artifact repositories
  • Using version control to integrate pipelines with application environments
  • Setting up build processes (e.g. AWS CodeBuild)
  • Managing build and deployment secrets (e.g. AWS Secrets Manager, AWS Systems Manager Parameter Store)
  • Determining appropriate deployment strategies (e.g. AWS CodeDeploy)

Task 1.2 — Integrate automated testing into CI/CD pipelines

Knowledge of: Different types of tests (unit, integration, acceptance, UI, security scans); reasonable use of different tests at different pipeline stages.

Skills in:

  • Running builds or tests when generating pull requests or code merges (e.g. CodeBuild)
  • Running load/stress tests, performance benchmarking, and application testing at scale
  • Measuring application health based on application exit codes
  • Automating unit tests and code coverage
  • Invoking AWS services in a pipeline for testing

Task 1.3 — Build and manage artifacts

Knowledge of: Artifact use cases and secure management; methods to create and generate artifacts; artifact lifecycle considerations.

Skills in:

  • Creating and configuring artifact repositories (e.g. AWS CodeArtifact, Amazon S3, Amazon ECR)
  • Configuring build tools for generating artifacts (e.g. CodeBuild, AWS Lambda)
  • Automating Amazon EC2 instance and container image build processes (e.g. EC2 Image Builder)

Task 1.4 — Implement deployment strategies for instance, container, and serverless environments

Knowledge of: Deployment methodologies for various platforms (EC2, ECS, EKS, Lambda); application storage patterns (EFS, S3, EBS); mutable vs. immutable deployment patterns; tools for distributing code (CodeDeploy, Image Builder).

Skills in:

  • Configuring security permissions to allow access to artifact repositories (e.g. IAM, CodeArtifact)
  • Configuring deployment agents (e.g. CodeDeploy agent)
  • Troubleshooting deployment issues
  • Using different deployment methods (e.g. blue/green, canary)
Domain 2 · 17%

Configuration Management and IaC

Codifying infrastructure and governance. Covers reusable IaC templates, multi-account/multi-Region account automation, and large-scale operational automation.

Task 2.1 — Define cloud infrastructure and reusable components to provision and manage systems throughout their lifecycle

Knowledge of: IaC options and tools for AWS; change management processes for IaC-based platforms; configuration management services and strategies.

Skills in:

  • Composing and deploying IaC templates (e.g. AWS SAM, AWS CloudFormation, AWS CDK)
  • Applying CloudFormation stack sets across multiple accounts and AWS Regions
  • Determining optimal configuration management services (e.g. AWS OpsWorks, AWS Systems Manager, AWS Config, AWS AppConfig)
  • Implementing infrastructure patterns, governance controls, and security standards into reusable IaC templates (e.g. AWS Service Catalog, CloudFormation modules, AWS CDK)

Task 2.2 — Deploy automation to create, onboard, and secure AWS accounts in a multi-account or multi-Region environment

Knowledge of: AWS account structures, best practices, and related AWS services.

Skills in:

  • Standardizing and automating account provisioning and configuration
  • Creating, consolidating, and centrally managing accounts (e.g. AWS Organizations, AWS Control Tower)
  • Applying IAM solutions for multi-account and complex organization structures (e.g. SCPs, assuming roles)
  • Implementing and developing governance and security controls at scale (AWS Config, Control Tower, Security Hub, Amazon Detective, Amazon GuardDuty, Service Catalog, SCPs)

Task 2.3 — Design and build automated solutions for complex tasks and large-scale environments

Knowledge of: AWS services and solutions to automate tasks and processes; methods and strategies to interact with the AWS software-defined infrastructure.

Skills in:

  • Automating system inventory, configuration, and patch management (e.g. Systems Manager, AWS Config)
  • Developing AWS Lambda function automations for complex scenarios (e.g. AWS SDKs, Lambda, AWS Step Functions)
  • Automating the configuration of software applications to the desired state (e.g. OpsWorks, Systems Manager State Manager)
  • Maintaining software compliance (e.g. Systems Manager)
Domain 3 · 15%

Resilient Cloud Solutions

Designing for failure and growth. Covers high availability, scalability, and automated recovery against measurable RTO/RPO targets.

Task 3.1 — Implement highly available solutions to meet resilience and business requirements

Knowledge of: Multi-AZ and multi-Region deployments (compute and data layers); SLAs; replication and failover methods for stateful services; techniques to achieve high availability.

Skills in:

  • Translating business requirements into technical resiliency needs
  • Identifying and remediating single points of failure in existing workloads
  • Enabling cross-Region solutions where available (e.g. DynamoDB, RDS, Route 53, S3, CloudFront)
  • Configuring load balancing to support cross-AZ services
  • Configuring applications and related services to support multiple AZs and Regions while minimizing downtime

Task 3.2 — Implement solutions that are scalable to meet business requirements

Knowledge of: Appropriate metrics for scaling services; loosely coupled and distributed architectures; serverless architectures; container platforms.

Skills in:

  • Identifying and remediating scaling issues
  • Identifying and implementing appropriate auto scaling, load balancing, and caching solutions
  • Deploying container-based applications (e.g. Amazon ECS, Amazon EKS)
  • Deploying workloads in multiple Regions for global scalability
  • Configuring serverless applications (e.g. Amazon API Gateway, AWS Lambda, AWS Fargate)

Task 3.3 — Implement automated recovery processes to meet RTO and RPO requirements

Knowledge of: Disaster recovery concepts (RTO, RPO); AWS Backup and recovery strategies (e.g. pilot light, warm standby); recovery procedures.

Skills in:

  • Testing failover of Multi-AZ and multi-Region workloads (e.g. RDS, Aurora, Route 53, CloudFront)
  • Identifying and implementing appropriate cross-Region AWS Backup and recovery strategies (e.g. AWS Backup, S3, Systems Manager)
  • Configuring a load balancer to recover from backend failure
Domain 4 · 15%

Monitoring and Logging

Observability at scale. Covers collecting and storing logs and metrics, analyzing them to detect issues, and automating monitoring and event management.

Task 4.1 — Configure the collection, aggregation, and storage of logs and metrics

Knowledge of: How to monitor applications and infrastructure; CloudWatch metrics (namespaces, metrics, dimensions, resolution); real-time log ingestion; encryption options for logs/metrics (client-side, server-side, KMS); security configurations (IAM roles/permissions for log collection).

Skills in:

  • Securely storing and managing logs
  • Creating CloudWatch metrics from log events using metric filters
  • Creating CloudWatch metric streams (e.g. Amazon S3 or Kinesis Data Firehose)
  • Collecting custom metrics (e.g. using the CloudWatch agent)
  • Managing log storage lifecycles (e.g. S3 lifecycles, CloudWatch log group retention)
  • Processing log data using CloudWatch log subscriptions (e.g. Kinesis, Lambda, OpenSearch Service)
  • Searching log data using filter and pattern syntax or CloudWatch Logs Insights
  • Configuring encryption of log data (e.g. AWS KMS)

Task 4.2 — Audit, monitor, and analyze logs and metrics to detect issues

Knowledge of: Anomaly detection alarms (CloudWatch anomaly detection); common CloudWatch metrics and logs; Amazon Inspector and common assessment templates; AWS Config rules; CloudTrail log events.

Skills in:

  • Building CloudWatch dashboards and Amazon QuickSight visualizations
  • Associating CloudWatch alarms with CloudWatch metrics (standard and custom)
  • Configuring AWS X-Ray for different services (e.g. containers, API Gateway, Lambda)
  • Analyzing real-time log streams (e.g. using Kinesis Data Streams)
  • Analyzing logs with AWS services (e.g. Amazon Athena, CloudWatch Logs Insights)

Task 4.3 — Automate monitoring and event management of complex environments

Knowledge of: Event-driven, asynchronous design patterns (e.g. S3 Event Notifications or EventBridge events to SNS or Lambda); auto scaling capabilities across services; alert notification and action capabilities; health check capabilities (ALB target groups, Route 53).

Skills in:

  • Configuring solutions for auto scaling (e.g. DynamoDB, EC2 Auto Scaling groups, RDS storage auto scaling, ECS capacity provider)
  • Creating CloudWatch custom metrics, metric filters, alarms, and notifications (e.g. SNS, Lambda)
  • Configuring S3 events to process log files (e.g. via Lambda) and deliver to another destination (e.g. OpenSearch Service, CloudWatch Logs)
  • Configuring EventBridge to send notifications based on a particular event pattern
  • Installing and configuring agents on EC2 instances (e.g. SSM Agent, CloudWatch agent)
  • Configuring AWS Config rules to remediate issues
  • Configuring health checks (e.g. Route 53, ALB)
Domain 5 · 14%

Incident and Event Response

Detect, respond, and recover. Covers managing event sources, applying configuration changes in response to events, and troubleshooting system and application failures.

Task 5.1 — Manage event sources to process, notify, and take action in response to events

Knowledge of: AWS services that generate, capture, and process events (e.g. AWS Health, EventBridge, CloudTrail); event-driven architectures (e.g. fan out, event streaming, queuing).

Skills in:

  • Integrating AWS event sources (e.g. AWS Health, EventBridge, CloudTrail)
  • Building event processing workflows (e.g. SQS, Kinesis, SNS, Lambda, Step Functions)

Task 5.2 — Implement configuration changes in response to events

Knowledge of: Fleet management services (e.g. Systems Manager, AWS Auto Scaling); configuration management services (e.g. AWS Config).

Skills in:

  • Applying configuration changes to systems
  • Modifying infrastructure configurations in response to events
  • Remediating a non-desired system state

Task 5.3 — Troubleshoot system and application failures

Knowledge of: AWS metrics and logging services (e.g. CloudWatch, X-Ray); AWS service health services (e.g. AWS Health, CloudWatch, Systems Manager OpsCenter); root cause analysis.

Skills in:

  • Analyzing failed deployments (e.g. CodePipeline, CodeBuild, CodeDeploy, CloudFormation, CloudWatch synthetic monitoring)
  • Analyzing incidents regarding failed processes (e.g. auto scaling, ECS, EKS)
Domain 6 · 17%

Security and Compliance

Automating security at scale. Covers identity and access management for large organizations, automated security controls and data protection, and security monitoring and auditing.

Task 6.1 — Implement techniques for identity and access management at scale

Knowledge of: Appropriate usage of IAM entities for human and machine access (users, groups, roles, identity providers, identity-based/resource-based/session policies); identity federation (IAM identity providers, IAM Identity Center); permission delegation via IAM permissions boundaries; organizational SCPs.

Skills in:

  • Designing policies to enforce least privilege access
  • Implementing role-based and attribute-based access control patterns
  • Automating credential rotation for machine identities (e.g. AWS Secrets Manager)
  • Managing permissions to control access to human and machine identities (e.g. MFA, AWS STS, IAM profiles)

Task 6.2 — Apply automation for security controls and data protection

Knowledge of: Network security components (security groups, network ACLs, routing, Network Firewall, WAF, Shield); certificates and PKI; data management (classification, encryption, key management, access controls).

Skills in:

  • Automating the application of security controls in multi-account and multi-Region environments (e.g. Security Hub, Organizations, Control Tower, Systems Manager)
  • Combining security controls to apply defense in depth (e.g. ACM, WAF, AWS Config + rules, Security Hub, GuardDuty, security groups, network ACLs, Detective, Network Firewall)
  • Automating the discovery of sensitive data at scale (e.g. Amazon Macie)
  • Encrypting data in transit and at rest (e.g. AWS KMS, AWS CloudHSM, ACM)

Task 6.3 — Implement security monitoring and auditing solutions

Knowledge of: Security auditing services and features (CloudTrail, AWS Config, VPC Flow Logs, CloudFormation drift detection); services for identifying vulnerabilities and events (GuardDuty, Inspector, IAM Access Analyzer, AWS Config); common cloud security threats (insecure web traffic, exposed access keys, public/unencrypted S3 buckets).

Skills in:

  • Implementing robust security auditing
  • Configuring alerting based on unexpected or anomalous security events
  • Configuring service and application logging (e.g. CloudTrail, CloudWatch Logs)
  • Analyzing logs, metrics, and security findings
Candidate Prep

How We Help DOP-C02 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 750+.

Who Takes This Exam

  • Working DevOps and platform engineers — already running CI/CD and AWS infrastructure who want to validate senior capability.
  • Associate-level graduates — DVA or SOA holders moving up to the Professional tier on the operations track.
  • SREs and cloud automation leads — engineers responsible for reliability, observability, and incident response at scale.

What Candidates Struggle With

  • Deployment strategy nuance — blue/green vs. canary vs. rolling, and how CodeDeploy implements each across EC2, ECS, EKS, and Lambda. The exam tests trade-offs, not definitions.
  • Multi-account governance — Organizations, Control Tower, SCPs, and how account factory and StackSets automate provisioning and guardrails at scale.
  • Event-driven automation chains — wiring EventBridge, SNS, SQS, Lambda, Step Functions, and Config remediation into reliable, least-overhead workflows.
  • Resilience math — matching RTO/RPO targets to the right DR strategy (backup/restore, pilot light, warm standby, multi-site) and the cost/complexity each implies.
  • Observability depth — metric filters, subscription filters, metric streams, X-Ray, anomaly detection, and which one a given symptom calls for.
  • Security automation — least-privilege at scale, credential rotation, and combining GuardDuty/Security Hub/Config/Macie into defense in depth.

Recommended Study Path

  1. Domain 1 — SDLC Automation first. Highest weight (22%) and the spine of the exam. Build the CodePipeline → CodeBuild → CodeDeploy mental model and the blue/green vs. canary distinction before anything else.
  2. Domain 2 — Configuration Management & IaC. Tied for second (17%). CloudFormation/CDK/StackSets and multi-account governance underpin many cross-domain scenarios.
  3. Domain 6 — Security & Compliance. Also 17%. IAM at scale, data protection automation, and security monitoring recur throughout the exam.
  4. Domains 3 & 4 — Resilience and Observability. 15% each. Resilient architectures and deep monitoring are heavily interleaved with incident response.
  5. Domain 5 — Incident & Event Response last. Smallest (14%) but it synthesizes everything else — event sources, automated remediation, and troubleshooting failed deployments and processes.

AI Explanation Quality for This Exam

DOP-C02 questions are long, multi-service scenarios where several options work but one is best (least overhead, most automated, meets all constraints). Every explanation must cover WHY a plausible option is inferior — not just "this is wrong." Common patterns to cover:

  • For deployment questions: explain why a strategy fails a constraint (e.g. "in-place deployment cannot meet the zero-downtime requirement; blue/green can shift traffic only after the new fleet is healthy").
  • For automation questions: explain why a manual or higher-overhead option loses (e.g. "a custom Lambda poller reinvents what an EventBridge rule does natively — more code to maintain").
  • For resilience questions: explain why an option misses the stated RTO/RPO (e.g. "backup/restore cannot meet a 5-minute RTO; warm standby keeps a scaled-down copy already running").

Question Writing Rules (DOP-C02 specific)

  • Long scenario stems — "A company runs a multi-account workload and needs to…" Professional questions establish real constraints (downtime budget, RTO/RPO, account structure, compliance) before asking.
  • Best-of-several-workable answers — most distractors are technically valid but lose on overhead, automation, or one unmet constraint. Engineer the crux to be the trade-off.
  • Automation and least-operational-overhead framing — the right answer is usually the most automated, managed, event-driven option. Make manual/custom-code options the plausible-but-wrong choices.
  • Cross-domain wiring — DOP scenarios routinely span pipelines, IaC, monitoring, and response. Let questions combine services as the real exam does.
  • No DB tuning, no full-stack code, no advanced routing — keep questions inside the in-scope service set; cut anything that hinges on out-of-scope depth.

Distractor Quality Guidelines

  • Use real DevOps confusions as distractors: blue/green vs. canary vs. rolling; CodeDeploy vs. CodePipeline vs. CodeBuild; EventBridge vs. SNS vs. SQS; pilot light vs. warm standby.
  • For deployment questions: make one distractor a strategy that cannot meet the stated downtime/rollback constraint, and one a higher-overhead manual approach.
  • For automation questions: make one distractor a custom Lambda/cron solution that reinvents a native managed capability (EventBridge rule, Config remediation, StackSets).
  • For resilience questions: make each distractor a DR strategy whose RTO/RPO or cost profile is wrong for the scenario.

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — the conceptual reason it best meets every stated constraint
  • Why each distractor is wrong — one sentence per option, naming the specific unmet constraint or excess overhead
  • Concept link — which domain/task statement this tests (e.g. "D1 · Task 1.4 — Deployment strategies")
  • Service context — if a service appears, one line on what it does and what distinguishes it from the similar services in the distractors
