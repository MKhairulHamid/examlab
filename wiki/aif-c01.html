AWS Certified AI Practitioner (AIF-C01)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

Source: All domain content on this page is derived from the official AWS Certified AI Practitioner Exam Guide. This is the authoritative reference for question authoring.

What This Exam Is

The AWS Certified AI Practitioner (AIF-C01) is a Foundational-level certification. It validates that a candidate understands AI/ML and Generative AI concepts and knows how to apply them using AWS services — without requiring any hands-on coding or model-building skills.

AttributeValue
Exam codeAIF-C01
LevelFoundational
Target audienceBusiness analysts, project managers, developers, data scientists — anyone with up to 6 months of AWS AI/ML exposure
PrerequisitesNone (recommended: basic AWS familiarity)
Duration90 minutes
Scored questions50
Unscored questions15 (not counted toward score, used by AWS to evaluate future items)

Scoring modelCompensatory — no minimum per domain

Key insight for simulation design: Candidates see 65 questions but only 50 count. They cannot tell which 15 are unscored. Our simulations should mirror this ratio — include a small fraction of "calibration" questions — or simply use 50 questions and be transparent that the real exam has 15 extra unscored items.
Exam Structure

Question Format & Scoring

Four question types, one scoring model, zero guessing penalty.

Question Types

TypeDescriptionSimulation notes
Multiple choice Pick 1 correct answer from 4 options (1 correct + 3 distractors) Standard format — most questions will be this type. All or nothing scoring.
Multiple response Pick 2 or more correct answers from 5+ options. All selections must be correct. Must label "Select X answers". Partial credit is NOT awarded — candidate must get all correct.
Ordering Arrange 3–5 items in the correct sequence Drag-and-drop or numbered inputs. All items must be in exact correct order to score.
Matching Match responses to 3–7 prompts. All pairs must be correct. Two-column layout. All matches must be correct for any points.
Important for question authors: Multiple response, ordering, and matching questions are all-or-nothing. Do not design partial-credit scoring logic for AIF-C01 simulations.

Scoring Model

The exam uses a compensatory scoring model. This means a weak domain score can be compensated by a strong score in another domain — there is no domain-level pass/fail requirement. The scaled score of 700/1,000 is the single threshold.

Out-of-Scope Skills (Do NOT Test)

These topics are explicitly excluded from the exam. Do not write questions about them.

  • Developing or coding AI/ML models
  • Implementing data engineering or feature engineering pipelines
  • Hyperparameter tuning
  • Building or deploying ML pipelines end-to-end
  • Mathematical or statistical analysis of model internals
  • Implementing security controls or compliance frameworks
The AIF-C01 tests understanding and judgment, not hands-on implementation. Questions should ask candidates to identify, describe, select, or explain — not to write code or configure infrastructure step-by-step.

Domain 1 — 20%

Fundamentals of AI and ML

~10 scored questions · Task statements 1.1 – 1.3

Task 1.1 — Explain basic AI concepts and terminologies

  • Define: AI, ML, deep learning, neural networks, computer vision, NLP, model, algorithm, training vs. inferencing, bias, fairness, fit, LLM, GenAI, agentic AI
  • Distinguish between AI, ML, GenAI, deep learning, and agentic AI
  • Types of inferencing: batch, real-time, asynchronous, serverless
  • Types of data: labeled/unlabeled, tabular, time-series, image, text, structured/unstructured
  • Learning methods: supervised, unsupervised, reinforcement learning

Task 1.2 — Identify practical use cases for AI

  • Scenarios where AI/ML adds value: decision support, scalability, automation
  • When AI/ML is not appropriate: cost-benefit tradeoffs, deterministic outcome requirements
  • Matching technique to problem: regression, classification, clustering
  • Real-world applications: computer vision, NLP, speech recognition, recommendation systems, fraud detection, forecasting, knowledge bases, agentic AI
  • AWS managed AI services: Amazon SageMaker AI, Amazon Transcribe, Amazon Translate, Amazon Comprehend, Amazon Lex, Amazon Polly
  • Choosing between traditional ML vs. foundation models (regulatory, explainability, operational constraints)

Task 1.3 — Describe the AI/ML development lifecycle

  • ML pipeline stages and differences
  • Foundation model sources: open-source pre-trained vs. custom training
  • Production deployment: managed API service vs. self-hosted API
  • AWS services for lifecycle management: Amazon Bedrock, Amazon Q, SageMaker AI
  • MLOps concepts: experimentation, repeatability, scalable systems, technical debt, model monitoring, retraining
  • Model performance metrics: accuracy, precision, recall, F1 score
  • Business metrics: cost per user, development costs, customer feedback, ROI
Domain 2 — 24%

Fundamentals of Generative AI

~12 scored questions · Task statements 2.1 – 2.3

Task 2.1 — Explain basic GenAI concepts

Core concepts

  • Tokens, chunking, embeddings, vectors — how text is represented and processed
  • Prompt engineering fundamentals
  • Transformer-based LLMs and how they work at a high level
  • Foundation models (FMs) vs. traditional ML models
  • Multi-modal models (text + image + audio)
  • Diffusion models (image generation)

GenAI use cases

  • Image, video, and audio generation
  • Summarization, translation, code generation
  • AI assistants, customer service agents
  • Search, recommendation engines

Foundation Model lifecycle

Data selection → Model selection → Pre-training → Fine-tuning → Evaluation → Deployment → Feedback (loop back)

Agentic AI concepts

  • Multi-agent system patterns
  • Model Context Protocol (MCP)
  • Multi-agent communication patterns
  • Memory management, tool usage, workflow orchestration

Also: token-based pricing model and context engineering in FM applications.

Task 2.2 — Capabilities and limitations of GenAI

AdvantagesLimitations
Adaptability across tasksHallucinations (factually incorrect outputs)
Responsive / conversationalLimited interpretability
Rich content generationInaccuracy in specific domains
General-purpose knowledgeNondeterminism (same input → different outputs)

Model selection factors: modality, performance requirements, constraints, compliance, cost, latency, complexity.

Business value metrics: cross-domain performance, ROI, efficiency, conversion rate, ARPU, accuracy, customer lifetime value.

Task 2.3 — AWS infrastructure for GenAI applications

  • Amazon Bedrock — fully managed FM access and customization
  • Amazon SageMaker AI — end-to-end ML platform
  • SageMaker JumpStart — pre-trained model hub
  • Amazon Q, Amazon Kiro, Amazon Strands Agents — specialized AI assistants
  • Amazon Bedrock AgentCore — agentic application infrastructure

AWS advantages: accessibility, lower barrier to entry, built-in security/compliance, speed to market.

Cost tradeoffs: responsiveness, availability, redundancy, regional coverage, token-based pricing, provisioned throughput, custom models.

Domain 3 — 28%

Applications of Foundation Models

~14 scored questions · Task statements 3.1 – 3.4 · Highest-weight domain

This is the largest domain by weight. Question writers should prioritize depth here — especially RAG, prompt engineering techniques, and model evaluation.

Task 3.1 — Design considerations for FM applications

  • FM selection criteria: cost, modality, latency, multi-lingual support, model size, complexity, customization needs, input/output length, prompt caching
  • Inference parameters and effects: temperature (randomness), input/output token length
  • Retrieval Augmented Generation (RAG): what it is, business applications, implemented via Amazon Bedrock Knowledge Bases
  • Vector database services: Amazon OpenSearch, Aurora, Neptune, RDS for PostgreSQL
  • Cost tradeoffs of customization approaches: pre-training, fine-tuning, in-context learning, RAG, model distillation
  • AI agents and their business applications

Task 3.2 — Effective prompt engineering

Core techniques

TechniqueDescription
Zero-shotNo examples provided; model relies entirely on instruction
Single-shot (one-shot)One example provided in the prompt
Few-shotMultiple examples provided to guide the model's output format
Chain-of-thoughtPrompt instructs the model to reason step-by-step before answering
Prompt templatesReusable prompt structures with placeholders

Prompt constructs

  • Context, instruction, negative prompts (what NOT to do)
  • Specificity and concision best practices
  • Using guardrails and response quality controls
  • Prompt versioning and management: Amazon Bedrock Prompt Management

Prompt risks and attacks

  • Prompt injection — malicious input overrides instructions
  • Prompt poisoning — corrupting the model's behavior via crafted inputs
  • Prompt hijacking — redirecting the model to unintended tasks
  • Jailbreaking — bypassing model safety guardrails

Task 3.3 — Training and fine-tuning

ApproachWhen to use
Pre-trainingTrain a model from scratch on large general datasets — very expensive
Continuous pre-trainingExtend a model's knowledge on new unlabeled domain data
Fine-tuningAdapt a pre-trained model to a specific task using labeled examples
Instruction tuningFine-tune to follow instructions more reliably
Domain adaptationFine-tune on domain-specific text (e.g., legal, medical)
Transfer learningReuse learned representations for a related but different task
RLHFReinforcement Learning from Human Feedback — align output with human preferences
DistillationTrain a smaller model to mimic a larger one — reduces cost/latency

Data preparation considerations: curation, governance, dataset size, labeling quality, representativeness.

Task 3.4 — FM performance evaluation

Evaluation approaches

  • Human-in-the-loop — human raters judge output quality
  • Benchmark datasets — standardized test sets with known answers
  • LLM-as-a-judge — a separate model evaluates the output
  • Amazon Bedrock Model Evaluation — managed evaluation service

Automated metrics

MetricWhat it measures
ROUGERecall-oriented — overlap between generated and reference summaries
BLEUPrecision-oriented — n-gram overlap; used for translation quality
BERTScoreSemantic similarity using BERT embeddings; captures meaning beyond exact word match
LLM-as-a-judgeFlexible scoring of open-ended outputs using another LLM

Business metrics

  • Task completion rate, user satisfaction, cost per interaction
  • Productivity improvements, user engagement, business objective alignment
Domain 4 — 14%

Guidelines for Responsible AI

~7 scored questions · Task statements 4.1 – 4.2

Task 4.1 — Developing responsible AI systems

Pillars of responsible AI

  • Bias — systematic errors that produce unfair outcomes for certain groups
  • Fairness — equitable treatment across demographic groups
  • Inclusivity — diverse and representative training data
  • Robustness — reliable performance across varied inputs and edge cases
  • Safety — preventing harmful outputs
  • Veracity — factual accuracy and groundedness

Legal risks of GenAI

  • Intellectual property infringement (training data, generated content)
  • Biased or discriminatory model outputs
  • Loss of customer trust from hallucinations or errors
  • End user harm from incorrect advice

Dataset considerations

  • Inclusivity and diversity of training data
  • Curated and balanced datasets reduce bias variance
  • Effects of overfitting and underfitting on demographic groups

AWS tools for responsible AI

  • Amazon Bedrock Guardrails — content filtering and safety policies
  • Amazon SageMaker Clarify — bias detection and explainability
  • SageMaker Model Monitor — continuous monitoring for drift and bias in production
  • Amazon Augmented AI (A2I) — human review workflows for ML predictions

Task 4.2 — Transparent and explainable models

ConceptDescription
Transparent modelDecisions can be traced to specific inputs or rules (e.g., decision trees)
Explainable AI (XAI)Techniques to explain why a model made a specific prediction
Black-box modelInternal logic not interpretable (e.g., large neural networks)

AWS tools: SageMaker Model Cards (document model behavior), SageMaker Clarify, Bedrock Model Evaluations, open-source models and licensing.

Tradeoff: Higher interpretability often means lower performance — and vice versa. Candidates must recognize when each is appropriate.

Human-centered design principle: provide user-feedback mechanisms and make AI decision criteria visible to end users.

Domain 5 — 14%

Security, Compliance, and Governance for AI Solutions

~7 scored questions · Task statements 5.1 – 5.2

Task 5.1 — Securing AI systems

AWS security services for AI

  • IAM — roles, policies, and least-privilege permissions
  • Encryption at rest and in transit
  • Amazon Macie — sensitive data discovery in S3
  • AWS PrivateLink — private connectivity to AWS services without internet exposure
  • Amazon Bedrock Guardrails — content filtering, output validation
  • Amazon Bedrock AgentCore Identity and policy — identity and access for agentic workloads
  • AWS Shared Responsibility Model — customer vs. AWS security obligations

Secure data engineering

  • Assessing data quality before training
  • Privacy-enhancing technologies (differential privacy, anonymization)
  • Data access controls and integrity checks
  • Data lineage and cataloging: SageMaker Model Cards

AI-specific security threats

  • Prompt injection — malicious input hijacking model behavior
  • Data leakage — model memorizing and exposing training data
  • Hallucination risk — incorrect outputs leading to harmful decisions
  • Toxicity — harmful or offensive content generation

Hallucination mitigation

  • RAG grounding — anchor responses to retrieved factual sources
  • Output validation — post-processing checks on generated content
  • Confidence scoring — flag low-confidence outputs for human review

Task 5.2 — Governance and compliance

AWS governance services

  • AWS Config — resource configuration compliance tracking
  • Amazon Inspector — vulnerability assessments
  • AWS Audit Manager — continuous audit evidence collection
  • AWS Artifact — compliance reports and agreements
  • AWS CloudTrail — API activity logging and audit trail
  • AWS Trusted Advisor — best-practice recommendations

Data governance strategies

  • Data lifecycle management (creation → archival → deletion)
  • Logging and observability of AI interactions
  • Data residency requirements (where data is stored and processed)
  • Monitoring for model drift and anomalies
  • Retention policies

Governance protocols

  • AI governance policies and review cadence
  • Generative AI Security Scoping Matrix
  • Transparency standards for AI-generated content
  • Team training requirements for AI safety
Candidate Prep

How We Help AIF-C01 Candidates

Strategy for what the platform delivers — beyond just questions.

Who Takes This Exam

The AIF-C01 candidate is typically not a hands-on ML engineer. They are often:

  • Business analysts or project managers evaluating AI solutions
  • Software developers wanting to add AI knowledge to their skill set
  • Cloud practitioners expanding from a CLF-C02 foundation
  • Data professionals who use AI tools but don't build models

This shapes how we write questions: scenario-based judgment calls, not implementation syntax.

What Candidates Struggle With

Pain pointOur response
Confusing AI/ML/GenAI terminology (tokens, embeddings, RAG, fine-tuning)Define terms clearly in explanations; link related concepts
Knowing when to use which AWS serviceComparison questions; service selection scenarios
Understanding prompt engineering without having tried itConcrete examples in question stems; explain the "why" in AI explanations
Responsible AI feels abstractScenario questions with real business consequences (bias in hiring, hallucination in medical advice)
Security domain — overlap with other certsFocus on AI-specific threats (prompt injection, data leakage) not generic AWS security

Recommended Study Path

  1. Start with Domain 1 — build the vocabulary first. Without it, D2–D3 questions are confusing.
  2. Deep-dive Domain 2 + 3 together — they are tightly coupled (GenAI concepts feed directly into FM application design).
  3. Study Domain 4 — Responsible AI is scenario-heavy; practice applying the concepts.
  4. Finish with Domain 5 — Security/governance overlaps partially with other AWS certs; prioritize the AI-specific content.
  5. Take a full 65-question timed simulation — build exam stamina and identify weak domains.

AI Explanation Quality for This Exam

Because the exam tests conceptual understanding, our Claude-powered explanations should:

  • Explain why each distractor is wrong (not just why the correct answer is right)
  • Relate the concept to a real AWS service or concrete scenario when possible
  • Flag terminology that appears elsewhere in the exam (e.g., "temperature also comes up in Domain 3 when selecting inference parameters")

Question Writing Rules

  • Scenario-based stems — "A company wants to…" not "What is X?" Abstract recall is rarely tested; applied judgment is.
  • Single best answer — for multiple choice, there must be one clearly best answer. Other options should be plausible but wrong for a specific reason.
  • No implementation questions — never ask candidates to write code, configure step-by-step, or calculate model math.
  • Service knowledge — questions can name specific AWS services (Bedrock, SageMaker Clarify, etc.), but test whether the candidate can identify the right one, not how to configure it.
  • Responsible AI scenarios — ground D4 questions in real business impact: hiring bias, loan approval, medical advice, content moderation.
  • Prompt engineering — questions should test recognition of technique names and when to apply them, not memorization of syntax.

Distractor Quality Guidelines

  • Each distractor should be wrong for a specific, teachable reason — not just randomly plausible.
  • Avoid distractors that are obviously out of scope (e.g., writing Python code for a question about service selection).
  • Common distractor traps: confusing similar AWS services, mixing up fine-tuning vs. RAG vs. prompt engineering as solutions, confusing ROUGE with BLEU.

AI Explanation Template

Each question should have an explanation that covers:

  1. Why the correct answer is right — with a brief conceptual reason
  2. Why each distractor is wrong — one sentence per option
  3. Concept link — which domain/task statement this tests
  4. AWS service context — if the question involves a service, briefly describe what it does
Sample question to validate difficulty: A good AIF-C01 question should be answerable by a thoughtful candidate who has studied the official exam guide — but should stump a candidate who only skimmed. If a question requires deep model-building knowledge, it is out of scope.
