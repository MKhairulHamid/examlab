AWS Certified Solutions Architect – Professional (SAP-C02)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified Solutions Architect – Professional (SAP-C02) validates advanced technical skill in designing optimized AWS solutions grounded in the AWS Well-Architected Framework. It is a senior-level exam: it does not test whether you know the services, it tests whether you can make the right architectural call when several options all work, across an entire organization.

  • Target audience: Solutions architects with 2+ years of hands-on AWS experience who evaluate application requirements, make architectural recommendations, and guide design across multiple applications and projects within a complex organization.
  • Prerequisite experience: 2+ years using AWS to design and implement cloud solutions. No certification is required, but the Associate (SAA-C03) covers roughly a third of the assumed knowledge.
  • Why candidates take it: To move into senior, lead, and principal architect roles, to validate enterprise-scale design judgment, and to complete the architecture track after the Associate.
The professional difference: SAP-C02 questions are long, scenario-heavy, and frequently have two technically correct answers — the credit goes to the one that best fits the stated constraints (cost, RTO/RPO, operational overhead, blast radius, time-to-migrate). Reading the requirement precisely matters as much as knowing the services.

Out-of-Scope Skills (Do NOT Test)

The official exam guide explicitly excludes these. Any question that touches them is out of scope — cut it.

  • Frontend development for mobile apps
  • 12-factor app methodology
  • In-depth knowledge of operating systems
Exam tip: SAP-C02 tests architectural judgment under constraints. If a question reduces to a single fact lookup, it is more Associate-level than Professional. Engineer stems that force a trade-off — two valid services, one best fit for the stated RTO/RPO, cost ceiling, operational overhead, or migration timeline.

Emerging Topics (unscored pretest)

The guide notes you may also see unscored questions on emerging topics — currently security and responsible-AI controls: content filtering and compliance for generative AI (Amazon Bedrock Guardrails), access controls for generative and agentic AI applications (AgentCore Identity), and human-oversight/approval workflows (AWS Step Functions). These do not affect the score and are not a focus of our simulations.

Domain 1 · 26%

Design Solutions for Organizational Complexity

The enterprise-scale domain. Covers connectivity across many VPCs and on-premises, cross-account security and identity, resilient design, multi-account governance, and cost visibility at scale.

Task 1.1 — Architect network connectivity strategies

Knowledge of: AWS Global Infrastructure; AWS networking concepts (Amazon VPC, AWS Direct Connect, AWS VPN, transitive routing, AWS container services); hybrid DNS (Route 53 Resolver, on-premises DNS integration); network segmentation (subnetting, IP addressing, connectivity among VPCs); network traffic monitoring.

Skills in:

  • Evaluating connectivity options for multiple VPCs (peering, Transit Gateway, PrivateLink)
  • Evaluating connectivity options for on-premises, co-location, and cloud integration
  • Selecting Regions and Availability Zones based on network and latency requirements
  • Troubleshooting traffic flows using AWS tools (VPC Flow Logs, Reachability Analyzer)
  • Using service endpoints for service integrations

Task 1.2 — Prescribe security controls

Knowledge of: IAM and IAM Identity Center; route tables, security groups, and network ACLs; encryption keys and certificate management (AWS KMS, ACM); security, identity, and compliance tools (CloudTrail, IAM Access Analyzer, Security Hub, Amazon Inspector).

Skills in:

  • Evaluating cross-account access management
  • Integrating with third-party identity providers (SAML, OIDC federation)
  • Deploying encryption strategies for data at rest and in transit
  • Developing a strategy for centralized security event notifications and auditing

Task 1.3 — Design reliable and resilient architectures

Knowledge of: RTOs and RPOs; disaster recovery strategies (AWS Elastic Disaster Recovery, pilot light, warm standby, multi-site); data backup and restoration.

Skills in:

  • Designing disaster recovery solutions based on RTO and RPO requirements
  • Implementing architectures that automatically recover from failure
  • Developing the optimal architecture by considering scale-up and scale-out options
  • Designing an effective backup and restoration strategy

Task 1.4 — Design a multi-account AWS environment

Knowledge of: AWS Organizations and AWS Control Tower; multi-account event notifications; AWS resource sharing across environments (RAM).

Skills in:

  • Evaluating the most appropriate account structure for organizational requirements
  • Recommending a strategy for central logging and event notifications
  • Developing a multi-account governance model (SCPs, OUs, guardrails)

Task 1.5 — Determine cost optimization and visibility strategies

Knowledge of: Cost and usage monitoring tools (Trusted Advisor, Pricing Calculator, Cost Explorer, AWS Budgets); purchasing options (Reserved Instances, Savings Plans, Spot); rightsizing visibility tools (Compute Optimizer, S3 Storage Lens).

Skills in:

  • Monitoring cost and usage with AWS tools
  • Developing an effective tagging strategy that maps costs to business units
  • Understanding how purchasing options affect cost and performance
Domain 2 · 29%

Design for New Solutions

The largest domain. Greenfield design against demanding requirements — deployment, business continuity, security, reliability, performance, and cost, all balanced against the Well-Architected Framework.

Task 2.1 — Design a deployment strategy to meet business requirements

Knowledge of: Infrastructure as code (CloudFormation); CI/CD; change management; configuration management (AWS Systems Manager).

Skills in:

  • Determining an application or upgrade path for new services and features
  • Selecting services for deployment strategies and appropriate rollback mechanisms
  • Adopting managed services to reduce provisioning and patching overhead
  • Making advanced technologies accessible by delegating complex tasks to AWS

Task 2.2 — Design a solution to ensure business continuity

Knowledge of: Global Infrastructure; networking (Route 53, routing methods); RTOs and RPOs; DR scenarios (backup & restore, pilot light, warm standby, multi-site); DR solutions on AWS.

Skills in:

  • Configuring DR solutions and data/database replication
  • Performing disaster recovery testing
  • Architecting an automated, cost-effective backup solution across AZs or Regions
  • Designing for application and infrastructure availability during a disruption
  • Using centralized monitoring to proactively recover from failures

Task 2.3 — Determine security controls based on requirements

Knowledge of: IAM; route tables, security groups, network ACLs; encryption at rest and in transit; service endpoints; credential management; managed security services (Shield, WAF, GuardDuty, Security Hub).

Skills in:

  • Specifying IAM users and roles that follow least privilege
  • Specifying inbound/outbound flows with security group and network ACL rules
  • Developing attack-mitigation strategies for large-scale web applications
  • Developing encryption strategies and specifying service endpoints
  • Developing patch-management strategies to remain compliant

Task 2.4 — Design a strategy to meet reliability requirements

Knowledge of: Global Infrastructure; storage and replication strategies (S3, RDS, ElastiCache); Multi-AZ and multi-Region; auto scaling policies and events; application integration (SNS, SQS, Step Functions); service quotas and limits.

Skills in:

  • Designing highly available environments based on business requirements
  • Designing for failure and ensuring seamless recoverability
  • Implementing loosely coupled dependencies
  • Operating and maintaining HA architectures (application and database failovers)
  • Implementing DNS routing policies (latency-based, geolocation, simple)

Task 2.5 — Design a solution to meet performance objectives

Knowledge of: Performance monitoring technologies; storage options; instance families and use cases; purpose-built databases.

Skills in:

  • Designing large-scale architectures for a variety of access patterns
  • Designing an elastic architecture based on business objectives
  • Applying caching, buffering, and replicas to meet performance objectives
  • Developing a methodology for selecting purpose-built services
  • Designing a rightsizing strategy

Task 2.6 — Determine a cost optimization strategy to meet solution goals

Knowledge of: Cost and usage monitoring tools (Cost Explorer, Trusted Advisor, Pricing Calculator); pricing models (Reserved Instances, Savings Plans); storage tiering; data transfer costs; managed service offerings.

Skills in:

  • Identifying opportunities to select and rightsize cost-effective resources
  • Identifying appropriate pricing models
  • Performing data-transfer modeling and selecting services to reduce transfer costs
  • Implementing controls for expenditure and usage awareness
Domain 3 · 25%

Continuous Improvement for Existing Solutions

The brownfield domain. You are handed a system that already runs and must improve it — operationally, in security, performance, reliability, and cost — usually without a full rebuild.

Task 3.1 — Determine a strategy to improve overall operational excellence

Knowledge of: Alerting and automatic remediation; DR planning; monitoring and logging (CloudWatch); CI/CD and deployment strategies (blue/green, all-at-once, rolling); configuration management (Systems Manager).

Skills in:

  • Determining the most appropriate logging and monitoring strategy
  • Evaluating current deployment processes for improvement
  • Prioritizing opportunities for automation within a solution stack
  • Recommending solutions for configuration-management automation
  • Engineering failure-scenario activities to exercise recovery actions

Task 3.2 — Determine a strategy to improve security

Knowledge of: Data retention, sensitivity, and regulatory requirements; automated monitoring and remediation (AWS Config rules); secrets management (Systems Manager, Secrets Manager); least privilege; security-specific solutions; patching; backup practices.

Skills in:

  • Evaluating secure management of secrets and credentials
  • Auditing an environment for least-privilege access
  • Reviewing solutions to ensure security at every layer and full traceability
  • Prioritizing automated responses to detected vulnerabilities
  • Designing and implementing patch, update, and backup processes; employing remediation techniques

Task 3.3 — Determine a strategy to improve performance

Knowledge of: High-performing architectures (auto scaling, instance fleets, placement groups); global offerings (Global Accelerator, CloudFront, edge computing); monitoring tools (CloudWatch); SLAs and KPIs.

Skills in:

  • Translating business requirements into measurable metrics
  • Testing potential remediation solutions and making recommendations
  • Proposing adoption of new technologies and managed services
  • Assessing solutions and applying rightsizing based on requirements
  • Identifying and examining performance bottlenecks

Task 3.4 — Determine a strategy to improve reliability

Knowledge of: Global Infrastructure; data replication methods; scaling methodologies (load balancing, auto scaling); high availability and resiliency; DR methods and tools; service quotas and limits.

Skills in:

  • Understanding application growth and usage trends
  • Evaluating existing architecture for insufficient reliability
  • Remediating single points of failure
  • Enabling data replication, self-healing, and elastic features

Task 3.5 — Identify opportunities for cost optimizations

Knowledge of: Cost-conscious choices (Spot, scaling policies, rightsizing); price-model adoption (Reserved Instances, Savings Plans); networking and data-transfer costs; cost management, alerting, and reporting.

Skills in:

  • Analyzing usage reports to identify under/over-utilized resources
  • Using AWS solutions to identify unused resources
  • Designing billing alarms based on expected usage patterns
  • Investigating Cost and Usage Reports at a granular level
  • Using tagging for cost allocation and reporting
Domain 4 · 20%

Accelerate Workload Migration and Modernization

The migration domain. From choosing what to migrate and how (the 7 Rs, TCO, the right transfer tool) to landing it on a better architecture and modernizing toward serverless and purpose-built services.

Task 4.1 — Select existing workloads and processes for potential migration

Knowledge of: Migration assessment and tracking tools (AWS Migration Hub); portfolio assessment; asset planning; prioritization and migration of workloads (wave planning).

Skills in:

  • Completing an application migration assessment
  • Evaluating applications against the seven common migration strategies (7 Rs)
  • Evaluating total cost of ownership (TCO)

Task 4.2 — Determine the optimal migration approach for existing workloads

Knowledge of: Data migration tools (DataSync, Transfer Family, Snow Family, S3 Transfer Acceleration); application migration tools (Application Discovery Service, Application Migration Service); networking and DNS (Direct Connect, Site-to-Site VPN, Route 53); identity (IAM Identity Center, Directory Service); database migration (DMS, SCT); governance (Control Tower, Organizations).

Skills in:

  • Selecting the appropriate database transfer mechanism
  • Selecting the appropriate application transfer mechanism
  • Selecting the appropriate data transfer service and migration strategy
  • Applying appropriate security methods to migration tools
  • Selecting the appropriate governance model

Task 4.3 — Determine a new architecture for existing workloads

Knowledge of: Compute (EC2, Elastic Beanstalk); containers (ECS, EKS, Fargate, ECR); storage (EBS, EFS, FSx, S3, Storage Gateway Volume Gateway); databases (DynamoDB, OpenSearch Service, RDS, self-managed on EC2).

Skills in:

  • Selecting the appropriate compute platform
  • Selecting the appropriate container hosting platform
  • Selecting the appropriate storage service
  • Selecting the appropriate database platform

Task 4.4 — Determine opportunities for modernization and enhancements

Knowledge of: Serverless compute (AWS Lambda); containers (ECS, EKS, Fargate); storage (S3, EFS); purpose-built databases (DynamoDB, Aurora Serverless, ElastiCache); integration services (SQS, SNS, EventBridge, Step Functions).

Skills in:

  • Identifying opportunities to decouple application components
  • Identifying opportunities for serverless solutions
  • Selecting the appropriate service for containers
  • Identifying opportunities for purpose-built databases
  • Selecting the appropriate application integration service
Candidate Prep

How We Help SAP-C02 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 750+.

Who Takes This Exam

  • Experienced architects — already designing on AWS, now formalizing senior-level breadth and trade-off judgment.
  • SAA holders leveling up — Associate-certified engineers moving from "can design" to "can own the architecture."
  • Migration and platform leads — responsible for moving and modernizing real workloads at organizational scale.

What Candidates Struggle With

  • Long, multi-constraint stems — Professional questions bury the deciding constraint (RTO/RPO, cost ceiling, operational overhead, blast radius) in a wall of text. Missing it picks the wrong "correct" answer.
  • Two-valid-answers problem — both options work; only one best fits the requirement. Candidates who pattern-match on keywords instead of constraints get these wrong.
  • Connectivity at scale — VPC peering vs. Transit Gateway vs. PrivateLink vs. Direct Connect vs. VPN, plus hybrid DNS. Easy to confuse transitive routing and shared-services patterns.
  • Multi-account governance — Organizations, SCPs, Control Tower, and RAM interact in ways that trip up single-account thinkers.
  • DR strategy selection — mapping backup & restore / pilot light / warm standby / multi-site to a specific RTO/RPO and budget.
  • The 7 Rs and migration tooling — rehost vs. replatform vs. refactor, and which transfer tool (DataSync, DMS, Snow, Transfer Family) fits the data size, downtime window, and connectivity.
  • Pacing — 75 questions in 180 minutes with long stems is ~2.4 min/question. Time management is itself a skill the exam tests.

Recommended Study Path

  1. Domain 2 — New Solutions first. Largest weight (29%) and the broadest application of the Well-Architected pillars. Build the trade-off muscle here.
  2. Domain 1 — Organizational Complexity. Connectivity, cross-account security, multi-account governance — the topics most unique to the Professional level and the biggest jump from the Associate.
  3. Domain 3 — Continuous Improvement. Reuses D1/D2 knowledge but reframed as "improve what exists." Strong overlap, efficient to study after the first two.
  4. Domain 4 — Migration & Modernization. Smallest (20%) and the most tool-and-process oriented (7 Rs, TCO, transfer tools). A well-bounded final sweep.

AI Explanation Quality for This Exam

SAP-C02 explanations must do more than mark right/wrong — they must show why this option beats the other valid one. Patterns to cover:

  • For two-valid-answer questions: name the deciding constraint and show how the chosen option satisfies it while the runner-up violates it (e.g. "warm standby meets the 5-minute RTO that pilot light cannot").
  • For connectivity questions: explain why a given pattern scales (Transit Gateway as a hub vs. the O(n²) mesh of peering) and when the simpler option still wins.
  • For migration questions: tie the tool to the data size, downtime tolerance, and link bandwidth (Snowball for petabytes over weeks vs. DataSync over an existing link).
  • For cost questions: quantify the trade-off (Savings Plans for steady baseline, Spot for fault-tolerant burst) rather than just naming the cheaper option.

Question Writing Rules (SAP-C02 specific)

  • Constraint-driven stems — every scenario states the deciding constraint (RTO/RPO, cost ceiling, operational overhead, compliance, migration window). The crux is matching the option to the constraint, not recalling a fact.
  • Two-valid-answers by design — for comparative questions, make at least one distractor a genuinely workable option that simply fits worse. Explain the deciding factor.
  • Cross-domain scenarios — Professional questions blend connectivity + security + cost, or migration + reliability. Reflect that in Sim 2.
  • Enterprise framing — multi-account, hybrid, large-scale. Avoid single-account toy problems.
  • No out-of-scope content — no mobile frontend, no 12-factor methodology, no deep OS internals.

Distractor Quality Guidelines

  • Use real architectural confusions: Transit Gateway vs. VPC peering vs. PrivateLink; pilot light vs. warm standby; Savings Plans vs. Reserved Instances vs. Spot; DataSync vs. DMS vs. Snow.
  • For comparative questions, make one distractor the option that is correct under a different constraint, so the candidate must read the stated requirement carefully.
  • For migration questions, make a distractor a tool that technically works but ignores the stated downtime, data-size, or bandwidth constraint.

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — the conceptual reason tied to the stated constraint
  • Why each distractor is wrong — one sentence per option, naming the specific reason (including why a workable-but-worse option loses)
  • Concept link — which domain/task statement this tests (e.g. "D1 · Task 1.3 — DR strategy selection by RTO/RPO")
  • Service context — if a service appears, one line on what it does and what distinguishes it from similar services
