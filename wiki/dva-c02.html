AWS Certified Developer – Associate (DVA-C02)

Everything we need to know about this exam — structure, domains, candidate prep strategy, and simulation design.

What This Exam Is

The AWS Certified Developer – Associate (DVA-C02) validates a candidate's ability to develop, test, deploy, and debug AWS Cloud-based applications. Unlike the foundational Cloud Practitioner exam, DVA-C02 is hands-on: it expects you to read and reason about code, configure services through SDKs and the CLI, and operate CI/CD pipelines.

  • Target audience: Software developers and application engineers who build and maintain applications on AWS.
  • Prerequisite experience: 1+ years of hands-on experience developing and maintaining AWS applications; proficiency in at least one high-level programming language.
  • Why candidates take it: To validate cloud-native development skills, qualify for cloud/backend developer roles, or build toward the DevOps Engineer Professional path. Often paired with SAA for a stronger profile.

Out-of-Scope Job Tasks (Do NOT Test)

The official exam guide explicitly excludes these tasks from the target candidate role. Any question that requires them is out of scope — cut it.

  • Designing architectures (distributed systems, microservices, database schemas and modeling)
  • Designing and creating CI/CD pipelines (the candidate uses pipelines, does not design them)
  • Administering IAM users and groups
  • Administering servers and operating systems
  • Designing AWS networking infrastructure (Amazon VPC, AWS Direct Connect)
Exam tip: DVA-C02 tests developer skills — writing application code, calling service APIs, configuring Lambda, deploying through existing pipelines, and debugging. If a question is about designing the architecture or networking from scratch, it leans toward SAA territory and is out of scope here.

Domain 1 · 32%

Development with AWS Services

The core of the exam. Covers writing resilient application code, building on AWS Lambda, and using data stores (DynamoDB front and center).

Task 1.1 — Develop code for applications hosted on AWS

Skills in:

  • Describing architectural patterns (event-driven, microservices, monolithic, choreography, orchestration, fanout)
  • Distinguishing stateful vs. stateless, tightly vs. loosely coupled, synchronous vs. asynchronous patterns
  • Creating fault-tolerant and resilient applications in a programming language (Java, C#, Python, JavaScript, TypeScript, Go)
  • Creating, extending, and maintaining APIs (request/response transformations, validation rules, status-code overrides)
  • Writing and running unit tests in development environments (e.g. using AWS SAM)
  • Writing code that uses messaging services and interacts with AWS services via APIs and SDKs
  • Handling streaming data; using Amazon EventBridge to implement event-driven patterns
  • Implementing resilient code for third-party integrations (retry logic, circuit breakers, error handling)

Task 1.2 — Develop code for AWS Lambda

Skills in:

  • Accessing private resources in VPCs from Lambda code
  • Configuring functions: environment variables, memory, concurrency, timeout, runtime, handler, layers, extensions, triggers, destinations
  • Handling the event lifecycle and errors (Lambda Destinations, dead-letter queues)
  • Writing and running test code; integrating Lambda functions with AWS services
  • Tuning Lambda functions for optimal performance
  • Using Lambda to process and transform data in near real time

Task 1.3 — Use data stores in application development

Skills in:

  • Designing high-cardinality partition keys for balanced partition access
  • Understanding consistency models (strongly consistent vs. eventually consistent)
  • Distinguishing query vs. scan operations; defining DynamoDB keys and indexing (LSI/GSI)
  • Serializing/deserializing data for persistence; managing and maintaining data stores
  • Managing data lifecycles (e.g. TTL); using data caching services (ElastiCache, DAX)
  • Using specialized data stores based on access patterns (e.g. Amazon OpenSearch Service)
Domain 2 · 26%

Security

Application-level security from a developer's seat: authenticating users and services, encrypting data, and keeping secrets out of code.

Task 2.1 — Implement authentication and/or authorization for applications and AWS services

Skills in:

  • Using an identity provider for federated access (Amazon Cognito, IAM)
  • Securing applications with bearer tokens; configuring programmatic access to AWS
  • Making authenticated calls to AWS services; assuming an IAM role
  • Defining permissions for IAM principals; implementing application-level (fine-grained) authorization
  • Handling cross-service authentication in microservice architectures

Task 2.2 — Implement encryption by using AWS services

Skills in:

  • Defining encryption at rest and in transit; describing certificate management (e.g. AWS Private CA)
  • Distinguishing client-side vs. server-side encryption
  • Using encryption keys to encrypt/decrypt data; using encryption across account boundaries
  • Generating certificates and SSH keys for development; enabling and disabling key rotation

Task 2.3 — Manage sensitive data in application code

Skills in:

  • Describing data classification (PII, PHI)
  • Encrypting environment variables that contain sensitive data
  • Using secret management services (AWS Secrets Manager, SSM Parameter Store) to secure data
  • Sanitizing sensitive data; implementing application-level data masking and sanitization
  • Implementing data access patterns for multi-tenant applications
Domain 3 · 24%

Deployment

Packaging application artifacts, testing in development environments, automating deployment tests, and shipping code through AWS CI/CD services.

Task 3.1 — Prepare application artifacts to be deployed to AWS

Skills in:

  • Managing code-module dependencies (environment variables, configuration files, container images) within the package
  • Organizing files and directory structure for deployment; using code repositories in deployment environments
  • Applying application resource requirements (memory, cores)
  • Preparing application configurations for specific environments (e.g. using AWS AppConfig)

Task 3.2 — Test applications in development environments

Skills in:

  • Testing deployed code by using AWS services and tools
  • Writing integration tests and mock APIs for external dependencies
  • Testing applications using development endpoints (e.g. configuring stages in Amazon API Gateway)
  • Deploying application stack updates to existing environments (e.g. a SAM template to a different staging environment)
  • Testing event-driven applications

Task 3.3 — Automate deployment testing

Skills in:

  • Creating application test events (JSON payloads for testing Lambda, API Gateway, SAM resources)
  • Deploying API resources to various environments
  • Creating environments that use approved versions for integration testing (Lambda aliases, container image tags, Amplify branches, Copilot environments)
  • Implementing and deploying IaC templates (AWS SAM, AWS CloudFormation)
  • Managing environments in individual AWS services (differentiating dev/test/prod in API Gateway)

Task 3.4 — Deploy code by using AWS CI/CD services

Skills in:

  • Describing Lambda deployment packaging options; API Gateway stages and custom domains
  • Updating existing IaC templates (SAM, CloudFormation)
  • Deploying an application version using deployment strategies; performing rollbacks
  • Committing code to a repository to invoke build, test, and deployment actions
  • Using orchestrated workflows to deploy code to different environments
  • Using labels and branches for version and release management
  • Configuring deployment strategies (blue/green, canary, rolling) for application releases
Domain 4 · 18%

Troubleshooting and Optimization

The smallest domain, but where the developer earns their keep: finding root causes, instrumenting code for observability, and optimizing performance and cost.

Task 4.1 — Assist in a root cause analysis

Skills in:

  • Debugging code to identify defects; interpreting application metrics, logs, and traces
  • Querying logs to find relevant data (e.g. CloudWatch Logs Insights)
  • Implementing custom metrics (e.g. CloudWatch embedded metric format [EMF])
  • Reviewing application health using dashboards and insights
  • Troubleshooting deployment failures using service output logs
  • Debugging service-integration issues in applications

Task 4.2 — Instrument code for observability

Skills in:

  • Describing differences between logging, monitoring, and observability
  • Implementing an effective logging strategy; implementing structured logging for events and user actions
  • Implementing code that emits custom metrics; adding annotations for tracing services
  • Implementing notification alerts for specific actions (quota limits, deployment completions)
  • Implementing tracing with AWS services and tools (e.g. AWS X-Ray)
  • Configuring application health checks and readiness probes

Task 4.3 — Optimize applications by using AWS services and features

Skills in:

  • Defining concurrency; profiling application performance
  • Determining minimum memory and compute power for an application
  • Using subscription filter policies to optimize messaging
  • Caching content based on request headers; implementing application-level caching
  • Optimizing application resource usage; analyzing performance issues
  • Using application logs to identify performance bottlenecks

Candidate Prep

How We Help DVA-C02 Candidates

Who takes this exam, where they struggle, and how to build a study path that gets them to 720+.

Who Takes This Exam

  • Working developers — backend and full-stack engineers who already write code and want to validate AWS-native development skills.
  • Cloud Practitioners moving up — candidates who hold CLF-C02 and want their first associate-level, role-based certification.
  • DevOps aspirants — engineers using DVA as the on-ramp to the DevOps Engineer Professional path.

What Candidates Struggle With

  • DynamoDB data modeling — partition vs. sort keys, LSI vs. GSI, query vs. scan, capacity modes, and RCU/WCU math. The single densest topic on the exam.
  • Lambda configuration depth — concurrency (reserved vs. provisioned), memory/timeout tuning, layers, environment variables, VPC access, and error handling with DLQs and Destinations.
  • Cognito user pools vs. identity pools — authentication (who you are) vs. authorization to AWS resources (what you can access). Constantly confused.
  • KMS and envelope encryption — how data keys, the CMK, and client- vs. server-side encryption fit together.
  • Secrets Manager vs. SSM Parameter Store — automatic rotation and cost vs. free hierarchical config. Choosing the right one for the scenario.
  • Deployment strategies — blue/green vs. canary vs. rolling vs. all-at-once, and how Lambda aliases and traffic shifting implement them.
  • CI/CD service roles — what CodePipeline, CodeBuild, CodeDeploy, and CodeArtifact each do and where they sit in a pipeline.

Recommended Study Path

  1. Domain 1 — Development first. Highest weight (32%) and the foundation everything else builds on. Master DynamoDB and Lambda before anything else — they recur across every other domain.
  2. Domain 2 — Security. Application-level auth, KMS encryption, and secrets management. Cognito and IAM roles for applications are the highest-yield subtopics.
  3. Domain 3 — Deployment. Learn the CI/CD service map and deployment strategies as a single connected story (commit → build → test → deploy → rollback).
  4. Domain 4 — Troubleshooting last. Smallest domain (18%), but it ties the others together. CloudWatch, X-Ray, and Lambda optimization reuse concepts you already learned — efficient marks close to exam day.

AI Explanation Quality for This Exam

DVA-C02 questions use concrete developer scenarios. Every wrong answer needs an explanation that says WHY the distractor is wrong — not just "this is not correct." Common patterns to cover:

  • For DynamoDB questions: explain the access-pattern reason (e.g. "a Scan reads every item and then filters — use a Query on the GSI to read only the matching partition").
  • For Lambda questions: explain the configuration consequence (e.g. "provisioned concurrency keeps instances warm to remove cold starts; reserved concurrency only caps the maximum").
  • For security questions: explain the authN vs. authZ or key-handling distinction (e.g. "a Cognito user pool authenticates users; an identity pool vends temporary AWS credentials").
  • For deployment questions: explain what each strategy optimizes for (e.g. "canary shifts a small percentage of traffic first to limit blast radius before a full rollout").

Question Writing Rules (DVA-C02 specific)

  • Scenario-based, developer-framed stems — "A developer needs to…" or "An application running on Lambda must…" Never "What is the definition of…?"
  • Code-aware, not code-heavy — questions may reference SDK calls, IAM policy snippets, or buildspec/SAM fragments, but they test the decision, not syntax recall.
  • Service operation over service awareness — unlike CLF, DVA asks HOW to configure or use a service (e.g. which concurrency setting, which DynamoDB index), not just which service to pick.
  • Best-practice framing — many questions reduce to "which is the MOST secure / MOST cost-effective / LEAST operational-overhead way to do X."
  • Ground every question in a task statement — tie each to a specific 1.x–4.x skill so the set stays on-blueprint.

Distractor Quality Guidelines

  • Use real developer confusions as distractors: Query vs. Scan; LSI vs. GSI; reserved vs. provisioned concurrency; user pool vs. identity pool; Secrets Manager vs. Parameter Store; SNS vs. SQS vs. EventBridge.
  • For security questions: make one distractor confuse authentication with authorization, and one mishandle key management (e.g. storing a plaintext key, or client-side where server-side is required).
  • For deployment questions: make each distractor a real but wrong strategy for the stated goal (e.g. "all-at-once for a zero-downtime production release" is plausible but wrong).

AI Explanation Template

Each question requires an explanation covering:

  • Why the correct answer is right — with a brief conceptual reason
  • Why each distractor is wrong — one sentence per option, naming the specific reason
  • Concept link — which domain/task statement this tests (e.g. "D1 · Task 1.3 — DynamoDB keys and indexing")
  • Service context — if a service appears, one line on what it does and what distinguishes it from similar services
