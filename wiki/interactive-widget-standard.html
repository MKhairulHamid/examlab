Interactive Widget Standard

How to build inline interactive learning widgets for study sessions. Derived from the Precision/Recall widget through many revisions — each revision fixed a violation of one of the four pillars below.

When to build one: only when a session has one core insight that direct manipulation makes unavoidable — a trade-off, a threshold, a transformation. If prose teaches it just as well, don't build a widget. Widgets reinforce a lesson; they never carry it.
Architecture

Architecture

Registry pattern. One key, one component, triggered from the course data.

// src/components/study/SessionCourse.jsx
const INTERACTIVE_WIDGETS = {
  'precision-recall': PrecisionRecallWidget,
  'inference-parameters': InferenceParamsWidget,
  // 'key': Component
};

// src/data/<exam>Course.js — trigger by tagging a section:
{ heading: 'Precision vs. recall', body: '...', interactive: 'precision-recall' }

EnhancedSection renders the widget inline wherever a section carries an interactive key.

Component rules

  • Self-contained, pure client-side React. One state object → derived stats → SVG. No external deps, no duplicated state.
  • Simpler widgets share the WidgetShell helper for the card + header chrome. Draggable-SVG widgets inline their own chrome.
  • Drag/resize: SVG coords via getBoundingClientRect(); a startDrag(type, e) closure attaches window mouse + touch move/up listeners; clamp() to bounds; layer pointer-events (dots pointerEvents:none, handles receive events).
Pillars

The Four Pillars

Every shipped widget satisfies all four. Most bugs were a violation of one of these.

Pillar 1 — Make the core insight unavoidable most important

  • Pick the ONE idea to teach. Engineer the data so both extremes and the intermediate range are reachable, but the "cheat" state that hides the lesson is impossible.
  • Precision/recall: an 8-dot cluster + 2 outliers among the greens — you can hit 100% precision OR 100% recall, never both.
  • Verify numerically with a throwaway node -e script at the key box positions before shipping.

Pillar 2 — Intuitive encoding

  • Colors match real-world meaning (fraud = red, innocent = green).
  • Concrete, relatable scenario (100 transactions, not "samples").
  • Realistic proportions (90/10 imbalance, not 50/50).

Pillar 3 — Direct manipulation + live feedback

  • Manipulate the thing itself (drag/resize a box > an abstract slider).
  • Everything recomputes instantly from the single state object.
  • A live coaching banner interprets the current state in plain language — the widget teaches, it doesn't just display.

Pillar 4 — Mobile-first

  • Invisible touch targets much larger than the visible knobs (~40px hit vs ~13px handle).
  • touchAction:'none' + passive:false listeners so dragging doesn't scroll the page.
  • Wire both onMouseDown and onTouchStart.

Build & Ship

Build & Ship

  • Worktree Node is v12 and can't run Vite. Copy the file to the main repo at /Users/hamid/CloudExamLab, then source ~/.nvm/nvm.sh && nvm use 20 && npm run build.
  • Commit in both the worktree and the main repo, then push to master.

Pre-ship checklist

  • ☐ One insight, and the cheat state is provably unreachable (verified with a node -e check)
  • ☐ Encoding maps to real-world meaning; realistic proportions
  • ☐ Direct manipulation with a live plain-language coaching banner
  • ☐ Touch targets ≥ ~40px; both mouse and touch wired; page doesn't scroll while dragging
  • ☐ Single state object → derived stats → SVG; no duplicated state
